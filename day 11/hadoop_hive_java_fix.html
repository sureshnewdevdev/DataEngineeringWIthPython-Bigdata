Fixing the Hadoop / Hive Java version conflict

OS Ubuntu 24.04 Hadoop 3.3.6 Hive 4.1.0 Target JDK OpenJDK 11

01 — Symptom

Hive is installed but won't come up, and the trail leads back to Hadoop's Java.

Both Hadoop and Hive are installed, yet Hive fails to start. The cause is the Java version Hadoop is configured to run on — not Hive itself.

i

A trap worth knowing: hive --version succeeds and prints the version, because it never actually touches Hadoop. The failure only surfaces when Hive talks to HDFS or the metastore — so the version check looks healthy while real work breaks.

02 — Root cause

JAVA_HOME pointed at Java 17, but Hadoop 3.3.6 supports only Java 8 and 11 at runtime. Under Java 17, Hadoop 3.3.x hits the JPMS module-encapsulation wall and throws reflection errors like InaccessibleObjectException: module java.base does not "opens java.lang".

Hive 4.1.0 inherits JAVA_HOME from Hadoop, so once Hadoop points at an unsupported JDK, Hive inherits the same broken runtime.

Java 11 is the single version supported by both Hadoop 3.3.6 and Hive 4.1.0, so aligning the whole environment on Java 11 resolves the conflict.

03 — Diagnose

Check what's installed and what each layer actually points at:

bash — diagnose
# All installed JDKs
update-alternatives --list java
ls /usr/lib/jvm/

# What the shell and Hadoop point at
echo $JAVA_HOME
grep -i JAVA_HOME $HADOOP_HOME/etc/hadoop/hadoop-env.sh

# Versions in play
java -version
hive --version

Here, java -version reported 11 (correct) but echo $JAVA_HOME still printed the java-17 path — the classic inconsistency. A stale export in ~/.bashrc was the source.

04 — Fix

Target JDK: /usr/lib/jvm/java-11-openjdk-amd64

Step 1 · Install Java 11 if missing

bash
sudo apt update && sudo apt install -y openjdk-11-jdk

Step 2 · Fix the stale JAVA_HOME in ~/.bashrc

bash
grep -n JAVA_HOME ~/.bashrc
sed -i 's|java-17-openjdk-amd64|java-11-openjdk-amd64|g' ~/.bashrc
source ~/.bashrc
echo $JAVA_HOME     # -> /usr/lib/jvm/java-11-openjdk-amd64

Step 3 · Point Hadoop at Java 11

In $HADOOP_HOME/etc/hadoop/hadoop-env.sh:

hadoop-env.sh
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64

Step 4 · Align Hive only if it overrides JAVA_HOME

Check $HIVE_HOME/conf/hive-env.sh. If it sets JAVA_HOME, match it to Java 11; if it doesn't, leave it — Hive inherits Hadoop's value.

Step 5 · Make Java 11 the system default

bash
sudo update-alternatives --set java /usr/lib/jvm/java-11-openjdk-amd64/bin/java
java -version     # -> 11.x

05 — Start & verify

bash — startup
source ~/.bashrc
start-dfs.sh
start-yarn.sh
jps
hive

A healthy cluster shows five daemons in jps:

  • NameNode
  • DataNode
  • SecondaryNameNode
  • ResourceManager
  • NodeManager
!

If a daemon is missing from jps, read its log under $HADOOP_HOME/logs/ — the stack trace names the real cause.

06 — Notes & gotchas

  • The SLF4J "multiple bindings" and log4j "package scanning" messages are harmless warnings, not the failure.
  • Avoid grep -rn across a full Anaconda directory — it crawls thousands of files and appears to hang. Grep the specific config files instead.
  • If you must stay on Java 17, upgrade Hadoop to the 3.5+/4.x line rather than forcing 3.3.6 onto an unsupported JDK.
  • Keep every node in a real cluster on the same JDK — mismatched Java across nodes causes intermittent failures.
