Starting Hive & Beeline
on a Fresh Machine

A copy-paste sequence to bring HiveServer2 up on an Ubuntu + Hadoop box — and to sidestep the Derby lock trap that stops the server from ever opening its port.

Hive 4.1.0 Metastore Derby (embedded) Client Beeline → HiveServer2 Hadoop HDFS running

What this guide assumes

Hadoop/HDFS is already installed and running, and Hive 4.1.0 lives under /opt/bigdata/apache-hive-4.1.0-bin. Adjust that path if your install differs.

The golden rule with embedded Derby: only one Hive process may touch the metastore at a time, and you must always launch from the same directory. Breaking either rule produces the dreaded ERROR XSDB6: Another instance of Derby may have already booted the database. Follow the steps in order and you'll avoid it.

Part A · First-time setup
1

Use Hive's own Beeline, not Spark's

Many Hadoop boxes ship Spark, and Spark bundles an older Beeline (2.3.9) that often wins on your PATH. Mixing it with a 4.1.0 server causes strange failures. Check which one you're getting:

which beeline

If the result is anything like /opt/bigdata/spark/bin/beeline, don't fight the PATH — just call Hive's binary by its full path everywhere in this guide.

2

Move to your home directory and stay there

Derby writes its metastore_db folder into whatever directory you're standing in. Pick one home base and always start Hive from it, or Derby will create duplicate stores and lock conflicts.

cd ~
3

Initialize the metastore schema (once per machine)

Before the very first start, create the Derby schema. Skip this on machines where it's already been done — running it twice errors harmlessly but isn't needed.

/opt/bigdata/apache-hive-4.1.0-bin/bin/schematool -dbType derby -initSchema
Looks right when It finishes with schemaTool completed and no stack trace.
Part B · Start the server & connect
4

Start HiveServer2 detached, logging to a file

Running it with nohup … & sends all its noisy logs into a file and gives your terminal back, so you don't have to juggle windows.

# from ~ , launch in background, redirect logs
nohup /opt/bigdata/apache-hive-4.1.0-bin/bin/hiveserver2 > ~/hs2.log 2>&1 &
5

Wait for the port — don't rush it

HiveServer2 is genuinely slow to open port 10000 (often 1–2 minutes, longer on small heaps). Watch the log, then confirm the port is actually listening.

# follow startup — Ctrl+C stops the tail, NOT the server
tail -f ~/hs2.log
# the real proof: is anything listening on 10000?
ss -tlnp | grep 10000
Ready when ss prints a line for port 10000. Until it does, Beeline will say “Connection refused” — that just means keep waiting, not that something broke. 
LISTEN 0  50  *:10000  *:*  users:(("java",pid=7748,...))
6

Connect with Beeline

Once the port is open, connect using Hive's own Beeline (full path, to dodge the Spark version).

/opt/bigdata/apache-hive-4.1.0-bin/bin/beeline -u jdbc:hive2://localhost:10000
Prompt check SQL only works at the Hive prompt 0: jdbc:hive2://localhost:10000>. If your prompt still starts with vmuser@host:~$ you're in Linux — SQL like SHOW will fail with “command not found.”
7

Verify it works

At the Hive prompt, run:

SHOW DATABASES;
Finish line You should see default in the result. Hive is fully up. To leave Beeline cleanly, type !quit.
Part C · The Derby lock trap (XSDB6)
Symptom HiveServer2 starts as a process but never opens port 10000, and ~/hs2.log (or /tmp/<user>/hive.log) contains:

ERROR XSDB6: Another instance of Derby may have already booted the database /home/<user>/metastore_db.

Cause: a leftover Hive/Beeline/HiveServer2 process from an earlier attempt is still holding the Derby lock. Derby allows only one holder.

The fix — kill everything, clear the lock, start clean:

A

Kill every Hive-related process

# list java processes to see what's running
jps
# then stop the Hive ones
pkill -f hiveserver2
pkill -f RunJar
B

Delete the stale Derby lock files

These .lck files are locks, not your data — safe to remove.

rm -f ~/metastore_db/*.lck
rm -f ~/metastore_db/db.lck
C

Start fresh from home

Then go back to Step 4 above and bring HiveServer2 up again. This time the log should show Hive Session ID = … lines with no XSDB6 error, and the port will open.

cd ~
nohup /opt/bigdata/apache-hive-4.1.0-bin/bin/hiveserver2 > ~/hs2.log 2>&1 &
Quick reference · Errors → fixes
You seeWhat it means & the fix
Connection refused Nothing is listening on 10000 yet. Wait 1–2 min, then re-check ss -tlnp | grep 10000. If the process died, read the log.
ERROR XSDB6 Derby lock held by an old process. Do Part C: pkill, delete *.lck, restart from ~.
SHOW: command not found You typed SQL at the Linux shell. Connect Beeline first; run SQL only at the jdbc:hive2> prompt.
Beeline version 2.3.9 You're using Spark's Beeline. Call Hive's full path: /opt/bigdata/apache-hive-4.1.0-bin/bin/beeline.
Stuck at . . . .> Beeline is waiting to finish a statement. Type ; then Enter, then !quit. Or Ctrl+C.
HiveServer2 running as process N. Stop it first. An instance already exists. Don't start another — either connect to it, or kill N and restart clean.
Long-term tip Embedded Derby's one-process limit is the root of most of this pain. For anything beyond learning/single-user, switch the metastore to MySQL or PostgreSQL — then multiple sessions can connect at once and the lock trap disappears.
