1. Title & Outcomes

Key Point: Hadoop architecture has two major parts: HDFS for storage and YARN for resource management. HDFS answers “where is the data stored?” YARN answers “where will the processing run?”
  • Explain Hadoop 2.x and 3.x architecture.
  • Work practically with NameNode, DataNode, Secondary NameNode, ResourceManager and NodeManager.
  • Inspect blocks and replication.
  • Understand fault tolerance, high availability and reliability.
  • Configure rack awareness in a lab.
Memory Hook: Hadoop is a warehouse plus traffic control. HDFS is the warehouse; YARN is traffic control.

2. Introduction

Hadoop is a distributed platform for storing and processing large data. Distributed means work is spread across many machines. This is useful because a single machine may not have enough disk, memory, CPU, or reliability for huge data.

HDFS, the Hadoop Distributed File System, stores large files by splitting them into blocks and storing replicas across DataNodes. YARN, Yet Another Resource Negotiator, manages compute resources such as memory and CPU for applications.

Hadoop 2.x introduced YARN and made Hadoop a general resource-management platform, not only a MapReduce engine. Hadoop 3.x keeps the same core design and adds operational improvements such as erasure coding and improved high-availability-related capabilities.

3. Beginner-Friendly Analogy

Imagine a logistics company. A huge shipment is split into boxes. Boxes are stored in different warehouses. A master register knows which box is in which warehouse. Copies of important boxes are kept in multiple warehouses so that one warehouse failure does not destroy the shipment.

In Hadoop, the shipment is a file, boxes are HDFS blocks, warehouses are DataNodes, and the master register is the NameNode. The Secondary NameNode helps maintain checkpoints of the register; it is not the emergency manager who instantly takes over.

Now add workers and trucks. Someone must decide which worker runs which task. That is YARN. ResourceManager is the central dispatcher. NodeManager is the local supervisor on each machine. Containers are the approved resource packages where work runs.

  1. File is split into blocks.
  2. Blocks are replicated on DataNodes.
  3. NameNode records metadata.
  4. YARN receives processing request.
  5. ResourceManager allocates containers.
  6. NodeManagers launch containers.
Remember: NN = brain, DN = body, SNN = checkpoint helper, RM = boss, NM = supervisor.

4. Technical Architecture

Technical definition: Hadoop uses a master-worker architecture. HDFS has NameNode as metadata master and DataNodes as storage workers. YARN has ResourceManager as cluster resource master and NodeManagers as compute-resource workers.

Diagram 1: HDFS + YARN Architecture
Purple = masterGreen = workerOrange = block/container

How to read: File operations go through NameNode and DataNodes. Processing operations go through ResourceManager and NodeManagers.

Flow: Client asks NameNode for metadata → DataNodes store blocks → Client submits job to ResourceManager → NodeManagers run containers.

5. Practical Lab Setup

Run these commands on your Hadoop VM. In pseudo-distributed mode all services may run on one machine; in a real cluster they run across multiple machines.

# Start HDFS daemons
start-dfs.sh

# Start YARN daemons
start-yarn.sh

# Verify Java daemons
jps

Expected daemons: NameNode, DataNode, SecondaryNameNode, ResourceManager, NodeManager. Some installations also show JobHistoryServer if started.

# HDFS health
hdfs dfsadmin -report
hdfs dfs -ls /

# YARN health
yarn node -list
yarn application -list
Watch out: jps only confirms a Java process exists. It does not guarantee the cluster is healthy. Always check reports and logs also.

6. Practical: NameNode and DataNode

NameNode practical steps

# Check NameNode process
jps | grep NameNode

# Hadoop 3.x NameNode UI, commonly
http://localhost:9870

# Check safe mode
hdfs dfsadmin -safemode get

# Create namespace entry
hdfs dfs -mkdir -p /user/training/architecture-lab
hdfs dfs -ls /user/training

What this proves: NameNode controls namespace. Creating a folder changes metadata, not DataNode block storage.

DataNode practical steps

# Check DataNode process
jps | grep DataNode

# See live DataNodes
hdfs dfsadmin -report

# Create and upload a file
echo "Hadoop architecture practical" > sample.txt
hdfs dfs -put sample.txt /user/training/architecture-lab/

# Read file back
hdfs dfs -cat /user/training/architecture-lab/sample.txt

What this proves: DataNode stores actual blocks. NameNode only remembers metadata and block locations.

7. Practical: Secondary NameNode

The Secondary NameNode helps with checkpointing. It periodically merges fsimage and edits. It is not a hot standby NameNode.

# Check Secondary NameNode process
jps | grep SecondaryNameNode

# Force checkpoint in lab
hdfs secondarynamenode -checkpoint force

# Check checkpoint directory from configuration
hdfs getconf -confKey dfs.namenode.checkpoint.dir
Interview warning: Never say “Secondary NameNode is backup NameNode.” Correct answer: “It is a checkpointing helper. HA standby NameNode is different.”

8. Practical: ResourceManager and NodeManager

ResourceManager

jps | grep ResourceManager

# RM web UI, commonly
http://localhost:8088

# List YARN applications
yarn application -list
yarn application -list -appStates ALL

What this proves: ResourceManager is the YARN master that tracks applications, scheduler, queues and resource allocation.

NodeManager

# Check NM process
jps | grep NodeManager

# List registered YARN nodes
yarn node -list

# Get node id from previous output, then:
yarn node -status <node_id>

What this proves: NodeManager is the per-node YARN agent that launches and monitors containers.

9. Practical: Replication and Blocks

An HDFS file is split into blocks. Each block is replicated according to the replication factor. In real clusters, replicas are placed on different DataNodes. In one-node labs, true physical multi-node replication is not possible.

# Create sample file
for i in {1..10000}; do echo "Hadoop block test line $i"; done > block-test.txt

# Upload to HDFS
hdfs dfs -put block-test.txt /user/training/architecture-lab/

# Inspect blocks and locations
hdfs fsck /user/training/architecture-lab/block-test.txt -files -blocks -locations

# Check replication factor
hdfs dfs -stat %r /user/training/architecture-lab/block-test.txt

# Change replication factor to 2
hdfs dfs -setrep -w 2 /user/training/architecture-lab/block-test.txt

# Re-check
hdfs fsck /user/training/architecture-lab/block-test.txt -files -blocks -locations
Remember: Block = file piece. Replication = multiple copies. NameNode knows location. DataNode stores content.

11. Practical: Rack Awareness

Rack awareness means Hadoop knows which rack each node belongs to. HDFS can place replicas across racks to improve fault tolerance and network efficiency.

# 1. Create topology mapping file
sudo mkdir -p /etc/hadoop/conf
sudo tee /etc/hadoop/conf/topology.data > /dev/null <<'EOF'
192.168.1.101 /rack1
192.168.1.102 /rack1
192.168.1.103 /rack2
192.168.1.104 /rack2
EOF

# 2. Create topology script
sudo tee /etc/hadoop/conf/topology.sh > /dev/null <<'EOF'
#!/bin/bash
HADOOP_CONF=/etc/hadoop/conf
while [ $# -gt 0 ] ; do
  nodeArg=$1
  exec<${HADOOP_CONF}/topology.data
  result=""
  while read line ; do
    ar=( $line )
    if [ "${ar[0]}" = "$nodeArg" ] ; then result="${ar[1]}"; fi
  done
  shift
  if [ -z "$result" ]; then echo -n "/default-rack "; else echo -n "$result "; fi
done
EOF
sudo chmod +x /etc/hadoop/conf/topology.sh

# 3. Add to core-site.xml
# 
#   net.topology.script.file.name
#   /etc/hadoop/conf/topology.sh
# 

# 4. Restart and verify
stop-yarn.sh && stop-dfs.sh
start-dfs.sh && start-yarn.sh
hdfs dfsadmin -printTopology
Memory hook: Rack awareness is Hadoop’s seating plan. It tries not to put all important copies in the same row.
