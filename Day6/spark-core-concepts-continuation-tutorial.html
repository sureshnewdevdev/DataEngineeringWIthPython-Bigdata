3. Spark Core Mental Model

Definition: Spark Core is the base execution engine of Apache Spark. It handles distributed task scheduling, memory usage, fault recovery, RDD operations and communication between the driver and executors.

Think of Spark Core as the engine room. Even when we use Spark SQL or DataFrames, Spark still needs a lower-level execution engine to schedule and run tasks.

Spoon-feeding Explanation

When we write Spark code, we are not directly controlling every machine. We write high-level operations. Spark Core converts our high-level operations into distributed work and sends that work to executors.

4. Spark Architecture

A Spark application normally contains a Driver, a Cluster Manager, and multiple Executors. The Driver coordinates the application. Executors run tasks and store cached data.

Architecture Flow

User submits application: Example: spark-submit smart_ride.py.
Driver starts: Driver creates SparkSession or SparkContext.
Cluster Manager allocates resources: Executors are launched on worker nodes.
Driver creates jobs/stages/tasks: Based on actions such as count, collect or save.
Executors run tasks: They process partitions and return results/status.
Expected Interview Line: A Spark application is coordinated by the driver program using SparkContext, while executors run tasks and store data for that application.

5. Driver Program

Driver: The Driver is the main control process of a Spark application. It runs the main method/script, creates SparkContext/SparkSession, converts operations into jobs, creates stages and schedules tasks.

Responsibilities of Driver

  • Creates SparkContext or SparkSession.
  • Tracks RDD/DataFrame lineage.
  • Builds DAG when transformations are defined.
  • Starts a job when an action is called.
  • Splits jobs into stages and stages into tasks.
  • Schedules tasks on executors.
  • Collects final result or job status.
Code
→
Driver
→
DAG
→
Task Scheduling

What Driver Should Not Do

Important: Do not bring massive data back to the Driver using collect(). The Driver has limited memory. If 500 GB comes to the Driver, the application can crash.

6. Executor Process

Executor: An Executor is a worker-side process launched for a Spark application. Executors run tasks, store cached data and communicate status/results back to the Driver.

Executor Responsibilities

  • Runs tasks assigned by the Driver.
  • Processes one or more partitions.
  • Stores cached/persisted RDDs or DataFrames.
  • Reports task success/failure to the Driver.
  • Participates in shuffle operations when data must move across partitions.
DriverExecutor
Planning and coordinationActual data processing
Creates jobs, stages and tasksRuns tasks
Maintains application stateStores cached data
Usually fewer in numberMultiple executors can run in parallel
Memory Aid: Driver = manager. Executor = worker. Task = assigned work. Partition = piece of data.

Lazy Evaluation Example

// Scala spark-shell
val trips = sc.textFile("data/trips.csv")
val chennaiTrips = trips.filter(line => line.contains("Chennai"))

// No job is triggered yet because filter is a transformation.

chennaiTrips.count()
// Now Spark starts a job because count() is an action.
Expected Output Idea: If two lines contain Chennai, count() returns 2.
Classroom Activity: Draw four partitions on the board. Ask four learners to act as tasks. One learner is Driver. The Driver tells each learner to count Chennai records in their partition. Then the Driver combines the counts.

8. SparkContext

SparkContext: SparkContext is the main entry point for Spark Core. It represents the connection between the Spark application and the Spark cluster.

Why SparkContext is Important

  • It connects the application to the cluster.
  • It creates RDDs using parallelize() and textFile().
  • It creates broadcast variables and accumulators.
  • It launches jobs when actions are executed.

Common Spark Shell Variable

In spark-shell and older PySpark shell usage, sc commonly represents SparkContext.

// Scala spark-shell
sc

// Create RDD
val numbers = sc.parallelize(Seq(10, 20, 30, 40))
numbers.count()
numbers.collect()

PySpark Example

from pyspark import SparkContext

sc = SparkContext("local[*]", "SmartRideCoreApp")
rdd = sc.parallelize([10, 20, 30, 40])
print(rdd.count())
print(rdd.collect())
sc.stop()
Common Mistake: Creating multiple active SparkContexts in the same application can cause errors. Normally one SparkContext should be active in one JVM/application.

9. SparkConf

SparkConf: SparkConf is a configuration object used to set Spark application properties such as application name, master URL, executor memory and driver memory.

Scala Example

import org.apache.spark.{SparkConf, SparkContext}

val conf = new SparkConf()
  .setAppName("SmartRideCoreApp")
  .setMaster("local[*]")

val sc = new SparkContext(conf)

PySpark Example

from pyspark import SparkConf, SparkContext

conf = SparkConf() \
    .setAppName("SmartRideCoreApp") \
    .setMaster("local[*]") \
    .set("spark.driver.memory", "2g")

sc = SparkContext(conf=conf)
Memory Aid: SparkConf is like the application settings page. SparkContext uses those settings to start and connect the application.

10. SparkSession

SparkSession: SparkSession is the modern unified entry point for Spark SQL, DataFrame, Dataset and many Spark operations. It was introduced to simplify the older approach of using separate contexts.

Why SparkSession?

Earlier, developers had to use separate objects like SparkContext, SQLContext and HiveContext. SparkSession gives one unified object named commonly as spark.

11. SQLContext and HiveContext

SQLContext

SQLContext was used in older Spark versions to work with SQL and DataFrames.

# Older PySpark style
from pyspark.sql import SQLContext
sqlContext = SQLContext(sc)

# DataFrame operations were accessed through sqlContext

HiveContext

HiveContext was used in older Spark versions when we wanted Hive metastore support and HiveQL compatibility.

# Older PySpark style
from pyspark.sql import HiveContext
hiveContext = HiveContext(sc)

Modern Replacement

In modern Spark applications, use SparkSession. For Hive support, build SparkSession with enableHiveSupport().

from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("SmartRideHiveApp") \
    .enableHiveSupport() \
    .getOrCreate()

spark.sql("SHOW DATABASES").show()
Old ComponentPurposeModern Way
SQLContextSQL + DataFrame operationsSparkSession
HiveContextHive metastore + Hive queriesSparkSession + enableHiveSupport()
Interview Memory: SQLContext and HiveContext are older entry points. SparkSession is the modern unified entry point and keeps backward compatibility concepts.

12. Features of Spark

Fast Processing
Spark can keep intermediate data in memory, which helps repeated and iterative workloads.
Distributed Execution
Data and tasks are divided across executors.
Lazy Evaluation
Transformations are planned first and executed when an action is called.
Fault Tolerance
Lost partitions can be recomputed using lineage.
Multi-language APIs
Supports Scala, Python, Java, SQL and R.
Unified Engine
Batch, SQL, streaming, ML and graph workloads can be handled in the Spark ecosystem.

13. Limitations of Spark

Spark is powerful, but it is not magic. Good Spark developers know when Spark is useful and when it is overkill.

LimitationExplanationTrainer Example
Memory pressureSpark can be fast with memory, but poor memory planning causes spills or failures.Caching a very large dataset without enough memory.
Small file problemToo many small files create too many tasks and scheduling overhead.1 lakh tiny trip files instead of fewer larger files.
Shuffle costGroupBy/join operations can move data across the network.Revenue by city may shuffle records by city.
Driver bottleneckLarge collect results can crash the Driver.collect() on 500 GB trip data.
Debugging complexityFailures can happen in code, data, network, executor, memory or cluster setup.Task failure due to bad row in CSV.
Not needed for tiny dataFor small files, Python/Pandas/SQL may be simpler.100-row file does not require Spark cluster.
Important: Spark is best when the data is large enough or the workload benefits from distributed processing. For very small data, Spark startup and scheduling overhead may be more than the actual computation.

15. Best Practices

Use SparkSession in new projects
Prefer SparkSession.builder for DataFrame and SQL workloads.
Use SparkContext for RDD-level teaching
RDD operations are useful to explain Spark Core internals.
Avoid unnecessary collect
Use take(), show(), or write to storage instead.
Cache carefully
Cache only if the same data is reused multiple times.
Understand partitions
Too few partitions reduce parallelism; too many add overhead.
Monitor Spark UI
Use the UI to inspect jobs, stages, tasks, storage and executors.

Rule of Thumb

SituationRecommended Choice
Structured CSV/Parquet/JSON tablesUse DataFrame and Spark SQL
Low-level distributed collection teachingUse RDD
Repeated dataset used in many actionsConsider cache/persist
Large final outputSave to file/table instead of collect

17. Interview Questions and Answers

Q1. What is Spark Core?
Spark Core is the base execution engine of Spark. It manages distributed task scheduling, memory, fault tolerance, RDD operations and communication between driver and executors.
Q2. Explain Spark architecture.
A Spark application has a Driver program, Cluster Manager and Executors. The Driver coordinates the application, the Cluster Manager allocates resources and Executors run tasks on partitions.
Q3. What is Driver in Spark?
The Driver is the main process that runs user code, creates SparkContext/SparkSession, builds jobs/stages/tasks and schedules those tasks on executors.
Q4. What is Executor in Spark?
An Executor is a worker-side process that runs tasks, stores cached data and sends task status/results back to the Driver.
Q5. What is a Task?
A Task is the smallest unit of execution in Spark. Usually one task processes one partition of data.
Q6. What is a Job?
A Job is created when an action is called, such as count, collect, take or saveAsTextFile.
Q7. What is a Stage?
A Stage is a group of tasks that can run together without requiring a shuffle boundary.
Q8. What is a Partition?
A Partition is a logical split of distributed data. Spark processes partitions in parallel using tasks.
Q9. What is SparkContext?
SparkContext is the entry point to Spark Core and represents the connection between the application and Spark cluster.
Q10. What is SparkConf?
SparkConf is used to configure Spark application properties such as app name, master URL, driver memory and executor memory.
Q11. What is SparkSession?
SparkSession is the modern unified entry point for Spark SQL, DataFrame, Dataset and many Spark operations.
Q12. Difference between SparkContext and SparkSession?
SparkContext is mainly for Spark Core/RDD-level operations. SparkSession is the newer unified entry point and internally provides access to SparkContext.
Q13. What is SQLContext?
SQLContext was an older entry point for SQL and DataFrame operations before SparkSession became the recommended unified entry point.
Q14. What is HiveContext?
HiveContext was an older entry point used for Hive integration and HiveQL support. Modern Spark uses SparkSession with enableHiveSupport().
Q15. What is lazy evaluation?
Lazy evaluation means transformations are not executed immediately. Spark builds a plan and executes it only when an action is called.
Q16. Why should we avoid collect on large data?
collect brings all distributed data to the Driver. If the result is large, the Driver can run out of memory.
Q17. What is Spark UI used for?
Spark UI helps monitor jobs, stages, tasks, storage, executors and environment information for a running Spark application.
Q18. Mention limitations of Spark.
Spark can face memory pressure, shuffle cost, small file overhead, driver bottlenecks and debugging complexity. It is also not necessary for very small data.
