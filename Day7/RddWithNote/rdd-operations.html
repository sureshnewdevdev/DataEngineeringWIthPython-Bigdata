1. RDD Operations

An RDD (Resilient Distributed Dataset) is Spark's core data structure - a collection of records split across many machines. "RDD operations" are the two kinds of things you can do to that collection: reshape it (Transformations) or pull a result out of it (Actions).

Learning objectives:

  • Tell a Transformation apart from an Action, and explain why it matters.
  • Explain Narrow vs Wide transformations and what a "shuffle" is.
  • Use map, flatMap, filter, mapPartitions, distinct, intersection, groupByKey, reduceByKey, sortByKey, join, repartition and coalesce correctly.
  • Use the common actions: count, collect, take, top, reduce, countByValue, countByKey.
  • Choose the efficient operation (e.g. reduceByKey over groupByKey) and justify it in an interview.
Before you startPrerequisites: you know what an RDD is and can write a lambda in Python. Estimated time: 50-70 min. After this you can read a Spark job, predict where it will shuffle, and pick the right operation.

2. Introduction

Every Spark program is really just a chain of RDD operations. You start with some data, apply a series of steps to reshape it, and finally ask for a result. Understanding the two categories of operation - Transformations and Actions - is the single most important idea in Spark, because it explains when your code actually runs and where it will be slow.

Why it matters. A Transformation (a "transformation" simply means a step that produces a new RDD from an existing one, such as map or filter) does not run immediately - Spark just remembers it. Only an Action (a step that returns a value to your program, such as count or collect) forces all the remembered steps to run. If you do not know this, your program will behave in confusing ways: an error written on line 3 might only appear on line 30 where the action lives.

Where it is used. Real data pipelines use these operations constantly: filter out bad rows, map raw text into structured records, reduceByKey to total revenue per product, join two datasets, then collect or write the small result. Picking the right operation is the difference between a job that finishes in one minute and one that runs for an hour.

Memory hookTransformations are a to-do list Spark writes down. An Action is you finally saying "go do it". Remember it as: plan lazily, run on action.

3. Beginner-Friendly Explanation

Imagine you run a large mail-sorting room. Trucks drop off millions of letters, and the letters are already spread across many sorting tables (these tables are the partitions - each holds one chunk of the data).

You give the workers a list of instructions:

  1. "Throw away any letter with no stamp" - each worker does this at their own table, touching only their own pile. Nobody needs to walk to another table. This is a narrow step (like filter).
  2. "Write the city name in the corner of every letter" - again done in place, one letter at a time. Narrow (like map).
  3. "Now gather ALL letters going to the same city into one bag" - this forces workers to carry letters across the room to whichever table owns that city. Letters travel. This walking-around is the expensive part, called a shuffle, and it makes the step a wide one (like groupByKey).

Here is the key twist: the workers do not lift a finger while you are just reading out instructions. They only start when you say the magic word - "Give me the final count of bags". That magic word is the Action. Everything before it was just planning.

Where the analogy breaks down: real workers might start early out of habit, but Spark is strict - it truly does nothing until an action is called, which lets it optimise the whole plan first.

Remember it asNarrow = work stays at your own table. Wide = letters travel across the room (a shuffle). Action = the magic word that makes work actually happen.
Check yourself: is "gather all letters for the same city" narrow or wide?

Wide - it forces letters to move between tables (a shuffle), exactly like groupByKey or reduceByKey.

4. Technical Explanation

Definition. An RDD operation is either a Transformation (a lazy operation that takes one or more RDDs and returns a new RDD) or an Action (an operation that runs the computation and returns a value to the driver, or writes to storage).

Key terms

  • Transformation - builds a new RDD; lazy (recorded, not run).
  • Action - triggers execution and produces a result (a number, a list, or a file).
  • Lazy evaluation - Spark delays running transformations until an action asks for a result, so it can optimise the whole chain.
  • Lineage / DAG - the recorded chain of transformations; Spark uses it to rebuild lost data and to plan execution.
  • Narrow transformation - each output partition depends on ONE input partition; no data moves between partitions (e.g. map, filter, flatMap, mapPartitions).
  • Wide transformation - output partitions depend on MANY input partitions, so data is redistributed - a shuffle (e.g. groupByKey, reduceByKey, sortByKey, join, distinct, intersection, repartition).
  • Shuffle - moving data across the network so records with the same key land together; the most expensive thing in Spark.

5. Simple Example

A tiny word-count pipeline in PySpark, explained one line at a time:

rdd    = sc.parallelize(["a cat", "a dog", "a cat"])
words  = rdd.flatMap(lambda line: line.split(" "))
pairs  = words.map(lambda w: (w, 1))
counts = pairs.reduceByKey(lambda a, b: a + b)
result = counts.collect()
print(result)
  • parallelize([...]) - creates the source RDD from a Python list.
  • flatMap(split) - each line becomes many words (one-to-many), so 3 lines become 6 words. Narrow.
  • map(w to (w,1)) - pairs each word with the number 1. Narrow, one-to-one.
  • reduceByKey(a+b) - adds the 1s for each identical word. Wide - this is where the shuffle happens.
  • collect() - the Action. Only now does Spark actually run the four steps above and send the answer to the driver.

Result: [('a', 3), ('cat', 2), ('dog', 1)]. Nothing computed until collect(), because every step before it was a lazy transformation.

Pro tipNotice map + reduceByKey instead of groupByKey. reduceByKey adds up the 1s on each machine BEFORE shuffling, so far less data travels the network.

6. Real-Time Industry Example

An online retailer wants the total sales amount per product category from a day of order events. There are billions of order lines, so a single machine cannot do it. The data team writes an RDD job: read the raw lines, throw away malformed ones, turn each order into a (category, amount) pair, total the amounts per category, then pull the small summary to the driver to store.

Components: Raw orders is the huge source RDD. filter removes rows with missing fields (narrow). map converts each order into a (category, amount) pair (narrow). reduceByKey sums amounts per category and is the only wide step - amounts for the same category must be brought together (a shuffle). collect is the action that triggers everything and returns the small per-category totals.

Flow: (1) Spark records filter → map → reduceByKey but runs nothing. (2) collect() is called. (3) Spark splits into two stages at the shuffle. (4) Stage 1 runs filter+map on every partition in parallel and pre-combines amounts locally. (5) The shuffle groups partial sums by category. (6) Stage 2 finishes the totals. (7) The tiny result (a few dozen categories) is sent to the driver and saved.

Watch outThe result here is small (one number per category), so collect() is safe. Never collect() the raw billions of orders - only collect small, aggregated results.

7. Code Examples

Example A - the narrow transformations (map, flatMap, filter, mapPartitions):

nums = sc.parallelize([1, 2, 3, 4], 2)   # 2 partitions

nums.map(lambda x: x * 10).collect()          # [10,20,30,40]  one-to-one
nums.filter(lambda x: x % 2 == 0).collect()   # [2,4]          keep evens
sc.parallelize(["a b","c"]).flatMap(
        lambda s: s.split(" ")).collect()      # ['a','b','c']  one-to-many

# mapPartitions: the function receives the WHOLE partition (an iterator)
def add_all(part):
    yield sum(part)                            # runs once per partition
nums.mapPartitions(add_all).collect()          # [3, 7] -> per-partition sums

What & why: map transforms each element to exactly one result. filter keeps elements passing the test. flatMap can emit many results per input and flattens them. mapPartitions runs your function once per partition instead of once per element - use it to do expensive setup (like opening a connection) a single time per partition rather than per row.

Example B - the wide transformations (groupByKey vs reduceByKey, join, distinct, sortByKey):

pairs = sc.parallelize([("a",1),("b",1),("a",1)])

pairs.reduceByKey(lambda a,b: a+b).collect()   # [('a',2),('b',1)]  efficient
pairs.groupByKey().mapValues(list).collect()   # [('a',[1,1]),('b',[1])]  heavy
pairs.sortByKey().collect()                     # sorted by key

left  = sc.parallelize([("a",1),("b",2)])
right = sc.parallelize([("a",9)])
left.join(right).collect()                      # [('a',(1,9))]  match on key

sc.parallelize([1,1,2,3]).distinct().collect()  # [1,2,3]  de-duplicate

What & why: reduceByKey and groupByKey both group by key, but reduceByKey combines values on each machine BEFORE the shuffle, so it moves far less data - prefer it for aggregations. join matches records from two RDDs by key (a shuffle). distinct and sortByKey also shuffle.

Example C - repartition vs coalesce, and the common actions:

big = sc.parallelize(range(100), 8)

big.repartition(16)   # full shuffle, can go UP or down, evens out data
big.coalesce(2)       # merges to 2 partitions, NO full shuffle (only reduces)

big.count()                       # 100          (action)
big.take(3)                       # [0,1,2]      (action, just a peek)
big.top(3)                        # [99,98,97]   (action, n largest)
big.reduce(lambda a,b: a+b)       # 4950         (action, squash to one)
sc.parallelize([1,1,2]).countByValue()   # {1:2, 2:1}  (action)
pairs.countByKey()                # {'a':2,'b':1}       (action)

What & why: repartition changes the number of partitions with a full shuffle (use it to increase parallelism or fix skew). coalesce only reduces partitions and avoids a full shuffle, so it is cheaper when you just want fewer output files. The rest are Actions: they end the lazy chain and return a value to the driver. collect is deliberately left out of the "safe" list here because it pulls everything back.

When to pick which: Reach for the operation that moves the least data. Combine before you shuffle (reduceByKey), shrink data early (filter first), and only bring small results to the driver. If you are just lowering the number of output files, use coalesce, not repartition.

9. Best Practices

Do:

  • Prefer reduceByKey / aggregateByKey over groupByKey. Why: they combine locally before the shuffle, moving far less data.
  • filter as early as possible. Why: smaller data makes every later step faster.
  • Use coalesce to reduce partitions, repartition only when you must increase or rebalance. Why: coalesce avoids a full shuffle.
  • Use mapPartitions for per-partition setup (connections, models). Why: you pay the setup cost once per partition, not once per row.

Common mistakes:

  • collect() on a large RDD. Why bad: it pulls everything into driver memory and can crash it.
  • groupByKey for aggregation. Why bad: it shuffles every value; reduceByKey is cheaper.
  • Chaining many wide transformations blindly. Why bad: each shuffle is expensive; minimise and combine them.
  • Forgetting operations are lazy. Why bad: you expect work to happen at a transformation, but it only happens at the action.

10. Edge Cases & Gotchas

  • Lazy errors: a bug inside a map lambda may only surface when the action runs, far from where you wrote it.
  • Key skew: if one key holds most of the records, its reduce/join task lags while others finish - the job appears stuck near the end.
  • coalesce can under-parallelise: coalescing to very few partitions before a heavy step starves you of parallelism; sometimes repartition is the right call despite the shuffle.
  • groupByKey memory blowups: a key with millions of values must fit its value list in memory on one executor - reduceByKey avoids this.
  • take vs top: take(n) returns any n (usually the first partition's), top(n) returns the largest n - do not use take expecting sorted results.
Watch out"Works on sample, hangs on full data" is almost always a wide operation plus skew - not a code bug in your lambda.

12. Interview Q&A (15)

BeginnerWhat are the two categories of RDD operations?
Transformations, which build a new RDD lazily (map, filter), and Actions, which trigger execution and return a value to the driver (count, collect).
BeginnerWhat does map do?
Applies a function to every element and returns a new RDD with exactly one output per input (one-to-one).
BeginnerWhat does filter do?
Returns a new RDD containing only the elements that pass a boolean test; it can shrink but never grow the data.
BeginnerWhat does collect do and why be careful?
It returns ALL elements to the driver as a list. On large data it can exhaust driver memory and crash the app.
BeginnerIs reduceByKey a transformation or an action?
A transformation - it returns a new RDD. reduce (without ByKey) is the action that returns a single value.
IntermediateNarrow vs wide transformation?
Narrow: each output partition depends on one input partition, no data movement (map, filter). Wide: output depends on many input partitions, causing a shuffle (groupByKey, join).
Intermediatemap vs flatMap?
map returns exactly one element per input; flatMap can return zero, one, or many and flattens the results into a single RDD.
IntermediategroupByKey vs reduceByKey - which is better?
reduceByKey, for aggregations. It combines values locally on each machine before the shuffle, so much less data crosses the network than groupByKey, which shuffles every value.
Intermediaterepartition vs coalesce?
repartition does a full shuffle and can increase or decrease partitions; coalesce only reduces partitions and avoids a full shuffle, making it cheaper for lowering file counts.
IntermediateWhat is mapPartitions and when do you use it?
It runs your function once per partition (receiving an iterator) instead of once per element - useful to amortise setup like opening a DB connection or loading a model once per partition.
AdvancedWhy are transformations lazy, and what is the benefit?
Spark records transformations as a lineage/DAG and only runs them on an action. Laziness lets Spark optimise the whole chain (pipelining narrow steps, minimising shuffles) instead of running each step blindly.
AdvancedWhat is a shuffle and why is it expensive?
A shuffle redistributes data across the network so records with the same key are co-located. It involves disk writes, network transfer, and serialization, making it the main performance cost of wide operations.
AdvancedHow does join work on RDDs and how do you optimise it?
join matches records by key, which normally requires a shuffle of both RDDs. If one side is small, broadcast it (broadcast join) so the large RDD is joined locally with no shuffle.
AdvancedWhat is lineage and how does it give fault tolerance?
Lineage is the recorded chain of transformations that built an RDD. If a partition is lost, Spark recomputes just that partition from its lineage instead of replicating all data.
AdvancedcountByKey vs reduceByKey then collect - trade-offs?
countByKey is an action that returns a map to the driver and can overwhelm it if there are many keys. reduceByKey stays distributed as an RDD; you collect only if the result is small. Prefer the distributed path for many keys.

15. Glossary

  • Action - operation that triggers execution and returns a value (count, collect, reduce).
  • Coalesce - reduce partition count without a full shuffle.
  • flatMap - transformation returning 0..many elements per input.
  • Lazy evaluation - transformations run only when an action is called.
  • Lineage / DAG - recorded chain of transformations used for planning and recovery.
  • Narrow transformation - no data movement between partitions.
  • reduceByKey - combine values per key with local pre-combine before the shuffle.
  • Repartition - change partition count via a full shuffle.
  • Shuffle - network redistribution of data by key; the costly part of wide ops.
  • Transformation - lazy operation returning a new RDD.
  • Wide transformation - depends on many input partitions; causes a shuffle.

16. Summary

RDD operations are either Transformations (lazy, return a new RDD) or Actions (trigger execution, return a value). Transformations are narrow (no shuffle: map, flatMap, filter, mapPartitions) or wide (shuffle: distinct, intersection, groupByKey, reduceByKey, sortByKey, join, repartition). Actions include count, collect, take, top, reduce, countByValue, and countByKey.

Quick-revision cheat list:

  • Transformation = lazy (new RDD) · Action = runs it (returns a value).
  • Narrow = no shuffle · Wide = shuffle (data travels).
  • map = one-to-one · flatMap = one-to-many · filter = keep some · mapPartitions = per partition.
  • reduceByKey beats groupByKey (pre-combine) · coalesce beats repartition to shrink partitions.
  • collect brings everything home - use only on small results; take(n) for a peek, top(n) for the largest.
  • reduceByKey = transformation · reduce = action.
