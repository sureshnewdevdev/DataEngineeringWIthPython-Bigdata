2. Avoiding unnecessary UDFs

A User-Defined Function adds custom logic that Spark’s optimizer may not fully understand. A normal Python UDF also crosses the JVM–Python boundary and commonly processes serialized rows, adding cost.

Why built-in functions are normally faster

  • Catalyst can inspect and optimize the expression.
  • Column pruning, predicate pushdown and code generation remain available.
  • Data stays inside Spark’s optimized execution path.
  • No ordinary Python row-by-row serialization boundary.

Bad: Python UDF for a built-in operation

from pyspark.sql import functions as F
from pyspark.sql.types import StringType

@F.udf(returnType=StringType())
def normalize_city(city):
    return city.strip().upper() if city else None

bad_df = orders.withColumn("city_clean", normalize_city("city"))

Better: built-in column functions

from pyspark.sql import functions as F

better_df = orders.withColumn(
    "city_clean",
    F.upper(F.trim(F.col("city")))
)

Bad and better business-rule examples

# Prefer when/otherwise to a UDF
optimized = orders.withColumn(
    "order_band",
    F.when(F.col("amount") >= 50000, "HIGH")
     .when(F.col("amount") >= 10000, "MEDIUM")
     .otherwise("LOW")
)

# Prefer built-in date, array, JSON, regex and string functions
optimized = optimized.withColumn(
    "order_year", F.year("order_date")
).withColumn(
    "email_domain", F.regexp_extract("email", r"@(.+)$", 1)
)
ChoiceWhen to usePerformance note
Built-in SQL/DataFrame functionWhenever it expresses the requirementFirst choice
SQL expressionComplex built-in combinationsStill Catalyst-visible
Pandas UDFVectorized Python library logic genuinely requiredBatch/Arrow overhead remains
Python UDFNo suitable built-in/vectorized solutionMeasure carefully
Scala/Java UDFCustom JVM logicAvoids Python boundary but can remain optimizer-opaque
Do not blindly remove all UDFs. Replace only when built-ins or a clearer native expression can implement the same logic. First validate null behavior and exact output.

3. Reducing shuffle

A shuffle redistributes data between executors, usually because rows with the same key must meet. It can cause network transfer, serialization, sorting, disk spill and uneven tasks.

Executor 1
A, B
Executor 2
A, C
⇄ network shuffle ⇄
Partition A
A, A
Partition B/C
B, C

Operations that commonly shuffle

groupBy, distinct, orderBy, repartition, large joins, reduceByKey and window operations partitioned by a key.

Techniques

TechniqueExampleWhy it helps
Filter earlyDate/status before joinFewer rows move
Project earlySelect needed columnsSmaller rows move
Pre-aggregateRevenue by product before dimension joinFewer fact rows
Broadcast small dimensionproducts tableAvoids large-side shuffle
Avoid unnecessary repartitionDo not repartition twiceRemoves extra exchange
Use coalesce for reducing partitions200 → 20 without full rebalanceOften avoids full shuffle
Handle skewSplit/salt hot keyPrevents one huge task
Use suitable partition countEnough tasks, not thousands of tiny tasksBalances parallelism and overhead
# Filter and project before join
fact = (
    orders
    .filter((F.col("order_date") == "2026-07-17") &
            (F.col("status") == "DELIVERED"))
    .select("product_id", "amount")
)

# Pre-aggregate before dimension join
product_revenue = fact.groupBy("product_id").agg(
    F.sum("amount").alias("revenue")
)

result = product_revenue.join(
    F.broadcast(products.select("product_id", "category")),
    "product_id"
).groupBy("category").agg(F.sum("revenue").alias("revenue"))

RDD example

# Avoid groupByKey when only an aggregate is needed
bad = sales_pair_rdd.groupByKey().mapValues(sum)

# reduceByKey performs local combining before shuffle
better = sales_pair_rdd.reduceByKey(lambda a, b: a + b)
Common mistake: coalesce(1) sends final work toward one partition and can create a bottleneck. Use it only for genuinely small outputs.

4. Disable DEBUG and INFO logging safely

For training and normal production runs, set the Spark application log level to WARN or ERROR. This reduces console noise; it does not improve a poorly designed query.

PySpark runtime setting

spark.sparkContext.setLogLevel("WARN")
# Use "ERROR" only when you need an even quieter console.

Permanent/custom Log4j2 configuration

# log4j2.properties
rootLogger.level = warn
rootLogger.appenderRef.console.ref = console

appender.console.type = Console
appender.console.name = console
appender.console.layout.type = PatternLayout
appender.console.layout.pattern = %d{HH:mm:ss} %-5p %c{1}: %m%n

# Keep your application package visible
logger.retail.name = com.company.retail
logger.retail.level = info

spark-submit with a custom file

spark-submit   --master yarn   --deploy-mode cluster   --conf "spark.driver.extraJavaOptions=-Dlog4j.configurationFile=log4j2.properties"   --conf "spark.executor.extraJavaOptions=-Dlog4j.configurationFile=log4j2.properties"   --files log4j2.properties   retail_optimization.py
Keep WARN and ERROR. They contain task failures, executor losses and memory diagnostics. During troubleshooting, temporarily restore INFO for the relevant logger instead of turning every logger to DEBUG.

5. Sort-merge join

A SortMergeJoin is commonly used for large equi-joins. Spark shuffles both sides by the join key, sorts partitions, and then merges matching sorted streams.

Large orders
shuffle + sort
Key-partitioned orders
merge
Joined rows
merge
Key-partitioned customers
sort + shuffle
Large customers

When it is suitable

  • Both sides are large.
  • The condition is an equality join.
  • Neither side safely fits in executor broadcast memory.
  • Shuffle and sort cost are acceptable.
large_join = orders.join(customers, "customer_id", "inner")
large_join.explain("formatted")

# Training-only way to discourage automatic broadcasting
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", -1)
Disabling broadcast is useful for comparison labs, not a universal production recommendation. AQE or normal planning may otherwise choose a faster strategy.

6. Broadcast hash join

A BroadcastHashJoin sends a small table to every executor. Each executor builds an in-memory hash table and joins its local large-table partitions without shuffling the large side.

Small products
broadcast copy
Executor 1 hash
Executor 2 hash
Executor 3 hash
from pyspark.sql.functions import broadcast

result = orders.join(
    broadcast(products),
    on="product_id",
    how="inner"
)
result.explain("formatted")
-- Spark SQL hint
SELECT /*+ BROADCAST(p) */
       o.order_id, p.category, o.amount
FROM orders o
JOIN products p
  ON o.product_id = p.product_id;

Broadcast safety calculation

Do not judge only from source-file size. Spark broadcasts a materialized/serialized relation and executor hash structures consume memory.

Practical check: broadcast footprint × concurrent joins + execution memory + safety margin must fit every executor. If a 200 MB relation expands significantly in memory, broadcasting it to every executor can still be unsafe.
Risks: driver collection/broadcast timeout, executor OOM, repeated copy on every executor and network load. A forced hint can override a safer cost-based choice.

8. Adaptive Query Execution (AQE)

AQE re-optimizes a Spark SQL physical plan using statistics collected while the query is running. It is enabled by default in modern Spark releases, but verify your actual cluster configuration.

What AQE can do

AQE featureProblemRuntime action
Coalesce shuffle partitionsToo many tiny post-shuffle tasksCombines contiguous small partitions
Skew join optimizationOne partition is much largerSplits skewed partition and may replicate the other side
Join conversionPlanned SMJ but a side is small at runtimeMay convert to broadcast hash join
Local shuffle readerUnnecessary remote shuffle read after conversionReads locally where possible
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")

# Starting shuffle count; AQE can coalesce it downward
spark.conf.set("spark.sql.shuffle.partitions", "200")

result.explain("formatted")

AQE is not magic. It cannot eliminate every shuffle, fix incorrect results or make an oversized broadcast fit. Good filters, partitions, statistics and join logic remain essential.

Definitions

SettingMeaning
spark.driver.memoryJVM heap for the driver
spark.driver.memoryOverheadNon-heap/native/container allowance for driver
spark.executor.memoryJVM heap per executor
spark.executor.memoryOverheadNon-heap/native/Python/container allowance per executor

spark.executor.instancesNumber of executors when static allocation is used
Version-sensitive: Spark computes default overhead using documented factors and minimums, which vary by environment/version. Explicit calculation is clearer for training; check your installed Spark configuration reference.

Worked example

Cluster: 3 worker nodes; each has 16 CPU cores and 64 GB RAM. Reserve 1 core and 4 GB per node for OS/daemons.

  1. Usable per node: 15 cores and 60 GB.
  2. Choose 5 cores per executor: 15 ÷ 5 = 3 executors per node.
  3. Total executors: 3 nodes × 3 = 9.
  4. Container budget per executor: 60 GB ÷ 3 = 20 GB.
  5. Choose executor heap = 16 GB.
  6. Choose memory overhead = 4 GB.
  7. Container total = 16 + 4 = 20 GB.
ConfigurationValueReason
Executors93 per worker × 3 workers
Executor cores5Balanced concurrency per JVM
Executor memory16gHeap
Executor overhead4gPython/native/shuffle safety allowance
Driver memory4gStart modestly; measure plan/result needs
Driver overhead1gNon-heap/container allowance

Useful formulas

Executors per node = floor(usable cores ÷ cores per executor)
Executor container budget = usable node memory ÷ executors per node
Executor container request ≈ executor heap + overhead + off-heap + separate PySpark memory (when explicitly configured)
Total parallel task slots ≈ executors × executor cores

How to adjust after measurement

SymptomLikely adjustment
Executor OOM / Python worker killedIncrease overhead or reduce concurrent cores/data per partition
Long JVM GCReduce objects, avoid huge executor heaps, improve serialization/partitions
Heavy disk spillReduce partition size/skew, add memory if justified
Driver OOMAvoid collect/toPandas; increase driver only after fixing result size
Low CPU utilizationCheck task count, I/O and cores/executor
Never calculate from RAM alone. Also check cluster-manager maximum allocation, queue capacity, dynamic allocation, Python/native memory, concurrent applications and required reliability margin.

With dynamic allocation enabled, --num-executors is not a fixed final count. Configure minimum, initial and maximum executors according to queue capacity and workload.

11. Simple end-to-end retail demo

from pyspark.sql import SparkSession, functions as F

spark = (
    SparkSession.builder
    .appName("RetailSparkOptimizationDemo")
    .config("spark.sql.adaptive.enabled", "true")
    .config("spark.sql.adaptive.coalescePartitions.enabled", "true")
    .config("spark.sql.adaptive.skewJoin.enabled", "true")
    .getOrCreate()
)
spark.sparkContext.setLogLevel("WARN")

orders = spark.read.option("header", True).option("inferSchema", True).csv(
    "data/retail_orders.csv"
)
products = spark.read.option("header", True).option("inferSchema", True).csv(
    "data/products.csv"
)

# Native functions, early filter and projection
filtered = (
    orders
    .filter((F.to_date("order_date") == F.lit("2026-07-17")) &
            (F.col("status") == "DELIVERED"))
    .select("product_id", F.col("amount").cast("double").alias("amount"))
)

# Reduce rows before joining
by_product = filtered.groupBy("product_id").agg(
    F.sum("amount").alias("product_revenue")
)

# Small dimension broadcast
result = (
    by_product
    .join(F.broadcast(products.select("product_id", "category")),
          "product_id")
    .groupBy("category")
    .agg(F.sum("product_revenue").alias("revenue"))
    .orderBy(F.desc("revenue"))
)

result.explain("formatted")
result.show(truncate=False)
spark.stop()

What was optimized?

CodeOptimization
to_date, sumNative functions instead of UDF
Filter/select earlyFewer rows and columns
Group by product before joinPre-aggregation reduces join input
broadcast(products)Avoids large-side shuffle when safe
AQE settingsRuntime partition/join/skew adaptation

13. Best practices and edge cases

  • Measure before and after using the same data and output validation.
  • Prefer DataFrame/SQL built-ins.
  • Filter and select columns before wide operations.
  • Broadcast only verified small relations.
  • Keep AQE enabled unless evidence shows a regression.
  • Use realistic statistics and representative data.
  • Watch skew and file/partition size.
  • Avoid collect() and toPandas() for large results.
  • Do not cache every DataFrame; cache only reused expensive results.
  • Tune memory and cores together.

Edge cases

CaseCaution
Tiny local demoJoin/shuffle differences may be invisible
Stale statisticsInitial join choice may be poor
Highly skewed keyAverage partition size hides one huge task
Python-heavy processingMemory overhead may matter more than JVM heap
Many concurrent appsSingle-job node calculation overcommits the queue
