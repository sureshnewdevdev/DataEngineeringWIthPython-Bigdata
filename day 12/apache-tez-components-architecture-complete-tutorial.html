1. What is Apache Tez?

Apache Tez is a distributed data-processing framework that runs on YARN. It executes work as a Directed Acyclic Graph (DAG) made of connected processing stages called vertices.

Simple meaning: Hive converts a SQL query into a plan. Tez turns that plan into connected tasks, requests containers from YARN, moves data between stages and monitors execution.
QuestionAnswer
What is it?A DAG execution engine on YARN.
Why use it?To reduce repeated startup, disk writes and unnecessary stages compared with classic MapReduce.
Who uses it?Hive, Pig and custom Hadoop applications.
Where does it run?ApplicationMaster and tasks run inside YARN containers.
Memory aid: Hive decides what the query means; Tez decides how its DAG runs; YARN decides where resources are allocated; HDFS stores the data.

3. Client-side process

The client builds the DAG, submits it to YARN and observes its progress. For Hive, the client-facing path may start through Beeline/HiveServer2, but the Tez client libraries perform DAG submission.

YARNCluster resource managerLaunches the ApplicationMaster container

What the client does step by step

  1. Receives or creates a logical query plan.
  2. Builds a Tez DAG containing vertices and edges.
  3. Adds configuration, libraries and security credentials.
  4. Submits the application to YARN.
  5. Receives an application ID.
  6. Polls progress and displays logs/status.
The client does not process the complete large dataset locally. It prepares and submits work; distributed containers perform it.

4. Tez ApplicationMaster inside a YARN container

The Tez ApplicationMaster (AM) is the brain of one Tez application. YARN launches it in a container. It understands the DAG, requests task containers and tracks every vertex and task.

5. Tez task inside a YARN container

A task container is the worker area. Each task processes a portion of the data. Many containers can run in parallel across cluster nodes.

ComponentMeaningWork
Tez Task HostContainer-side host processReceives and launches assigned task attempts

Tez Task ServicesRuntime supportInputs, outputs, counters, memory, events
FileSystemStorage connectionRead HDFS blocks and write output
YARNContainer environmentCPU/memory isolation and node allocation
Container reuse: Tez can reuse containers for later vertices. This avoids repeatedly starting a new JVM, which is one reason multi-stage Hive queries can run faster than classic MapReduce.

  1. User submits HiveQL through Beeline.
  2. Hive parses, validates and optimizes it.
  3. Hive creates a Tez DAG.
  4. The client submits the Tez application to YARN.
  5. YARN launches the Tez ApplicationMaster.
  6. The AM initializes the DAG and requests containers.
  7. Task containers read partitions from HDFS.
  8. Vertices execute filters, joins and aggregations.
  9. Edges move intermediate data: one-to-one, broadcast or scatter-gather shuffle.
  10. The final vertex writes output; the AM reports success and releases resources.

8. Simple retail query as a Tez DAG

Requirement: calculate delivered revenue per category for 17 July 2026.

SELECT p.category, SUM(o.amount) AS revenue
FROM sales_orders o
JOIN products p ON o.product_id = p.product_id
WHERE o.order_date = '2026-07-17'
  AND o.status = 'DELIVERED'
GROUP BY p.category;

Step 1 — Check services

Step 2 — Connect using Beeline

beeline -u jdbc:hive2://localhost:10000/default -n vmuser

Step 3 — Select Tez engine

SET hive.execution.engine=tez;
SET hive.execution.engine;

-- Useful optimization settings
SET hive.cbo.enable=true;
SET hive.vectorized.execution.enabled=true;
SET hive.auto.convert.join=true;
SET hive.exec.parallel=true;

Step 4 — Explain and run

USE retail_optimization_db;

EXPLAIN FORMATTED
SELECT status, COUNT(*) AS order_count, SUM(amount) AS total
FROM sales_orders
WHERE order_date='2026-07-17'
GROUP BY status;

SELECT status, COUNT(*) AS order_count, SUM(amount) AS total
FROM sales_orders
WHERE order_date='2026-07-17'
GROUP BY status;

Step 5 — Monitor

yarn application -list
yarn application -list -appStates FINISHED,FAILED,KILLED

# Replace with the real application ID
yarn logs -applicationId application_XXXXXXXXXXXX_XXXX
Environment note: Tez libraries and tez-site.xml must be correctly installed and available to Hive/Hadoop. Merely setting hive.execution.engine=tez cannot install Tez.

Key point: Tez does not replace YARN or HDFS. It replaces the execution engine used to coordinate the processing graph.

11. Best practices and limitations

Best practices

  • Keep table and column statistics current.
  • Use ORC, partition pruning and vectorization.
  • Size AM/task memory using evidence, not guesses.
  • Use map joins only for truly small dimensions.
  • Monitor skew, failed attempts, garbage collection and shuffle.
  • Place users into appropriate YARN queues.
  • Use Tez UI/YARN logs along with EXPLAIN FORMATTED.

Limitations

  • Installation and version compatibility require care.
  • Incorrect memory values can cause container kills or waste.
  • Data skew can still create slow tasks.
  • Tez improves execution but cannot repair poor partitioning, many tiny files or incorrect SQL.
  • Cluster security and monitoring configuration add operational complexity.
