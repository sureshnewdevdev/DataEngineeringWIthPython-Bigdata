1. Introduction

A data integration pipeline reads from one system, applies processing rules and writes a verified result to another system. In this tutorial the storage systems are HDFS, MySQL and Hive, while PySpark is the distributed processing layer placed between them.

HDFS stores large files across cluster machines. MySQL stores structured operational data in tables and supports transactions, indexes and constraints. Hive provides a catalog-oriented warehouse experience: a table name is linked to schema, storage format, location and partition metadata. Spark can interact with all three through a common DataFrame API.

A reliable integration is more than read followed by write. It must validate schema, isolate rejected rows, protect credentials, control parallel database connections, handle retries and reconcile source and target counts. Skipping these controls can create duplicate records, missing data, overloaded databases or unusable small files.

Without a standard pattern
Manual exports, hard-coded secrets, hidden filters, duplicate retries and no proof that the target is complete.
With a standard pattern
Reusable DataFrame logic, safe connectivity, controlled writes, audit counts and predictable recovery.
MEMORY HOOK: R-T-V-W-R — Read, Transform, Validate, Write, Reconcile.
Check yourself: Is PySpark only a file-copy tool?

No. PySpark is used for distributed cleansing, joining, aggregation, validation and publishing. Copying is only one possible operation.

2. Foundation-Friendly Explanation

Imagine a large logistics hub. Trucks arrive from a distributed storage yard, an operational shop database or a warehouse catalog. The hub unloads the goods, inspects labels, removes damaged items, repacks valid items and sends them to a destination.

HDFS
The distributed storage yard containing large batches of files.
MySQL
The organized operational store with named tables, keys and permissions.
Hive
The warehouse catalog telling users what data exists, its schema and location.
PySpark
The parallel sorting hub where many workers process partitions.
  1. Read: source files or database rows become Spark DataFrame partitions.
  2. Transform: invalid values are cleaned, business fields are derived and datasets may be joined.
  3. Validate: required keys, types, duplicates and row counts are checked.
  4. Write: Spark creates HDFS files, sends JDBC batches or registers Hive tables.
  5. Reconcile: the operator proves that source = valid + rejected and that valid = target for the intended load.
Where the analogy breaks
Spark transformations are lazy, tasks can retry, database writes can be transactional, and data partitions move over a network. Physical goods do not behave this way.

Remember it as: the source owns the input, Spark owns processing, and the target owns the published result.

3. Technical Architecture and Key Terms

Technically, a Spark application uses a data-source connector to construct a distributed DataFrame. Transformations create a logical plan. An action or write causes Spark to optimize the plan, schedule stages and execute tasks against partitions. The sink writer then commits files, JDBC batches or catalog-managed output.

DataFrame
A distributed table-like dataset with named columns and a schema.
Partition
A unit of data processed by one Spark task at a time.
JDBC driver
A JVM library used to connect Spark to MySQL.
Hive metastore
A catalog storing table, schema, partition and location metadata.
Predicate pushdown
Sending eligible filters to the source so fewer rows move to Spark.
Idempotency
A retry produces the correct final state without duplicate side effects.
Reconciliation
Comparing source, rejected and target measures.
Schema contract
The approved columns, types and validation rules for a dataset.

4. Simple Example — HDFS to HDFS

from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("minimal-integration").getOrCreate()

orders = spark.read.parquet("hdfs:///training/incoming/orders")

curated = (
    orders
    .filter(F.col("status") == "PAID")
    .withColumn("tax", F.round(F.col("amount") * 0.18, 2))
    .withColumn("net_amount", F.col("amount") + F.col("tax"))
)

curated.write.mode("overwrite").parquet(
    "hdfs:///training/curated/orders"
)

verified = spark.read.parquet(
    "hdfs:///training/curated/orders"
).count()

print("Verified rows:", verified)
spark.stop()

Line-by-line reasoning

  1. SparkSession is the entry point for DataFrame work.
  2. hdfs:/// uses the Hadoop default filesystem configuration available to Spark.
  3. The filter and calculated columns build a lazy logical plan.
  4. The write triggers execution and creates Parquet part files.
  5. The read-back count checks the committed destination rather than only trusting the in-memory DataFrame.
Expected result: Only PAID orders are written. Each row includes tax and net_amount. The verified count equals the intended output count.

5. Real-World Retail Scenario

A retailer receives hourly order files in HDFS. Customer and current order-status data live in MySQL. Analysts query historical sales through Hive. The engineering team wants the same validation and transformation rules across all source-target combinations.

Each run receives a run ID. The job validates order ID, customer ID and amount, calculates tax and net amount, writes rejected records separately and publishes valid records to HDFS, MySQL or Hive. Counts and duration are stored for operations.

7. HDFS → PySpark → HDFS

Choose this pattern for raw-to-standardized, standardized-to-curated or aggregate file processing inside a Hadoop data lake.

from pyspark.sql import SparkSession, functions as F, types as T

spark = SparkSession.builder.appName("hdfs-to-hdfs").getOrCreate()

schema = T.StructType([
    T.StructField("order_id", T.LongType(), False),
    T.StructField("customer_id", T.LongType(), False),
    T.StructField("order_ts", T.TimestampType(), True),
    T.StructField("amount", T.DecimalType(12, 2), True),
    T.StructField("status", T.StringType(), True),
])

source = (spark.read.option("header", True)
          .schema(schema)
          .csv("hdfs:///training/incoming/orders_csv"))

valid_condition = (
    F.col("order_id").isNotNull()
    & F.col("customer_id").isNotNull()
    & F.col("amount").isNotNull()
    & (F.col("amount") >= 0)
)

valid = source.filter(valid_condition)
rejected = source.filter(~valid_condition).withColumn(
    "rejection_reason", F.lit("Missing key/amount or negative amount")
)

curated = (valid
    .withColumn("order_date", F.to_date("order_ts"))
    .withColumn("tax", F.round(F.col("amount") * 0.18, 2))
    .withColumn("net_amount", F.col("amount") + F.col("tax")))

(curated.repartition("order_date")
 .write.mode("overwrite")
 .partitionBy("order_date")
 .parquet("hdfs:///training/curated/orders"))

rejected.write.mode("overwrite").json(
    "hdfs:///training/rejected/orders"
)

print("source", source.count())
print("valid", valid.count())
print("rejected", rejected.count())
Why Parquet?
It is columnar, compressed and schema-aware. Analytical queries can read only required columns and partitions.
Small-file risk
Too many Spark partitions create too many output files. Tune partition count based on volume and target file size instead of using a random number.

8. MySQL → PySpark → MySQL

Use this pattern for batch enrichment, aggregation or publishing from an operational source table to a reporting or serving table.

import os
from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("mysql-to-mysql").getOrCreate()
url = os.environ["MYSQL_URL"]
user = os.environ["MYSQL_USER"]
password = os.environ["MYSQL_PASSWORD"]

orders = (spark.read.format("jdbc")
    .option("url", url)
    .option("dbtable", "source_orders")
    .option("user", user)
    .option("password", password)
    .option("driver", "com.mysql.cj.jdbc.Driver")
    .option("partitionColumn", "order_id")
    .option("lowerBound", "1001")
    .option("upperBound", "1005")
    .option("numPartitions", "2")
    .option("fetchsize", "1000")
    .load())

run_id = "retail-20260714-001"
curated = (orders.filter(F.col("status") == "PAID")
    .withColumn("order_date", F.to_date("order_ts"))
    .withColumn("tax", F.round(F.col("amount") * 0.18, 2))
    .withColumn("net_amount", F.col("amount") + F.col("tax"))
    .withColumn("load_run_id", F.lit(run_id))
    .select("order_id", "customer_id", "order_date", "amount",
            "tax", "net_amount", "load_run_id"))

# Control the number of simultaneous database writers.
curated = curated.coalesce(2)

(curated.write.format("jdbc")
    .option("url", url)
    .option("dbtable", "curated_orders")
    .option("user", user)
    .option("password", password)
    .option("driver", "com.mysql.cj.jdbc.Driver")
    .option("batchsize", "1000")
    .mode("append")
    .save())
COMMON CONFUSION: JDBC lowerBound and upperBound determine partition stride. They do not automatically filter all rows to that range. Use a subquery or predicate for a required business window.
Production retry pattern: write to staging → validate → merge/upsert by key → record run ID → mark the run complete.

9. Hive → PySpark → Hive

Use this pattern when both source and target are warehouse datasets registered in a Hive-compatible catalog.

from pyspark.sql import SparkSession, functions as F

spark = (SparkSession.builder
    .appName("hive-to-hive")
    # Use this when the platform requires an external metastore URI:
    # .config("hive.metastore.uris", "thrift://metastore-host:9083")
    .enableHiveSupport()
    .getOrCreate())

spark.sql("CREATE DATABASE IF NOT EXISTS retail_dw")
source = spark.table("retail_raw.orders")

daily_sales = (source.filter(F.col("status") == "PAID")
    .withColumn("order_date", F.to_date("order_ts"))
    .groupBy("order_date")
    .agg(F.countDistinct("order_id").alias("order_count"),
         F.round(F.sum("amount"), 2).alias("gross_sales")))

(daily_sales.write
    .mode("overwrite")
    .format("parquet")
    .saveAsTable("retail_dw.daily_sales"))

spark.sql("""
SELECT order_date, order_count, gross_sales
FROM retail_dw.daily_sales
ORDER BY order_date
""").show()

Writing to an existing table

# insertInto uses the target table definition and positional column order.
ordered = daily_sales.select("order_date", "order_count", "gross_sales")
ordered.write.mode("append").insertInto("retail_dw.daily_sales")
WATCH OUT: With insertInto, select columns in the exact target order. Compatible types in the wrong order can produce semantically incorrect data.

10. HDFS → PySpark → MySQL

This pattern publishes a smaller, query-friendly serving dataset from a large HDFS history into MySQL.

import os
from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("hdfs-to-mysql").getOrCreate()
raw = spark.read.parquet("hdfs:///training/curated/orders")

serving = (raw.filter(F.col("status") == "PAID")
    .groupBy("customer_id")
    .agg(F.countDistinct("order_id").alias("paid_order_count"),
         F.round(F.sum("net_amount"), 2).alias("lifetime_value")))

# A bounded result should not open hundreds of database connections.
serving = serving.coalesce(4)

(serving.write.format("jdbc")
    .option("url", os.environ["MYSQL_URL"])
    .option("dbtable", "customer_value")
    .option("user", os.environ["MYSQL_USER"])
    .option("password", os.environ["MYSQL_PASSWORD"])
    .option("driver", "com.mysql.cj.jdbc.Driver")
    .option("batchsize", "1000")
    .mode("overwrite")
    .save())
Why aggregate first?
HDFS is suitable for large detailed history. MySQL should usually receive a bounded serving result rather than every raw event.
SELECT COUNT(*) AS customer_rows,
       SUM(lifetime_value) AS total_value
FROM retail_training.customer_value;

11. MySQL → PySpark → Hive

This pattern extracts operational records from MySQL and builds a historical analytical table in Hive.

import os
from pyspark.sql import SparkSession, functions as F

spark = (SparkSession.builder
    .appName("mysql-to-hive")
    .enableHiveSupport()
    .getOrCreate())

source_query = """
(
  SELECT order_id, customer_id, order_ts, amount, status
  FROM source_orders
  WHERE order_ts >= '2026-07-01'
    AND order_ts <  '2026-08-01'
) monthly_orders
"""

orders = (spark.read.format("jdbc")
    .option("url", os.environ["MYSQL_URL"])
    .option("dbtable", source_query)
    .option("user", os.environ["MYSQL_USER"])
    .option("password", os.environ["MYSQL_PASSWORD"])
    .option("driver", "com.mysql.cj.jdbc.Driver")
    .option("fetchsize", "1000")
    .load())

warehouse = (orders
    .withColumn("order_date", F.to_date("order_ts"))
    .withColumn("order_month", F.date_format("order_ts", "yyyy-MM")))

spark.sql("CREATE DATABASE IF NOT EXISTS retail_dw")

(warehouse.write
    .mode("append")
    .format("parquet")
    .partitionBy("order_month")
    .saveAsTable("retail_dw.orders"))
Remember it as: extract narrowly from the operational database, process in Spark and publish broadly for analytical use.

HDFS → Spark → HDFSData-lake refinementHigh-throughput distributed file processingSmall files and weak schema governance
MySQL → Spark → MySQLDatabase enrichment and servingDirect table integrationConnection overload and duplicate retries
Hive → Spark → HiveWarehouse transformationsCatalog discovery and SQL accessOverwrite, partition and metastore mistakes
HDFS → Spark → MySQLPublish aggregated serving dataLarge-scale preparation before database loadSending too much detail to MySQL
MySQL → Spark → HiveOperational-to-analytical ingestionHistorical warehouse creationHeavy extraction load on source MySQL
Selection rule: source = authoritative location; target = consumption pattern. Use HDFS for large analytical files, MySQL for bounded operational serving and Hive for catalog-driven SQL analytics.

13. Best Practices

Do

  • Use explicit file schemas. Why: avoids repeated inference and silent type changes.
  • Select only required columns. Why: reduces I/O, network and memory.
  • Push JDBC filters to MySQL. Why: fewer rows leave the database.
  • Limit database write partitions. Why: protects connection and transaction capacity.
  • Use run IDs and audit counts. Why: every load becomes traceable.
  • Design an idempotent retry. Why: distributed jobs and networks can fail.
  • Read back the destination. Why: verifies committed output rather than planned output.

Avoid

  • Hard-coded passwords. They leak through source control and logs.
  • collect() on large data. It can exhaust driver memory.
  • Uncontrolled overwrite. It can replace an entire path or table.
  • Hundreds of JDBC writers. They can overload MySQL.
  • High-cardinality Hive partitions. They create many tiny directories/files.
  • Assuming append is retry-safe. It can duplicate records.
PRO TIP: Log source count, valid count, rejected count, target count, run ID, duration and output location/table for every execution.

    21. Summary and Quick Revision

    • Use R-T-V-W-R for every pipeline: Read, Transform, Validate, Write, Reconcile.
    • HDFS integrations need schema control, efficient formats and sensible file counts.
    • MySQL integrations need Connector/J, protected secrets, filter pushdown, limited connections and batch writes.
    • Hive integrations need Hive support, correct metastore/catalog configuration and careful table/partition behavior.
    • Retries require staging, run IDs, stable keys and an idempotent target strategy.
    • Always verify the physical destination and log operational metrics.
    One-line cheat list: files scale storage, MySQL serves bounded relational results, Hive makes analytical data discoverable, and Spark performs the distributed work.
