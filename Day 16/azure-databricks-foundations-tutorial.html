Create a workspace: portal flow

Azure Portal → Create a resource → Azure Databricks.
Choose subscription and create/select a resource group.
Enter workspace name and choose a supported region.
Choose pricing tier and review networking requirements.
Select Review + create, then Launch Workspace.
Quota note: Classic compute uses Azure VM quota. A successful workspace deployment does not guarantee that every VM family can be provisioned.

4. Databricks architecture

Azure Databricks separates management services from the place where data processing occurs.

Control plane — Databricks-managed
Web UI · REST APIs · workspace metadata · job orchestration · compute management
secure control communication ↓
Compute plane
Classic: compute resources in your Azure subscription
Serverless: Databricks-managed serverless compute plane
reads / transforms / writes ↓
Data layer
ADLS Gen2 · Delta tables · Unity Catalog external locations · Azure services

Inside a Spark compute environment

Driver
Builds plan, coordinates work, collects metadata
Workers / executors
Run tasks on partitions and cache data
Storage
Source and result data remain durable outside compute

Request flow

  1. User runs a notebook cell.
  2. The control plane sends execution instructions to attached compute.
  3. The Spark driver creates jobs, stages and tasks.
  4. Executors process partitions, reading from ADLS.
  5. Results appear in the notebook; persistent output is written to Delta/ADLS.
Control plane controls. Compute plane computes. Data Lake stores.

5. Databricks compute types

ComputeBest forMain behavior
Serverless notebook computeInteractive Python/SQL notebooksFast startup, automatic scaling, minimal infrastructure management.
Classic all-purpose computeInteractive engineering/data-science needs requiring customizationAttach several notebooks; user controls VM/runtime/autoscaling within policy.
Jobs computeScheduled production tasksCreated for job runs and terminated after; stronger isolation and cost attribution.
SQL warehouseSQL editor, dashboards, BI and reportingSQL-optimized; serverless, pro and classic warehouse types may be available.
Serverless pipeline computeLakeflow pipelinesManaged provisioning and scaling for declarative pipelines.

Legacy terms you may still hear

Older/classroom termCurrent practical meaning
Interactive clusterAll-purpose/classic interactive compute
Automated clusterJob compute
High-concurrency clusterUse appropriate access mode, serverless compute, or SQL warehouse depending on workload

Selection guide

Notebook exploration

Prefer serverless when supported; choose classic if you need unsupported APIs, libraries or configuration.

Scheduled ETL

Use jobs compute or serverless jobs. Avoid keeping interactive compute running.

Power BI / SQL report

Use a SQL warehouse, normally serverless where available.

RDD requirement

Use classic all-purpose compute because RDD APIs are not supported in Spark Connect serverless mode.

Simple classroom configuration

Classic compute example: LTS runtime, small node type, single node or 1 worker, auto-termination 15 minutes, Photon only if needed, no public secrets in Spark config.

Why compute may fail to start

  • Regional vCPU quota exhausted.
  • VM SKU unavailable or restricted by policy.
  • User lacks unrestricted compute creation permission.
  • Subnet/IP capacity is insufficient.
  • Azure policy denies the resource configuration.
  • Library/init script conflicts with runtime.

dbutils.widgets.dropdown("write_mode", "overwrite",
                         ["overwrite", "append"], "Write mode")

environment = dbutils.widgets.get("environment")
write_mode = dbutils.widgets.get("write_mode")
print(environment, write_mode)

Temporary view

df_customer.createOrReplaceTempView("customer_bronze_vw")

# Use the same Spark session
display(spark.sql("""
  SELECT Country, COUNT(*) AS customer_count
  FROM customer_bronze_vw
  GROUP BY Country
  ORDER BY customer_count DESC
"""))
Scope: A temporary view belongs to the Spark session. A persisted Delta table/view is needed when separate jobs or sessions must reliably share the result.

storage_key = dbutils.secrets.get(

    key="storage-account-key"

# Chain notebooks

    "/Workspace/Users/user@example.com/customer_Silver",

Security: Do not print, return, log or concatenate secret values. If a key is exposed, rotate it; deleting a chat message is not credential rotation.

9. Azure Data Lake Storage

ADLS Gen2 is Azure Blob Storage with hierarchical namespace enabled, giving directory semantics and fine-grained access control suitable for analytics.

Storage typeEndpointSpark URI
Azure Blob Storageaccount.blob.core.windows.netwasbs://container@account.blob.core.windows.net/path
ADLS Gen2account.dfs.core.windows.netabfss://container@account.dfs.core.windows.net/path

Recommended production pattern

Use Unity Catalog with an external location and a managed identity through an Azure Databricks Access Connector. This avoids embedding or rotating storage keys in notebooks.

Training-only key pattern

storage_account_name = "newstoreforsession"
container_name = "datasource"

storage_key = dbutils.secrets.get(
    scope="azure-storage",
    key="storage-account-key"
)

spark.conf.set(
    f"fs.azure.account.key.{storage_account_name}.blob.core.windows.net",
    storage_key
)

source_path = (
    "wasbs://datasource@newstoreforsession.blob.core.windows.net/"
    "customers-100.csv"
)

df_customer = (spark.read
    .option("header", True)
    .option("inferSchema", True)
    .csv(source_path))

display(df_customer)
Account keys grant broad access. Use them only for controlled learning when identity-based access is unavailable. Prefer managed identity/Unity Catalog for real projects.

10. End-to-end lab: Customer Medallion pipeline

Bronze
Raw + ingestion metadata
Silver
Clean + validate + deduplicate
Gold
Country/monthly report

A. Bronze

from pyspark.sql import functions as F

bronze_path = "wasbs://datasource@newstoreforsession.blob.core.windows.net/bronze/customer"

df_bronze = (spark.read.option("header", True).option("inferSchema", True)
             .csv(source_path)
             .withColumn("_source_file", F.input_file_name())
             .withColumn("_ingested_at", F.current_timestamp()))

df_bronze.createOrReplaceTempView("customer_bronze_vw")

(df_bronze.write.format("delta").mode("overwrite")
 .option("overwriteSchema", True).save(bronze_path))

B. Silver cleansing

silver_path = "wasbs://datasource@newstoreforsession.blob.core.windows.net/silver/customer"

df_silver = spark.sql("""
SELECT
  trim(`Customer Id`)                    AS customer_id,
  initcap(trim(`First Name`))            AS first_name,
  initcap(trim(`Last Name`))             AS last_name,
  initcap(trim(City))                    AS city,
  initcap(trim(Country))                 AS country,
  lower(trim(Email))                     AS email,
  to_date(`Subscription Date`)           AS subscription_date,
  _source_file,
  _ingested_at
FROM customer_bronze_vw
WHERE `Customer Id` IS NOT NULL
  AND Email RLIKE '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+[.][A-Za-z]{2,}$'
""").dropDuplicates(["customer_id"])

df_silver.createOrReplaceTempView("customer_silver_vw")
(df_silver.write.format("delta").mode("overwrite")
 .option("overwriteSchema", True).save(silver_path))

C. Gold analysis report

gold_path = "wasbs://datasource@newstoreforsession.blob.core.windows.net/gold/customer_country_report"

df_gold = spark.sql("""
SELECT
  country,
  COUNT(DISTINCT customer_id) AS total_customers,
  COUNT(DISTINCT city)        AS cities_covered,
  MIN(subscription_date)      AS first_subscription,
  MAX(subscription_date)      AS latest_subscription
FROM customer_silver_vw
GROUP BY country
ORDER BY total_customers DESC
""")

df_gold.createOrReplaceTempView("customer_country_report_vw")
(df_gold.write.format("delta").mode("overwrite")
 .option("overwriteSchema", True).save(gold_path))

display(df_gold)

D. Report in SQL

%sql
SELECT country, total_customers
FROM customer_country_report_vw
ORDER BY total_customers DESC
LIMIT 10
Trainer discussion: Bronze preserves traceability. Silver improves quality. Gold answers a business question. Delta is the storage format; a temporary view is a session-level query object.

11. Cost, performance and security

AreaGood practice
Compute costUse serverless or job compute appropriately; auto-terminate classic compute.
Data layoutUse Delta; avoid unnecessary tiny files; partition only when justified by size and query filters.
SecurityPrefer managed identity and Unity Catalog external locations.
SecretsUse secret scopes; never store plaintext credentials in code or Git.
GovernanceUse catalogs, schemas, grants, lineage and least privilege.
ObservabilityMonitor job runs, query history, Spark UI, logs and cost tags.

Do / Avoid

Do

  • Use LTS runtimes for stability.
  • Test with representative data.
  • Use job retries carefully.
  • Parameterize paths and environment.

Avoid

  • Leaving classic compute running.
  • Collecting large data to the driver.
  • Using access keys in production.
  • Assuming temp views persist forever.

12. Troubleshooting guide

SymptomLikely causeFix
403 storage errorIdentity lacks data-plane permission or ACLAssign correct RBAC/ACL; verify container and identity.
Path does not existContainer repeated in path or wrong endpointParse account, container and relative file separately.
CONFIG_NOT_AVAILABLE / secret errorWrong scope/key name or permissionRun databricks secrets list-secrets azure-storage; use harmless key name.
Compute does not startQuota, SKU, network or policyReview event log; try permitted small SKU; check quota/admin policy.
RDD not supportedSpark Connect/serverless limitationUse DataFrame APIs or compatible classic compute.
Temp view not foundNew Spark session/notebook jobRead persisted Delta and recreate the temp view.
Many output CSV filesSpark writes one part per partitionUse Delta; for tiny export only, reduce partitions intentionally.

13. Interview questions and answers

1. What is Azure Databricks?

A managed analytics platform on Azure built around Apache Spark and the lakehouse model, supporting engineering, analytics, ML and governance.

2. What is the difference between control and compute planes?

The control plane hosts management services and the UI. The compute plane runs workloads and processes data.

3. Classic vs serverless compute?

Classic compute resources run in the customer’s Azure subscription and allow more customization. Serverless compute is managed by Databricks with automatic provisioning and scaling.

4. All-purpose vs jobs compute?

All-purpose supports interactive development and shared exploration; jobs compute is created for automated runs and normally terminates afterward.

5. What does the Spark driver do?

It builds execution plans, coordinates executors, schedules tasks and returns results.

6. What is autoscaling?

It increases or decreases worker count within configured bounds according to workload demand.

7. Why use auto-termination?

To stop idle classic compute and reduce unnecessary VM and DBU charges.

8. What is Databricks Runtime?

A tested distribution containing Spark, Delta Lake and optimized libraries/components.

9. What is Photon?

A Databricks vectorized execution engine that accelerates supported SQL and DataFrame workloads.

10. What is a temporary view?

A session-scoped SQL name over a DataFrame; it does not itself persist the data.

11. Why use Delta rather than CSV for Silver?

Delta adds a transaction log, ACID behavior, schema controls, efficient updates and reliable table operations.

12. What is ADLS Gen2?

Azure Blob Storage with hierarchical namespace, supporting directory semantics and fine-grained access for analytics.

13. Difference between wasbs and abfss?

wasbs targets Blob endpoints; abfss targets the ADLS Gen2 DFS endpoint.

14. What is dbutils.secrets?

A notebook utility for retrieving secret values from a secret scope without hardcoding them.

15. Best production approach for ADLS access?

Unity Catalog external locations with a managed identity/Access Connector, governed by least-privilege grants.

16. What is a SQL warehouse?

SQL-optimized compute for queries, dashboards and BI connectivity.

17. Why can compute creation be missing?

The user may lack entitlement, unrestricted cluster creation permission, or access through a compute policy.

18. Why should secrets never be printed?

Notebook output, driver logs and screenshots may expose them; redaction is not a substitute for safe handling.
