3. Concepts you must understand first

DataFrame

A distributed table with named columns. Transformations such as withColumn build a plan; actions such as count execute it.

Delta format

A reliable table format on data-lake storage. It adds transaction logs, schema handling and dependable reads/writes over Parquet data.

Temporary view

A SQL name for a DataFrame in the current Spark session. It is not permanent storage and normally disappears when the session ends.

Rejected records

Records that fail business-quality rules. Keeping them separately makes problems visible and recoverable instead of silently deleting them.

Why use wasbs:// here?

Your source URL is Azure Blob Storage, so the notes use a path such as:

wasbs://datasource@newstoreforsession.blob.core.windows.net/bronze/customer

If the account uses ADLS Gen2 hierarchical namespace, modern projects commonly use abfss://container@account.dfs.core.windows.net/path. Do not mix the Blob endpoint and DFS endpoint blindly.

4. Before running the notebooks

  1. The Bronze notebook must finish successfully.
  2. The secret scope should be azure-storage.
  3. The secret key name should be a safe label such as storage-account-key, not the actual access key.
  4. A Databricks compute resource must be attached and running.
  5. The compute must support Delta Lake.
Security: never type the storage access key directly in a notebook or share it in chat. Retrieve it through dbutils.secrets.get(). If a key has been exposed, regenerate it in Azure Storage.
storage_account_name = "newstoreforsession"
container_name = "datasource"

storage_account_key = dbutils.secrets.get(
    scope="azure-storage",
    key="storage-account-key"
)

spark.conf.set(
    f"fs.azure.account.key.{storage_account_name}.blob.core.windows.net",
    storage_account_key
)

This cell does not download data. It only tells Spark which credential to use when it later accesses that storage account.

5. Silver notebook — explained step by step

Define input and output paths

bronze_input_path = (
    f"wasbs://{container_name}@{storage_account_name}.blob.core.windows.net/"
    "bronze/customer"
)
silver_output_path = (
    f"wasbs://{container_name}@{storage_account_name}.blob.core.windows.net/"
    "silver/customer"
)
silver_rejected_path = (
    f"wasbs://{container_name}@{storage_account_name}.blob.core.windows.net/"
    "silver/customer_rejected"
)

Each Spark write creates a folder containing part files and metadata. The path is therefore a dataset location, not a single filename.

Read Bronze CSV

df_bronze = (
    spark.read.format("csv")
    .option("header", "true")
    .option("inferSchema", "true")
    .load(bronze_input_path)
)
df_bronze.printSchema()
display(df_bronze)

header=true uses the first row as column names. inferSchema=true lets Spark sample the CSV and guess types. For production, an explicit schema is safer because inference can be slow or inconsistent.

Standardize column names

import re
from pyspark.sql import functions as F

def standardize_column_name(name):
    name = name.strip().lower()
    name = re.sub(r"[^a-zA-Z0-9]+", "_", name)
    return name.strip("_")

df_standardized = df_bronze
for old_name in df_bronze.columns:
    df_standardized = df_standardized.withColumnRenamed(
        old_name, standardize_column_name(old_name)
    )

Example: Customer Id becomes customer_id, and Phone 1 becomes phone_1. Consistent names make SQL and PySpark easier.

Validate, separate and deduplicate

The record is considered valid only when customer ID, first name, last name, valid email and subscription date are present. Invalid records receive a human-readable reason.

df_valid_customers = (
    df_validated
    .filter(F.col("_is_valid_record"))
    .dropDuplicates(["customer_id"])
)

df_rejected_customers = (
    df_validated
    .filter(~F.col("_is_valid_record"))
)
Deduplication caution: dropDuplicates(["customer_id"]) keeps an arbitrary matching row. If duplicates have timestamps, use a window and deliberately keep the latest record.

Write valid and rejected Delta datasets

(df_customer_silver.write
    .format("delta")
    .mode("overwrite")
    .option("overwriteSchema", "true")
    .save(silver_output_path))

(df_rejected_customers.write
    .format("delta")
    .mode("overwrite")
    .option("overwriteSchema", "true")

overwrite replaces the dataset on every run. This is fine for a small training demo. Production incremental pipelines usually use append, merge/upsert, checkpoints or managed pipeline features.

Data-quality reconciliation

bronze_count = df_bronze.count()
valid_count = df_valid_customers.count()
rejected_count = df_rejected_customers.count()

print("Bronze:", bronze_count)
print("Valid before duplicate impact:", valid_count)
print("Rejected:", rejected_count)
After deduplication, valid + rejected can be less than Bronze because duplicate valid customer IDs were removed. Track duplicate count separately for a perfect reconciliation report.

Gold reads the persisted Silver Delta dataset. Its purpose is not additional raw cleansing; it creates business-friendly detail and aggregation tables.

Gold output 1 — Customer dimension

The dimension provides one reporting-ready row per customer, with a location, month name and year-month label. The attached code uses row_number to create customer_key.

Production caution: a row number is not a stable surrogate key when records change. Production dimensions normally preserve keys using a Delta table and MERGE logic.

Gold output 2 — Country summary

Answers: How many customers, cities and companies exist per country? What percentage of all customers belongs to each country? What are the first and latest subscription dates?

Gold output 3 — Monthly summary

Answers: How many distinct new customers subscribed each month? How many countries and companies were represented?

Gold output 4 — Company summary

Answers: How many customers belong to each company? Across how many countries and cities? What is the subscription date range?

8. Where temporary views fit

The attached notes mainly use DataFrames. You can create temporary views to teach SQL on the same data:

# In Silver notebook
df_bronze.createOrReplaceTempView("customer_bronze_vw")

display(spark.sql("""
SELECT `Customer Id`, `Country`, `Email`
FROM customer_bronze_vw
WHERE `Customer Id` IS NOT NULL
"""))
# In Gold notebook, after reading persisted Silver Delta
df_silver.createOrReplaceTempView("customer_silver_vw")

df_country_report = spark.sql("""
SELECT
  country,
  COUNT(DISTINCT customer_id) AS total_customers,
  COUNT(DISTINCT city) AS total_cities
FROM customer_silver_vw
GROUP BY country
ORDER BY total_customers DESC
""")
Key distinction: a temporary view is a convenient session-level SQL interface. Delta is the persisted file/table format. Gold should not depend on a temporary view created in a different notebook session; it should read Silver Delta and recreate its own view.

10. Common errors and fixes

Path not found
Confirm the container is datasource. Do not repeat it in the file path. The source file path is wasbs://datasource@newstoreforsession.blob.core.windows.net/customers-100.csv, while Bronze is under .../bronze/customer.
Secret does not exist
Run the CLI command that lists secret names in the azure-storage scope. The notebook key must exactly match the displayed safe key name, normally storage-account-key.
Invalid configuration value or authentication failure
Check that the storage account name, Blob endpoint and regenerated access key match. Never use the connection string where Spark expects only the account key.
Cannot resolve column
Run print(df_bronze.columns). The actual CSV headers differ from the assumed headers, or a column was removed. Update transformations and the final select.
Date parsing fails
Display distinct sample date strings. Use the one correct pattern for the source, or tolerant parsing supported by your Databricks runtime. Mixed formats require deliberate handling.
Delta format not found or write fails
Attach a Databricks runtime that supports Delta, verify storage permissions and ensure you are not reading a CSV path with format("delta") or a Delta path with format("csv").
Temporary view not found
The view belongs to a Spark session. Rerun the cell that creates it in the current notebook session, or read the persisted Delta dataset and recreate the view.

12. Interview questions and short answers

1. Why is Bronze kept close to the source?
It preserves raw history for audit, replay and future transformations when business rules change.
2. Why is Silver written in Delta?
Delta provides reliable transactions, schema features and consistent reads/writes on the data lake.
3. What is the difference between a transformation and an action?
Transformations build a logical plan; actions such as count, collect or writes trigger execution.
4. Why keep rejected records?
To support data-quality reporting, correction and replay without losing evidence.
5. Is a temporary view stored permanently?
No. It is session-scoped. Persist data as Delta or register a permanent table for cross-session use.
6. What does overwriteSchema do?
It allows the target Delta schema to be replaced during an overwrite operation. It should be controlled carefully.
7. Why is inferSchema not ideal for production?
It adds a sampling pass and may infer inconsistent types when source values change.
8. What is wrong with arbitrary dropDuplicates?
When duplicate keys contain different values, it does not express which record is authoritative. A window with ordering is more deterministic.
9. Why can the Gold layer not calculate revenue here?
The customer file has no transaction amount, order, quantity or price fields.
10. How would you productionize this pipeline?
Use explicit schemas, identity-based storage access, incremental ingestion, deterministic deduplication, quality metrics, Delta MERGE, orchestration, monitoring and governed tables.
11. DataFrame versus temp view?
A DataFrame is the programmatic distributed dataset API. A temp view assigns it a SQL-queryable session name.
12. Why add ingestion timestamps?
They support lineage, freshness checks, troubleshooting and incremental processing.
13. What is a surrogate key?
A warehouse-generated key used to identify a dimension row independently of its source business key.
14. Why use countDistinct for customer summaries?
It prevents repeated rows for a customer from inflating the business count.
15. What is data reconciliation?
It proves that input records are fully explained by valid, rejected, duplicate and otherwise categorized outputs.
