Hadoop · MapReduce · Word Count

Counting words the distributed way

A step-by-step walkthrough of a real MapReduce job — where the input file lives, what the Mapper and Reducer do, and where the answer comes out. Built around the SmartRide trip dataset.

i The one-sentence idea

MapReduce answers questions about huge amounts of data by splitting the work into small pieces, processing every piece at the same time on many machines, and then combining the answers. Our example question is simple: how many times does each word appear in the trip log?

It runs in three stages. Keep these colours in mind — they follow you through the whole page.

STAGE 1 Map

Read each line, break it into words, and tag every word with the number 1. Turns Chennai completed into (Chennai, 1) and (completed, 1).

▼

STAGE 2 Shuffle & Sort

Hadoop does this automatically. It gathers all identical words together, so every (Chennai, 1) ends up side by side, ready to be added up. This is the "group by" step.

▼

STAGE 3 Reduce

For each group of identical words, add up all the 1s. (Chennai, [1,1,1…]) becomes (Chennai, 8). That final number is the answer.

Mapper picks the group, Reducer does the math. That single rule is the whole flexibility of MapReduce. Change what the Mapper emits and you change what you're grouping by; change the Reducer and you change the calculation (count, sum, average, max…).

1 Where the input file lives

There are two separate storage worlds, and mixing them up is the most common beginner confusion:

StorageWhat it isHow you look inside
Local diskThe normal Linux filesystem on the machinels, cd, cat
HDFSHadoop's distributed filesystem — where jobs read & writehdfs dfs -ls, hdfs dfs -cat

Our input file smartride_trip_words.txt was created on the local disk first, then copied into HDFS so the job could read it. You cannot cd into HDFS — you always pass the full HDFS path to an hdfs dfs command.

Creating the input folder in HDFS and loading the file

# 1. make the input folder inside HDFS (-p builds the whole path) $ hdfs dfs -mkdir -p /smartride/mapreduce/input # 2. copy the local file INTO HDFS $ hdfs dfs -put ~/smartride/smartride_trip_words.txt /smartride/mapreduce/input/ # 3. confirm it landed $ hdfs dfs -ls /smartride/mapreduce/input -rw-r--r-- 1 vmuser supergroup 226 /smartride/mapreduce/input/smartride_trip_words.txt

2 What is mapper.py?

The Mapper is a small program that runs once for every chunk of input. It reads text line by line and emits key–value pairs. For word counting, the key is a word and the value is always 1 — meaning "I saw this word one time."

mapper.pythe "Map" stage
# mapper.py — runs once per line of the input file.
# Its ONLY job: turn each word into the pair (word, 1).

import sys

# sys.stdin = the input text streamed in by Hadoop, line by line
for line in sys.stdin:
    line = line.strip()          # remove leading/trailing spaces + newline
    words = line.split()         # "Chennai completed" -> ["Chennai", "completed"]

    for word in words:
        # print "word <TAB> 1". The TAB separates key from value.
        # Hadoop reads everything before the tab as the KEY (what we group by).
        print(f"{word}\t1")

So the line Chennai completed makes the Mapper print two lines: Chennai 1 and completed 1. Nothing is added up yet — the Mapper only tags.

3 The Shuffle & Sort (Hadoop's free magic)

You don't write any code for this stage — Hadoop does it for you. It takes every pair the Mappers produced and groups them by key, sorting so all identical words sit together. This is exactly what a SQL GROUP BY does.

After Map (unsorted)→After Shuffle (grouped by word)
Chennai 1→Chennai → [1, 1, 1, 1]
completed 1→completed → [1, 1, 1, 1, 1 …]
Hyderabad 1→Hyderabad → [1, 1, 1]
cancelled 1→cancelled → [1, 1]

4 What is reducer.py?

The Reducer receives the grouped keys and collapses each group into a single answer. For word counting, that means adding up all the 1s in a group to get the total count for that word.

reducer.pythe "Reduce" stage
# reducer.py — receives words already SORTED by Hadoop,
# so identical words arrive one after another. We add up their 1s.

import sys

current_word = None   # the word we are currently counting
current_count = 0     # running total for that word

for line in sys.stdin:
    word, count = line.strip().split("\t")   # split "word <TAB> 1"
    count = int(count)

    if word == current_word:
        current_count += count        # same word -> keep adding
    else:
        if current_word is not None:
            print(f"{current_word}\t{current_count}")  # new word -> print the finished total
        current_word = word           # start counting the new word
        current_count = count

# don't forget the very last word
if current_word is not None:
    print(f"{current_word}\t{current_count}")
Why the counting works so simply: because Hadoop already sorted the words, all the Chennai lines arrive in a row. The Reducer just watches for the word to change, and when it does, prints the total it built up.

5 Running the job

Because our Mapper and Reducer are Python (not Java), we launch them with Hadoop Streaming. We tell Hadoop: here are the two scripts, here is the input folder, and here is where to write the output.

The command

$ hadoop jar $HADOOP_HOME/share/hadoop/tools/lib/hadoop-streaming-*.jar \ -files mapper.py,reducer.py \ -mapper "python3 mapper.py" \ -reducer "python3 reducer.py" \ -input /smartride/mapreduce/input \ -output /smartride/mapreduce/output
Important rule: the -output folder must not already exist. Hadoop refuses to overwrite it, to protect your results. Before re-running, delete the old one first:
hdfs dfs -rm -r -f /smartride/mapreduce/output

While it runs, Hadoop prints its progress. A successful job looks like this:

map 0% reduce 0% map 50% reduce 0% map 100% reduce 0% map 100% reduce 100% Job completed successfully

6 Where the output comes out

The results are written back into HDFS, inside the output folder you named. Two files appear there:

FileWhat it holds
_SUCCESSAn empty marker file. It just says "the job finished cleanly." No data.
part-r-00000Your actual answer — every word and its count.

Viewing the result

# list what the job produced $ hdfs dfs -ls /smartride/mapreduce/output -rw-r--r-- 1 vmuser supergroup 0 _SUCCESS -rw-r--r-- 1 vmuser supergroup ... part-r-00000 # print the word counts $ hdfs dfs -cat /smartride/mapreduce/output/part-r-00000

For our 12-line sample file, the counts come out like this:

Chennai 4
Bangalore 3
Hyderabad 3
Mumbai 2
completed 10
cancelled 2
Reading the answer: "completed" appears 10 times and "cancelled" 2 times, so 10 of the 12 trips finished successfully. Chennai is the busiest city with 4 trips. That is the entire point of the job — a grouped count, produced by machines that could just as easily have handled billions of lines.

7 Want to group by something else?

Change one line in the Mapper. Instead of emitting each word, emit the key you want to group by. For example, to count completed vs cancelled per city, join the two words into one key:

mapper.py (city + status version)a custom group-by
for line in sys.stdin:
    city, status = line.strip().split()
    # key becomes "City_status" -> that's the new GROUP BY
    print(f"{city}_{status}\t1")

Same Reducer, new result:

Bangalore_cancelled 1 Bangalore_completed 2 Chennai_completed 4 Hyderabad_cancelled 1 Hyderabad_completed 2 Mumbai_completed 2

The takeaway for students: the Mapper's key is the grouping, the Reducer's logic is the calculation. Master those two ideas and you can express almost any data summary as a MapReduce job.
