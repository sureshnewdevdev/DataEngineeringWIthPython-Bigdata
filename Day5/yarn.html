1. Learning Goal

This tutorial explains YARN in a spoon-feeding style. The goal is to help learners understand how Hadoop runs processing jobs in a distributed cluster.

Trainer Opening Line: Before Hadoop 2.x, MapReduce was doing both job processing and resource management. Hadoop 2.x introduced YARN to separate resource management from processing logic.

2. SmartRide Use Case

Business Story: SmartRide is an Uber/Ola-like company. Every day, it receives millions of trip records from multiple cities. The company wants to process trip data and answer questions like:

  • How many completed trips happened city-wise?
  • Which city generated more trips?
  • Which driver completed more rides?
  • Can this job run on multiple machines?
  • If one machine fails, can the job continue?

Sample Input File: trip_words.txt

Chennai completed
Bangalore completed
Chennai completed
Hyderabad cancelled
Bangalore completed
Chennai completed
Hyderabad completed
Mumbai completed

Expected WordCount Output

Bangalore    2
Chennai      3
Hyderabad    2
Mumbai       1
completed    7
cancelled    1

To run this job in Hadoop 2.x or 3.x, we need YARN.

3. What is YARN?

YARN stands for Yet Another Resource Negotiator.

YARN is the resource management layer of Hadoop. It manages CPU, memory, containers, and application execution across the Hadoop cluster.

Simple Definition

YARN is like a cluster manager. It decides which application gets how much memory and CPU, and on which machine the work should run.

Technical Definition

YARN is a distributed resource management framework introduced in Hadoop 2.x. It separates resource management from data processing, allowing multiple processing engines such as MapReduce, Spark, Tez, and Flink to run on the same Hadoop cluster.

YARN Solves This Problem

Before YARNAfter YARN
Only MapReduce was tightly supportedMultiple engines can run
JobTracker was overloadedResource Manager and Application Master split responsibility
Scalability issuesBetter scalability
Less flexibleMore flexible

4. Classic MapReduce vs YARN

Classic MapReduce Architecture

In Hadoop 1.x, MapReduce had two main daemons:

  • JobTracker - master that handled scheduling, monitoring, and resource management
  • TaskTracker - worker that ran map and reduce tasks
Classic MapReduce / Hadoop 1.x +----------------+ | JobTracker | +----------------+ / | \ / | \ v v v +----------+ +----------+ +----------+ |TaskTracker| |TaskTracker| |TaskTracker| +----------+ +----------+ +----------+ Map/Reduce Map/Reduce Map/Reduce

Problem with Classic MapReduce

  • JobTracker had too many responsibilities.
  • Only MapReduce jobs were naturally supported.
  • Cluster scalability was limited.
  • Resource utilization was not very flexible.

YARN Architecture

YARN / Hadoop 2.x and later +----------------------+ | Resource Manager | +----------------------+ / \ / \ v v +----------------+ +----------------+ | Node Manager | | Node Manager | +----------------+ +----------------+ Containers Containers Each Application has its own Application Master

MasterJobTrackerResource Manager
WorkerTaskTrackerNode Manager
Application ControlJobTrackerApplication Master
Processing EnginesMainly MapReduceMapReduce, Spark, Tez, Flink
ScalabilityLimitedBetter
Resource HandlingFixed slotsFlexible containers

5. Advantages of YARN

1. Better Resource Management

YARN manages memory and CPU more flexibly using containers.

2. Supports Multiple Processing Engines

YARN can run different types of applications:

  • MapReduce
  • Spark
  • Apache Tez
  • Apache Flink
  • Hive on Tez

3. Better Scalability

YARN can handle larger clusters because responsibility is divided between Resource Manager, Node Manager, and Application Master.

4. Better Cluster Utilization

Instead of fixed map/reduce slots, YARN uses containers based on memory and CPU requirements.

5. Fault Tolerance

If a task fails, YARN can retry it. If an Application Master fails, it can be restarted.

Trainer Tip: Explain YARN as a railway platform manager. Different trains are different applications. The platform manager allocates platforms, tracks, and timing.

6. YARN Architecture

Main Components

  • Resource Manager - cluster-level master
  • Node Manager - worker-level manager
  • Application Master - application-level manager
  • Container - resource unit where tasks run
YARN Architecture User submits application | v +----------------------+ | Resource Manager | | Scheduler + App Mgr | +----------------------+ | | Starts Application Master v +----------------------+ | Application Master | +----------------------+ | | Requests containers v +----------------------+ +----------------------+ | Node Manager | | Node Manager | | Container,Container | | Container,Container | +----------------------+ +----------------------+

SmartRide Example

When SmartRide submits a WordCount job:

  1. User submits job.
  2. Resource Manager receives request.
  3. Application Master starts for that job.
  4. Application Master asks Resource Manager for containers.
  5. Node Managers launch containers.
  6. Containers run map and reduce tasks.
  7. Result is stored back into HDFS.

7. Resource Manager

The Resource Manager is the master service of YARN.

Main Responsibilities

  • Accept application submission
  • Allocate cluster resources
  • Coordinate with Node Managers
  • Manage scheduling
  • Start Application Master
  • Track application status

Simple Meaning

Resource Manager is like the head office manager of the cluster. It knows available resources and decides how they should be distributed.

Resource Manager UI

http://localhost:8088

Useful Commands

yarn application -list
yarn node -list
yarn application -status <application_id>

8. Node Manager

Node Manager runs on every worker machine in the Hadoop cluster.

Responsibilities

  • Launch containers
  • Monitor container CPU and memory usage
  • Send heartbeat to Resource Manager
  • Report node health
  • Manage logs for containers

Simple Meaning

Node Manager is like a branch manager. It manages work happening on one specific machine.

Node Manager in SmartRide Job

If SmartRide WordCount runs on three machines, each machine's Node Manager starts containers to process part of the input data.

Check Node Manager

jps

Expected daemon:

NodeManager

9. Application Master

Application Master is created for each application submitted to YARN.

Responsibilities

  • Negotiate resources from Resource Manager
  • Request containers
  • Coordinate tasks
  • Monitor job progress
  • Handle task failures
  • Report final status

Simple Meaning

Application Master is like the project manager for one specific job.

Example

If SmartRide runs one WordCount job, one Application Master manages that job. If another Spark job runs, another Application Master manages that Spark job.

ComponentLevelExample Role
Resource ManagerCluster levelOverall resource boss
Node ManagerMachine levelWorker machine manager
Application MasterApplication levelJob-specific manager

10. Application Submission in YARN

Step-by-Step Flow

  1. User submits application using command or API.
  2. Resource Manager receives application request.
  3. Resource Manager allocates first container for Application Master.
  4. Node Manager launches Application Master.
  5. Application Master registers with Resource Manager.
  6. Application Master requests more containers.
  7. Resource Manager allocates containers on suitable nodes.
  8. Node Managers launch containers.
  9. Tasks run inside containers.
  10. Application Master monitors execution.
  11. Final output is written to HDFS.
  12. Application Master unregisters from Resource Manager.
Application Submission Flow Client | v Resource Manager | | Allocate AM Container v Node Manager | | Launch Application Master v Application Master | | Request task containers v Resource Manager | | Allocate containers v Node Managers | | Run tasks v HDFS Output

Command Example

hadoop jar $HADOOP_HOME/share/hadoop/mapreduce/hadoop-mapreduce-examples-*.jar \
  wordcount \
  /smartride/yarn/input \
  /smartride/yarn/output

11. Node Manager Containers

A container is a bundle of resources allocated by YARN.

Container Includes

  • Memory
  • CPU cores / vcores
  • Environment variables
  • Local resources
  • Command to execute

Simple Meaning

A container is like a small rented room on a worker machine where one task can run with allocated memory and CPU.

Example

Container 1 = 1 GB memory + 1 vcore
Container 2 = 2 GB memory + 2 vcores

Why Containers Are Useful?

  • Flexible resource allocation
  • Better resource utilization
  • Supports different application types
  • Prevents one job from consuming all resources

Important Configuration Examples

yarn.nodemanager.resource.memory-mb
yarn.scheduler.minimum-allocation-mb
yarn.scheduler.maximum-allocation-mb
yarn.nodemanager.resource.cpu-vcores

12. Resource Manager Components

The Resource Manager has two major internal components:

ComponentPurpose
SchedulerAllocates resources to applications
Applications ManagerAccepts job submissions and starts Application Master

1. Scheduler

The Scheduler decides how cluster resources should be shared among applications.

It does not monitor tasks. It only allocates resources.

2. Applications Manager

The Applications Manager accepts submitted applications and negotiates the first container to start Application Master.

Common Mistake: Many beginners think Resource Manager directly runs tasks. Actually, tasks run inside containers launched by Node Managers.

Resource Manager Internal View

Resource Manager | +-- Scheduler | +-- Allocates resources | +-- Applications Manager +-- Accepts applications +-- Starts Application Master

14. Scheduling in YARN

Scheduling decides how cluster resources are distributed between users, teams, and applications.

Why Scheduling Is Needed?

Assume SmartRide has three teams:

  • Data Engineering team
  • Analytics team
  • Machine Learning team

All teams may submit jobs at the same time. Scheduler decides who gets how much resource.

Types of Schedulers

SchedulerPurpose
FIFO SchedulerFirst job gets resources first
Capacity SchedulerProvides guaranteed capacity to queues
Fair SchedulerShares resources fairly among running jobs

Fair Scheduler

Fair Scheduler tries to give each application a fair share of cluster resources over time.

Example: If analytics and ML jobs are running together, Fair Scheduler tries to balance resources between them.

Capacity Scheduler

Capacity Scheduler divides cluster resources into queues. Each queue gets guaranteed capacity.

Example: SmartRide can allocate 50% cluster capacity to production jobs, 30% to analytics, and 20% to development.

Capacity Scheduler Example

root
├── production     50%
├── analytics      30%
└── development    20%

15. Fault Tolerance in YARN

Fault tolerance means the system can continue working even when something fails.

Failure Scenarios

FailureWhat Happens?
Task failsApplication Master retries the task
Container failsNew container may be requested
Node Manager failsResource Manager marks node as unhealthy/dead
Application Master failsResource Manager can restart Application Master
Resource Manager failsHigh Availability setup can use standby RM

Heartbeats

Node Managers send regular heartbeats to Resource Manager.

Node Manager → heartbeat → Resource Manager

If heartbeat stops, Resource Manager assumes the node has failed.

SmartRide Example

If a container processing Chennai trip data fails, Application Master requests another container and reruns that part of the job.

Fault Tolerance Summary

  • Task retry
  • Container retry
  • Application Master restart
  • Node health monitoring
  • Resource Manager HA

17. Interview Questions and Answers

Q1. What is YARN?

YARN is the resource management layer of Hadoop. It manages CPU, memory and application execution across the cluster.

Q2. Why was YARN introduced?

YARN was introduced to separate resource management from MapReduce processing and support multiple engines such as Spark, Tez and Flink.

Q3. What is Resource Manager?

Resource Manager is the master service of YARN. It manages cluster resources and scheduling.

Q4. What is Node Manager?

Node Manager runs on each worker node and manages containers, node resources and health reporting.

Q5. What is Application Master?

Application Master manages one specific application. It negotiates resources and monitors tasks.

Q6. What is a container in YARN?

A container is a resource allocation unit containing memory, CPU and execution environment for a task.

Q7. What are Resource Manager components?

The main components are Scheduler and Applications Manager.

Q8. What is the Scheduler?

Scheduler allocates resources to applications based on scheduling policy.

Q9. What is the Applications Manager?

Applications Manager accepts applications and starts the Application Master.

Q10. Difference between Classic MapReduce and YARN?

Classic MapReduce used JobTracker and TaskTracker. YARN uses Resource Manager, Node Manager and Application Master.

Q11. What is Capacity Scheduler?

Capacity Scheduler provides guaranteed capacity to different queues.

Q12. What is Fair Scheduler?

Fair Scheduler shares resources fairly among applications over time.

Q13. What is YARN fault tolerance?

YARN handles failures by retrying tasks, restarting containers, monitoring nodes, and restarting Application Masters.

Q14. Can Spark run on YARN?

Yes. Spark can run on YARN by using YARN as the cluster manager.

Q15. What is heartbeat in YARN?

Heartbeat is a regular signal sent by Node Manager to Resource Manager to report node health and container status.

Q16. What happens if Node Manager fails?

Resource Manager marks the node as unhealthy or dead, and failed tasks may be rescheduled on other nodes.

Q17. What happens if Application Master fails?

Resource Manager can restart the Application Master based on retry configuration.

Q18. What is the role of Resource Manager UI?

It helps monitor applications, queues, nodes, resource usage and application status.
