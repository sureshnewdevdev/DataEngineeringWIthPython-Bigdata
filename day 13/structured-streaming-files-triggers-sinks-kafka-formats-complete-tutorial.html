1. What is Structured Streaming?

Structured Streaming is Spark’s incremental processing engine. We describe a continuously growing table with the same DataFrame operations used for batch processing. Spark tracks source progress, processes new rows, updates state when required, and sends results to a sink.

Source
files / Kafka
→
readStream
→
Transform
parse / filter / aggregate
→
writeStream
sink + checkpoint

Batch

Reads a bounded dataset once. The input has a known end.

Streaming

Treats new events as additions to an unbounded table. The query maintains progress between triggers.

Memory aid: S–T–O–C

Source → Transform → Output → Checkpoint. If a streaming design omits one of these, ask why.

Micro-batch workflow

  1. A trigger checks for newly available input.
  2. Spark determines the unprocessed file names or Kafka offsets.
  3. Transformations run only for the new input plus any maintained state.
  4. The sink commits output.
  5. The checkpoint records progress and state for recovery.
Exactly-once is an end-to-end property, not a magic label. It depends on replayable sources, checkpoint recovery, deterministic processing, and an idempotent or transactional sink. Kafka’s streaming sink is at-least-once and may contain duplicates; Delta supports transactional streaming writes.

3. File streaming

A file stream watches a directory and incrementally processes files that appear after the query starts or files not yet recorded in its checkpoint. Files should be placed atomically into the directory; repeatedly modifying an already discovered file is not a safe event-ingestion pattern.

4. Trigger intervals

The trigger answers: When should Spark check and process newly available data? It does not change the business transformation.

TriggerPySpark syntaxStops?UseDatabricks guidance
UnspecifiedNo .trigger(...)NoLowest-delay default micro-batchesCan cause frequent cloud listing calls; choose intentionally
Time interval.trigger(processingTime="30 seconds")NoBalance latency and overheadCommon on supported non-serverless compute
One time.trigger(once=True)YesOne micro-batch, then stopDeprecated; use AvailableNow
Available Now.trigger(availableNow=True)YesAll currently available input, possibly multiple batchesRecommended incremental-batch trigger
Continuous processing.trigger(continuous="1 second")NoExperimental OSS low-latency modeNot supported by Databricks

1) Unspecified trigger

PYTHONquery = (
  clean_orders.writeStream
  .format("console")
  .outputMode("append")
  .option("truncate", False)
  .start()
)

Spark starts another micro-batch as soon as it can. If there is no data, it keeps checking. This is simple for demonstrations but can create unnecessary source-listing overhead.

2) Time interval

PYTHONquery = (
  clean_orders.writeStream
  .format("delta")
  .option("checkpointLocation", f"{checkpoints}/orders_30s")
  .trigger(processingTime="30 seconds")
  .start(f"{base}/output/orders_30s")
)

If a batch takes longer than 30 seconds, Spark cannot run overlapping batches for the same query. The next batch begins after the prior batch finishes.

3) One-time vs AvailableNow

PYTHON — historical Once syntax# Syllabus completeness only: deprecated on Databricks.
.trigger(once=True)
PYTHON — recommendedquery = (
  clean_orders.writeStream
  .format("delta")
  .option("checkpointLocation", f"{checkpoints}/orders_available")
  .trigger(availableNow=True)
  .start(f"{base}/output/orders_available")
)
query.awaitTermination()
Once

Attempts to process available input in one micro-batch and stops.

AvailableNow

Consumes everything currently available, can respect per-trigger rate limits across several micro-batches, then stops.

4) Continuous processing

PYTHON — Apache Spark OSS concept# Experimental OSS feature with restricted operations/sinks.
query = df.writeStream     .format("console")     .trigger(continuous="1 second")     .start()
Do not run this on Databricks. Databricks does not support Spark’s continuous-processing trigger. It is also different from a Databricks Job configured to run continuously. For supported ultra-low-latency scenarios, review Databricks real-time mode availability and limitations for your runtime.

7. CSV, JSON and Avro functions

Do not confuse record functions with file readers. from_json, from_csv and from_avro parse a value inside a column—especially useful for Kafka’s binary value. spark.readStream.json/csv/format("avro") reads files.

1) from_csv and to_csv

 ("2001,601,Monitor,2,15000.0",),
 ("2002,602,Chair,4,8000.0",)
], ["value"])

parsed_csv = (

 F.to_csv(F.struct("order_id","customer_id","product","quantity",

3) from_avro and to_avro

PYTHON — Avro binary message round tripimport json

avro_schema = json.dumps({
 "type": "record",
 "name": "Order",
 "namespace": "training.streaming",
 "fields": [
   {"name":"order_id","type":"long"},
   {"name":"customer_id","type":["null","long"],"default":None},

 ]

Use the appropriate registry-aware functions/connector for your platform rather than manually trimming production payloads.

Watermark example

PYTHONwindowed = (
 parsed_orders
 .withWatermark("event_time", "10 minutes")
 .groupBy(F.window("event_time", "5 minutes"), "status")

10. Interview questions and answers

1. What is Structured Streaming?

A Spark engine that incrementally processes unbounded data using DataFrame/Dataset APIs and maintains progress and state through checkpoints.

2. read vs readStream?

read creates a bounded batch DataFrame. readStream creates a streaming DataFrame whose query consumes new input incrementally after start().

3. What is a trigger?

It controls when Spark initiates processing of newly available streaming input.

4. Unspecified vs processingTime?

Unspecified runs micro-batches as quickly as possible; processingTime enforces a minimum scheduling interval to balance latency and overhead.

5. Once vs AvailableNow?

Once runs one micro-batch. AvailableNow drains all currently available data, possibly across rate-limited batches, and is the Databricks recommendation.

6. Is Spark continuous processing supported in Databricks?

No. Spark’s experimental continuous trigger is not supported by Databricks and is distinct from a continuously running Databricks Job.

7. What is a checkpoint?

Fault-tolerant storage for query identity, processed offsets/files, commits and state required for recovery.

8. What are append, update and complete modes?

Append emits final new rows, update emits changed rows, and complete emits the full result table each trigger.

9. Why preserve Kafka metadata?

Topic, partition and offset enable lineage, debugging, replay analysis and duplicate investigation.

10. startingOffsets earliest vs latest?

Earliest reads retained records from the beginning; latest begins after current end offsets. The setting applies only when a new query starts.

11. Why cast Kafka value?

Kafka exposes key/value as binary. Text parsers such as from_json and from_csv need a string representation.

12. from_json vs spark.read.json?

from_json parses JSON inside a column. read.json reads JSON files into rows.

13. from_avro vs format("avro")?

from_avro decodes an Avro binary column; the Avro data source reads Avro container files.

14. Why use Avro with Kafka?

It is compact, schema-driven and supports record evolution, but it requires compatible schema management.

15. foreach vs foreachBatch?

foreach processes individual rows/partitions. foreachBatch exposes each micro-batch as a batch DataFrame, enabling batch writers, MERGE and batch-level idempotency.

16. What is watermarking?

A policy that bounds state for late event-time data and enables finalization/cleanup in supported stateful operations.

17. How do you control Kafka ingestion rate?

Use options such as maxOffsetsPerTrigger, then measure batch duration, lag, throughput and state.

18. How do you monitor a query?

Use a unique query name, Spark UI Streaming tab, query status/progress, input/output rates, batch duration, lag, state metrics and alerts.
