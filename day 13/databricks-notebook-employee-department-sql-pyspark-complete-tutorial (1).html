Logical SQL order

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT

WHERE vs HAVING

WHERE filters rows before grouping. HAVING filters groups after aggregation.

SQL — aggregate and HAVING%sql
SELECT dept_id,
       COUNT(*) AS employee_count,
       ROUND(AVG(salary),2) AS avg_salary,
       SUM(salary) AS monthly_payroll
FROM employees
WHERE dept_id IS NOT NULL
GROUP BY dept_id
HAVING COUNT(*) >= 2
ORDER BY avg_salary DESC;

SQL — inner, left and full joins%sql
-- Change INNER to LEFT, RIGHT or FULL OUTER and compare the row count.
SELECT e.emp_id, e.emp_name, e.dept_id, d.dept_name, d.location
FROM employees e
FULL OUTER JOIN departments d
  ON e.dept_id = d.dept_id
ORDER BY COALESCE(e.dept_id,d.dept_id), e.emp_id;
SQL — semi and anti joins%sql
-- Departments that have at least one employee
SELECT d.* FROM departments d
LEFT SEMI JOIN employees e ON d.dept_id = e.dept_id;

-- Departments with no employee
SELECT d.* FROM departments d
LEFT ANTI JOIN employees e ON d.dept_id = e.dept_id;
SQL — self join for managers%sql
SELECT e.emp_name AS employee,
       e.job_title,
       COALESCE(m.emp_name,'No manager') AS manager
FROM employees e
LEFT JOIN employees m
  ON e.manager_id = m.emp_id
ORDER BY manager, employee;
Classic mistake: A condition on the right table in the WHERE clause can turn a LEFT JOIN into an effective INNER JOIN. Put match-specific conditions in ON when unmatched left rows must survive.

Numeric, NULL and conditional functions

SQL%sql
SELECT emp_name,
       salary,
       COALESCE(commission,0) AS commission,
       salary + COALESCE(commission,0) AS total_monthly_pay,
       ROUND(salary * 1.08,2) AS salary_after_8_percent,
       CASE
         WHEN salary >= 100000 THEN 'A'
         WHEN salary >= 70000  THEN 'B'
         ELSE 'C'
       END AS salary_band
FROM employees
ORDER BY total_monthly_pay DESC;

Date functions

SQL%sql
SELECT emp_name, hire_date,
       YEAR(hire_date) AS hire_year,
       MONTH(hire_date) AS hire_month,
       DATEDIFF(CURRENT_DATE(),hire_date) AS days_served,
       ROUND(MONTHS_BETWEEN(CURRENT_DATE(),hire_date)/12,1) AS years_served,

Aggregate functions

SQL%sql
SELECT d.dept_name,
       COUNT(e.emp_id) AS headcount,
       COUNT(DISTINCT e.city) AS cities,
       MIN(e.salary) AS min_salary,
       MAX(e.salary) AS max_salary,
       ROUND(AVG(e.salary),2) AS avg_salary,
       SUM(e.salary) AS payroll
FROM departments d
LEFT JOIN employees e ON d.dept_id=e.dept_id
GROUP BY d.dept_name
ORDER BY payroll DESC NULLS LAST;

6. Advanced SQL: windows, CTEs, subqueries and Delta CRUD

Window functions: rows remain visible

SQL — ranking, lag and department average%sql
SELECT emp_name, dept_id, salary,
       ROW_NUMBER() OVER(PARTITION BY dept_id ORDER BY salary DESC) AS row_num,
       RANK()       OVER(PARTITION BY dept_id ORDER BY salary DESC) AS salary_rank,
       DENSE_RANK() OVER(PARTITION BY dept_id ORDER BY salary DESC) AS dense_rank,
       LAG(salary)  OVER(PARTITION BY dept_id ORDER BY salary) AS previous_salary,
       AVG(salary)  OVER(PARTITION BY dept_id) AS department_average,
       SUM(salary)  OVER(PARTITION BY dept_id ORDER BY hire_date
                         ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS running_payroll
FROM employees;
ROW_NUMBER / RANK / DENSE_RANK

For values 100, 100, 90: row numbers are 1,2,3; ranks are 1,1,3; dense ranks are 1,1,2.

SQL — CTE: top two salaries per department%sql
WITH ranked AS (
  SELECT e.*, DENSE_RANK() OVER(
    PARTITION BY dept_id ORDER BY salary DESC
  ) AS rnk
  FROM employees e
)
SELECT emp_name, dept_id, salary, rnk
FROM ranked
WHERE rnk <= 2
ORDER BY dept_id, rnk;
SQL — correlated subquery: above department average%sql
SELECT e.emp_name, e.dept_id, e.salary
FROM employees e
WHERE e.salary > (
  SELECT AVG(x.salary)
  FROM employees x
  WHERE x.dept_id = e.dept_id
);
SQL — temporary view and pivot%sql
CREATE OR REPLACE TEMP VIEW employee_department AS
SELECT e.*, d.dept_name FROM employees e
LEFT JOIN departments d ON e.dept_id=d.dept_id;

SELECT * FROM (
  SELECT dept_name, gender, salary FROM employee_department
)
PIVOT (
  SUM(salary) FOR gender IN ('F' AS female, 'M' AS male)
)
ORDER BY dept_name;

CREATE OR REPLACE TEMP VIEW employee_updates AS
SELECT 1004 AS emp_id, 90000.00 AS salary, 'Senior Data Engineer' AS job_title
UNION ALL
SELECT 1017, 55000.00, 'Support Engineer';

MERGE INTO employees AS target
USING employee_updates AS source
ON target.emp_id = source.emp_id

  target.salary = source.salary,
  target.job_title = source.job_title
WHEN NOT MATCHED THEN INSERT

PYTHON — select, filter, derive%python
result = (
    employees_df
    .filter((F.col("salary") >= 70000) & F.col("dept_id").isNotNull())
    .withColumn("annual_salary", F.col("salary") * 12)
    .withColumn("salary_band",
        F.when(F.col("salary") >= 100000, "A")
         .when(F.col("salary") >= 70000, "B")
         .otherwise("C"))
    .select("emp_id", "emp_name", "dept_id", "salary",
            "annual_salary", "salary_band")
    .orderBy(F.desc("salary"))
)
display(result)
PYTHON — aggregate and join%python
payroll = (
    employees_df
    .join(departments_df, "dept_id", "left")
    .groupBy("dept_id", "dept_name")
    .agg(
        F.count("emp_id").alias("headcount"),
        F.round(F.avg("salary"), 2).alias("avg_salary"),
        F.sum("salary").alias("payroll")
    )
    .orderBy(F.desc_nulls_last("payroll"))
)
display(payroll)

              .select("dept_id","emp_name","salary","salary_rank"))

Order amount = sum of its line amounts. Exclude cancelled orders only when the question says “sales/revenue”.

FROM order_items GROUP BY order_id;

-- Q16
SELECT c.customer_id,c.customer_name,
 ROUND(SUM(i.quantity*i.unit_price*(1-i.discount_pct/100)),2) delivered_revenue
FROM customers c JOIN orders o ON c.customer_id=o.customer_id
JOIN order_items i ON o.order_id=i.order_id
WHERE o.status='Delivered'
GROUP BY c.customer_id,c.customer_name ORDER BY delivered_revenue DESC;

-- Q21
WITH revenue AS (
 SELECT c.customer_id,c.customer_name,
        COALESCE(SUM(i.quantity*i.unit_price*(1-i.discount_pct/100)),0) revenue
 FROM customers c LEFT JOIN orders o
   ON c.customer_id=o.customer_id AND o.status<>'Cancelled'
 LEFT JOIN order_items i ON o.order_id=i.order_id
 GROUP BY c.customer_id,c.customer_name
)
SELECT *,DENSE_RANK() OVER(ORDER BY revenue DESC) revenue_rank FROM revenue;

-- Q22
WITH totals AS (
 SELECT o.customer_id,o.order_id,
   SUM(i.quantity*i.unit_price*(1-i.discount_pct/100)) order_total
 FROM orders o JOIN order_items i ON o.order_id=i.order_id
 GROUP BY o.customer_id,o.order_id
), r AS (
 SELECT *,ROW_NUMBER() OVER(PARTITION BY customer_id ORDER BY order_total DESC) rn
 FROM totals
) SELECT * FROM r WHERE rn<=2;

-- Q23/Q24
WITH monthly AS (
 SELECT DATE_TRUNC('month',o.order_date) month,
   SUM(i.quantity*i.unit_price*(1-i.discount_pct/100)) revenue
 FROM orders o JOIN order_items i ON o.order_id=i.order_id
 WHERE o.status<>'Cancelled' GROUP BY DATE_TRUNC('month',o.order_date)
)
SELECT *,SUM(revenue) OVER(ORDER BY month) running_revenue,
 revenue-LAG(revenue) OVER(ORDER BY month) mom_change FROM monthly;

-- Q25
SELECT * FROM order_items i
WHERE unit_price > (SELECT AVG(x.unit_price) FROM order_items x
                    WHERE x.category=i.category);

-- Q30
SELECT c.customer_id,c.customer_name,COUNT(DISTINCT o.order_id) order_count,
 MAX(o.order_date) last_order_date,
 COALESCE(ROUND(SUM(CASE WHEN o.status<>'Cancelled'
   THEN i.quantity*i.unit_price*(1-i.discount_pct/100) ELSE 0 END),2),0) lifetime_value
FROM customers c LEFT JOIN orders o ON c.customer_id=o.customer_id
LEFT JOIN order_items i ON o.order_id=i.order_id

9. Interview questions and concise answers

1. What is a Databricks notebook?

An interactive document containing executable cells, text, results and visualizations, commonly used for collaborative data engineering and analytics.

2. SQL table vs DataFrame?

A table is a named persisted or temporary relation. A DataFrame is a distributed logical dataset represented through an API. Both can be optimized by Spark's engine.

3. Why use Delta tables?

They add reliable transactions and table-management features such as updates, deletes, merge and history over data lake storage.

4. WHERE vs HAVING?

WHERE filters input rows; HAVING filters grouped results.

5. INNER vs LEFT JOIN?

INNER keeps matches only. LEFT keeps every left row and supplies NULL for absent right matches.

6. Semi vs anti join?

Semi returns left rows with a match. Anti returns left rows without a match; neither adds right-side columns.

7. Why can a join duplicate rows?

The key may not be unique on one or both sides. Many-to-many matches multiply rows.

8. GROUP BY vs window function?

GROUP BY collapses rows to group grain. A window computes across related rows while retaining row detail.

9. RANK vs DENSE_RANK?

Both give ties the same rank; RANK leaves gaps after ties, DENSE_RANK does not.

10. What does PARTITION BY mean in a window?

It restarts the window calculation independently for each key group.

11. CTE vs subquery?

Both express a logical intermediate result. A CTE often improves readability and can be referenced cleanly in a larger statement.

12. What is MERGE?

An atomic conditional operation that can update matching target rows and insert nonmatching source rows; often called an upsert pattern.

13. Why use COALESCE?

It returns the first non-NULL expression, useful for defaulting absent amounts or labels.

14. count(*) vs count(column)?

count(*) counts rows. count(column) ignores rows where that column is NULL.

15. What causes a Spark shuffle?

Data must move between partitions, commonly for joins, distinct, grouping, repartitioning and global ordering.

16. When is broadcast join appropriate?

When one relation is genuinely small enough to copy to executors, avoiding a large shuffle. Confirm with size and plan rather than guessing.

17. Why avoid SELECT * in production?

It reads unnecessary columns, creates brittle downstream schemas and obscures intent.

18. How do you debug an incorrect aggregate?

Confirm grain, row counts before and after joins, duplicate keys, NULL behavior, status filters and the exact formula.
