1. What is streaming?

Streaming is continuous processing of data that arrives over time instead of waiting for the entire dataset to finish arriving.

Simple example: A retail company receives one order event whenever a customer checks out. The dashboard should update every few seconds without waiting until midnight.

What?Continuous or near-real-time processing of unbounded events.
Why?Faster alerts, dashboards, personalization and operational decisions.

3. Discretized Streaming (DStreams)

A Discretized Stream (DStream) is the original Spark Streaming abstraction. Internally, it is a sequence of RDDs created at fixed batch intervals.

0–2 sec
RDD 1
→
2–4 sec
RDD 2
→
4–6 sec
RDD 3
→
6–8 sec
RDD 4

If the batch interval is 2 seconds, all records received in each 2-second window form one RDD. DStream transformations create new DStreams, and output operations run a Spark job for each batch.

Simple socket word count

from pyspark import SparkContext
from pyspark.streaming import StreamingContext

sc = SparkContext("local[2]", "DStreamWordCount")
sc.setLogLevel("WARN")

# Create one RDD every 2 seconds
ssc = StreamingContext(sc, 2)
ssc.checkpoint("checkpoint/dstream-wordcount")

lines = ssc.socketTextStream("localhost", 9999)
words = lines.flatMap(lambda line: line.split())
pairs = words.map(lambda word: (word.lower(), 1))
counts = pairs.reduceByKey(lambda a, b: a + b)
counts.pprint()

ssc.start()
ssc.awaitTermination()

→

  1. Create SparkContext and StreamingContext.
  2. Set a batch interval.
  3. Connect a source such as Kafka/Kinesis/socket.
  4. Split incoming records into RDD batches.
  5. Apply DStream transformations.
  6. Run output operations for every batch.
  7. Use checkpointing or reliable source offsets for recovery.
  8. Monitor batch processing time and scheduling delay.
DStream operationPurpose
map, flatMap, filterPer-record transformations
reduceByKeyAggregate within each batch
windowCalculate across multiple recent batches
updateStateByKeyMaintain state across batches
foreachRDDWrite/process every generated RDD
Stable streaming rule: batch processing time should remain below the batch interval; otherwise backlog grows.

5. Structured Streaming

Structured Streaming is Spark’s DataFrame/SQL-based streaming engine. It treats the live stream as an unbounded table that receives new rows continuously.

Static table
existing rows
+
New events
new rows
→
Incremental query
→
Updated result

Rate-source demonstration

from pyspark.sql import SparkSession, functions as F

spark = SparkSession.builder.appName("StructuredRateDemo").getOrCreate()
spark.sparkContext.setLogLevel("WARN")

events = (
    spark.readStream
    .format("rate")
    .option("rowsPerSecond", 5)
    .load()
)

classified = events.withColumn(
    "event_type",
    F.when(F.col("value") % 2 == 0, "EVEN").otherwise("ODD")
)

query = (
    classified.groupBy("event_type").count()
    .writeStream
    .format("console")
    .outputMode("complete")
    .option("checkpointLocation", "checkpoint/rate-demo")
    .start()
)

query.awaitTermination()
Use Structured Streaming for new Spark streaming applications unless a specific existing DStream integration requires the legacy API.

8. Output modes and triggers

Output modeMeaningTypical use
AppendOnly finalized new result rowsStateless transformations, finalized windows
UpdateRows changed since last triggerChanging aggregates when sink supports it
CompleteEntire result table every triggerSmall aggregate tables/testing
# Process approximately every 10 seconds
.writeStream.trigger(processingTime="10 seconds")

# Process all currently available data and stop
.writeStream.trigger(availableNow=True)
Valid mode depends on the query and sink. An append-mode stateful aggregation commonly needs a watermark so Spark knows when a window is final.

9. Event time, windows, watermarks and late data

Event time is when the event happened. Processing time is when Spark processed it. Network delay can make events arrive late or out of order.

10:00:02
network delay
Spark receives
10:00:25
watermark rule
Keep or drop from state
trends = (
    posts
    .withWatermark("created_at", "10 minutes")
    .groupBy(
        F.window("created_at", "5 minutes"),
        F.explode(F.split(F.lower("text"), r"\s+")).alias("word")

    .count()

ConceptPurpose
WindowGroup events into time ranges
WatermarkBound how long Spark retains state for late data
State storePersist intermediate counts/joins/deduplication state
A “10 minutes” watermark is not simply “drop every event older than exactly 10 minutes.” It advances based on observed event-time progress and the query semantics.

10. Checkpoints, offsets and fault tolerance

Source offsets
+
Query plan metadata
+
State store
→
Checkpoint

A checkpoint records progress so a restarted query can continue consistently. Store production checkpoints in durable distributed storage, not a temporary local folder.

query = (
    result.writeStream
    .format("parquet")
    .outputMode("append")
    .option("path", "hdfs:///retail/output/social_trends")
    .option("checkpointLocation", "hdfs:///retail/checkpoints/social_trends")
    .start()
)
Do not reuse one checkpoint directory for different queries. Avoid deleting checkpoints casually; it can cause replay, duplicates, lost state or incompatible recovery.

11. Processing real-time feeds from Twitter/X

Twitter is now X. For new integrations, use X API v2. The official filtered-stream endpoint delivers matching public posts in near real time after rules are configured. It requires developer access and bearer authentication; current access is pay-per-use or enterprise.

Important update: Old tutorials using Tweepy’s legacy Twitter streaming APIs or obsolete Spark Twitter connectors may no longer work. Never paste bearer tokens directly into source code or a tutorial.

Official filtered-stream steps

  1. Create an X developer account/project/application.
  2. Obtain the allowed authentication credentials.
  3. Store the bearer token in a secret manager or environment variable.
  4. Add filtered-stream rules such as spark OR pyspark lang:en.
  5. Connect to /2/tweets/search/stream.
  6. Parse returned JSON lines.
  7. Publish them to Kafka.
  8. Let Structured Streaming consume Kafka durably.

Add rule and connect

# Linux/macOS: TOKEN must come from a secure environment variable
curl -X POST "https://api.x.com/2/tweets/search/stream/rules" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"add":[{"value":"(spark OR pyspark) lang:en","tag":"spark-posts"}]}'

curl "https://api.x.com/2/tweets/search/stream?tweet.fields=created_at,lang,author_id" \
  -H "Authorization: Bearer $TOKEN"

Why Kafka between X and Spark?

  • Durable buffering when Spark is stopped or restarted.
  • Offsets support replay and recovery.
  • Producer/API connection is separated from analytics.
  • Multiple consumers can reuse the same feed.
  • Back-pressure is easier to manage operationally.

Illustrative ingestion producer

import os
import requests
from kafka import KafkaProducer

token = os.environ["X_BEARER_TOKEN"]
producer = KafkaProducer(
    bootstrap_servers=["localhost:9092"],
    value_serializer=lambda value: value.encode("utf-8")
)

url = (
    "https://api.x.com/2/tweets/search/stream"
    "?tweet.fields=created_at,lang,author_id"
)

with requests.get(
    url,
    headers={"Authorization": f"Bearer {token}"},
    stream=True,
    timeout=90
) as response:
    response.raise_for_status()
    for line in response.iter_lines(decode_unicode=True):
        if line:
            producer.send("social-posts", line)
            producer.flush()
Production code must implement reconnect with backoff, rate-limit handling, secret rotation, metrics, graceful shutdown, error/dead-letter handling and X developer terms.

Spark Kafka consumer

from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StructField, StringType

post_schema = StructType([
    StructField("id", StringType()),
    StructField("text", StringType()),
    StructField("author_id", StringType()),
    StructField("created_at", StringType()),
    StructField("lang", StringType())
])
envelope_schema = StructType([
    StructField("data", post_schema)
])

raw = (
    spark.readStream
    .format("kafka")
    .option("kafka.bootstrap.servers", "localhost:9092")
    .option("subscribe", "social-posts")
    .option("startingOffsets", "latest")
    .load()
)

posts = (
    raw.select(F.from_json(F.col("value").cast("string"), envelope_schema).alias("j"))
    .select("j.data.*")
    .withColumn("created_at", F.to_timestamp("created_at"))
    .filter(F.col("lang") == "en")
)

trends = (
    posts
    .withWatermark("created_at", "10 minutes")
    .groupBy(F.window("created_at", "5 minutes"))
    .agg(F.count("*").alias("post_count"))
)

query = (
    trends.writeStream
    .format("console")
    .outputMode("update")
    .option("checkpointLocation", "checkpoint/x-trends")
    .start()
)
query.awaitTermination()

Structured Streaming file demo

from pyspark.sql import SparkSession, functions as F
from pyspark.sql.types import StructType, StructField, StringType, TimestampType

spark = SparkSession.builder.appName("SocialFeedSimulator").getOrCreate()
spark.sparkContext.setLogLevel("WARN")

schema = StructType([
    StructField("post_id", StringType()),
    StructField("text", StringType()),
    StructField("author_id", StringType()),
    StructField("created_at", TimestampType())
])

posts = (
    spark.readStream
    .schema(schema)
    .option("maxFilesPerTrigger", 1)
    .json("data/incoming_posts")
)

words = (
    posts
    .withWatermark("created_at", "10 minutes")
    .select(
        "created_at",
        F.explode(F.split(F.lower(F.regexp_replace("text", r"[^a-zA-Z0-9 ]", "")), r"\s+"))
         .alias("word")
    )
    .filter(F.length("word") >= 4)
)

counts = words.groupBy(
    F.window("created_at", "5 minutes"), "word"
).count()

query = (
    counts.writeStream
    .format("console")
    .outputMode("update")
    .option("truncate", False)
    .option("checkpointLocation", "checkpoint/social-simulator")
    .start()
)

query.awaitTermination()
Create each new JSON file outside the input folder and then move it into the folder. Spark treats each newly discovered file as new streaming input.

17. Best practices and limitations

  • Prefer Structured Streaming for new applications.
  • Use Kafka/Kinesis or another durable source for production.
  • Define schemas explicitly; avoid repeated inference.
  • Keep checkpoints on durable storage and separate per query.
  • Use event time and realistic watermarks.
  • Deduplicate using stable event IDs where required.
  • Control state growth and monitor state-store metrics.
  • Make foreachBatch writes idempotent.
  • Protect API tokens with a secret manager/environment injection.
  • Respect source licensing, privacy, rate limits and developer terms.

Limitations

  • Micro-batch processing is not identical to record-at-a-time processing.
  • Stateful operations consume storage and memory.
  • Late-data guarantees depend on source, watermark, query and sink.
  • Console, memory, rate and socket are training/testing tools.
  • External API availability and pricing are outside Spark’s control.

Conclusion

DStreams divide a continuous feed into RDD batches, while Structured Streaming represents it as an unbounded table processed incrementally. A reliable social-feed design separates API ingestion from Spark through a durable broker, uses event-time controls and checkpoints, and monitors whether processing keeps pace with arrival.
