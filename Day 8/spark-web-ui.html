1. Spark Web UI - Overview

The Spark Web UI is a website that Spark serves from your own machine while a Spark application is running. It shows you, in real time, exactly what Spark is doing: which jobs ran, how they split into stages and tasks, how long each took, and where time and memory were spent. It is the single most important tool for understanding and fixing slow Spark jobs.

Learning objectives:

  • Open the Spark Web UI on your own machine (port 4040) and keep it open long enough to explore.
  • Read each tab - Jobs, Stages, Tasks, Storage, Environment, Executors, SQL - and say what it is for.
  • Use the UI to spot the three classic problems: a shuffle, data skew, and a spill.
  • Run the practice samples and predict what the UI will show before you look.
Before you startPrerequisites: Spark installed locally and the ability to run pyspark. Estimated time: 45-60 min. After this you can open a job's UI and explain in an interview exactly why it was slow.

2. Introduction

When a Spark job is slow, you cannot see inside it by reading your code - the code only says what you asked for, not how Spark actually executed it. Spark solves this by starting a small web server inside your application and publishing a dashboard to it. Point a browser at that dashboard and Spark shows you its own execution, step by step.

Why it matters. Almost every real Spark performance problem is diagnosed in this UI. Was there a shuffle you did not expect? Did one task take ten minutes while the other 199 took two seconds (data skew)? Did Spark run out of memory and write to disk (a spill)? None of this is visible from your code, but all of it is plainly visible in the UI. Engineers who can read this dashboard fix problems in minutes; those who cannot, guess.

Where it lives. While your application runs, the UI is served at http://localhost:4040. It disappears when the application ends - which is the single most common frustration for beginners. To view finished applications you use the History Server, covered in section 8.

Memory hookThe Web UI is Spark's black box flight recorder. Remember it as: your code says what; the UI shows how.

3. Beginner-Friendly Explanation

Imagine a large parcel depot. You hand over an order ("deliver these 10,000 parcels") and walk away. Later someone says the delivery was slow. You have no idea why - you only saw the order form.

Now imagine the depot has a big glass wall with live monitors. Through it you can see:

  1. The orders board (Jobs tab) - every order you placed, whether it finished, and how long it took.
  2. The workflow board (Stages tab) - each order broken into phases: "sort by postcode", then "load the vans". You can see which phase was the slow one.
  3. The individual workers (Tasks) - inside a phase, each worker's time. If 199 workers finished in seconds and one took an hour, you have found your problem - that worker got a giant pile (this is data skew).
  4. The staff list (Executors tab) - how many workers you have, how much memory each has, and whether any crashed.
  5. The storage room (Storage tab) - what you asked the depot to keep on a shelf for reuse (your cached data).

The Spark Web UI is that glass wall. Without it you are guessing; with it, you can point at the exact phase and the exact worker that caused the delay.

Where the analogy breaks down: a real depot keeps running after your order finishes, but Spark's live UI shuts down the moment your application ends - the glass wall vanishes. That is why the History Server exists: it is the archive of past deliveries.

Remember it asJobs = orders · Stages = phases · Tasks = individual workers · Executors = the staff · Storage = the shelf.
Check yourself: 199 tasks finish in 2 seconds, one takes 8 minutes. What is this called?

Data skew - one partition holds far more data than the others, so its task lags behind while the rest sit idle.

4. The Tabs Explained (Technical)

Definition. The Spark Web UI is an HTTP dashboard served by the driver (the process that coordinates your application) on port 4040 by default, exposing metrics for jobs, stages, tasks, executors, storage, environment, and SQL queries.

Key terms

  • Job - all the work triggered by one action (e.g. count(), show()). One action = one job.
  • Stage - a group of tasks that can run without moving data between machines. Spark starts a new stage at every shuffle.
  • Task - the smallest unit of work; one task processes one partition.
  • Shuffle - moving data across the network so records with the same key land together. Expensive; visible as Shuffle Read/Write in the UI.
  • Data skew - one partition is far larger than the rest, so its task takes much longer (a "straggler").
  • Spill - when data does not fit in memory, Spark writes it to disk. Shown as "Spill (Memory)" and "Spill (Disk)"; a strong slowness signal.
  • DAG - the Directed Acyclic Graph, the picture of how your operations connect; each stage shows one.
TabWhat it showsUse it to answer
JobsEvery job, its status, duration, and stages"Which action was slow?"
StagesEach stage's tasks, shuffle read/write, spill, and the DAG"Which phase cost the time? Was there a shuffle?"

How to read this: left to right = the chain from your code down to the machines. Teal = what you wrote / who ran it, purple = Spark's units of work, orange = where a shuffle splits the work into two stages.

Components: Your action is the trigger. It becomes one Job in the Jobs tab. The job is cut into Stages at every shuffle boundary - a shuffle is why Stage 2 exists at all. Each stage is made of Tasks, one per partition. Those tasks run on Executors, listed with their memory and failures.

Flow: (1) You call an action. (2) Spark creates a Job - see it appear in the Jobs tab. (3) Spark splits it at shuffles into Stages - the Stages tab shows Shuffle Read/Write bytes here. (4) Each stage runs Tasks in parallel, one per partition - click the stage to see per-task durations. (5) Executors report memory, GC, and failures. To debug, walk this chain top-down: slow Job → which Stage → which Task → which Executor.

Common confusionPort 4040 is per-application, not per-machine. If you start a second Spark app while the first is running, the second gets 4041, the third 4042, and so on. If 4040 "shows the wrong job", you are probably looking at another app's UI.

5. Open It (Simple Example)

The catch: the UI only lives while the app lives. So we deliberately keep the app alive with an input() pause, then browse the UI.

from pyspark.sql import SparkSession

spark = SparkSession.builder.appName("uiDemo").getOrCreate()

df = spark.range(1_000_000)          # 1 million rows
print(df.count())                     # ACTION -> creates Job 0

input("Open http://localhost:4040 then press Enter to quit...")
spark.stop()
  • SparkSession...getOrCreate() - starts the app; the driver immediately begins serving the UI on port 4040.
  • spark.range(1_000_000) - builds a DataFrame. Nothing runs yet (it is lazy).
  • df.count() - the action. This creates Job 0, which you will now see in the Jobs tab.
  • input(...) - the trick: it freezes the program so the app stays alive and the UI stays reachable.
  • spark.stop() - ends the app; the UI at 4040 immediately stops responding.

Result: open http://localhost:4040 in a browser. You will see one job ("count at ..."), with one or two stages, and a number of tasks equal to your partition count. If the page does not load, the app has already exited - that is the classic mistake.

Pro tipIn the pyspark interactive shell you do not need the input() trick at all - the shell keeps the app alive, so 4040 stays up as long as the shell is open. That is the easiest way to explore.

6. Real Debugging Story (Industry Example)

A retail data team has a nightly job that joins orders to customers and totals revenue per customer. It used to take 12 minutes; this week it takes 90. The code has not changed. Nobody can see the problem by reading it, so an engineer opens the Web UI while the job runs.

She works top-down. In the Jobs tab, one job dominates the runtime. She clicks into its Stages and finds one stage taking almost all the time, with a large Shuffle Read - so a shuffle is involved, as expected for a join. Then she opens that stage's task summary and sees the giveaway: the median task took 3 seconds, but the max took 46 minutes. One single task is holding up the entire job. That is data skew - one customer ID (a bulk corporate account) now owns a huge share of the orders, so all its rows landed in one partition. The Executors tab confirms it: that one executor shows heavy spill to disk while the others idle.

How to read this: left to right = the order you should click through the UI. Teal = start and conclusion, purple = the tabs you visit, orange = the decisive clue (the median-vs-max gap).

Components: The Jobs tab narrows you to the slow job. The Stages tab narrows to the slow stage and reveals shuffle bytes. The Task summary is the decisive step: comparing the median task time to the max exposes a straggler. The Executors tab corroborates with spill and GC. Together they yield the diagnosis.

Flow: (1) Open the UI while the job runs. (2) Jobs tab: find the job consuming the runtime. (3) Stages tab: find the stage consuming that job, and note Shuffle Read/Write. (4) Click the stage and read the task summary - if max is vastly greater than median, you have skew. (5) Executors tab: check spill and GC to confirm. (6) Fix the cause (salt the hot key, repartition, or broadcast the small side of the join), rerun, and compare the same screens.

Watch outDo not stop at "the stage is slow". The median-vs-max comparison in the task summary is what separates skew (one bad task) from a genuinely heavy stage (all tasks slow). They need completely different fixes.

7. Practice Samples

Run each sample in the pyspark shell (which keeps the UI alive at http://localhost:4040). Before you look, predict what the UI will show - then check.

Sample 1 - See a Job appear (no shuffle):

df = spark.range(1_000_000)
df.filter("id % 2 = 0").count()     # action -> one job

Look for: Jobs tab shows one new job. It has a single stage - filter is narrow, so no shuffle. Shuffle Read/Write should be ~0. Lesson: narrow operations do not create extra stages.

Sample 2 - Make a shuffle appear (two stages):

df = spark.range(1_000_000)
df.groupBy(df.id % 10).count().show()   # groupBy -> WIDE -> shuffle

Look for: the job now has two stages. In the Stages tab you will see non-zero Shuffle Write on the first stage and Shuffle Read on the second. Lesson: a new stage is created at every shuffle - this is how you spot shuffles you did not intend.

Sample 3 - Create data skew and see the straggler:

from pyspark.sql.functions import when, col
# 90% of rows get key 0 -> one giant partition
skewed = spark.range(2_000_000).withColumn(
    "k", when(col("id") % 10 != 0, 0).otherwise(col("id")))
skewed.groupBy("k").count().show()

Look for: open the slow stage and read the task summary table. The max task duration will be far larger than the median - that gap is the fingerprint of skew. Lesson: always compare median vs max, not just the total.

Sample 4 - Watch caching in the Storage tab:

df = spark.range(5_000_000).cache()
df.count()      # first action: computes AND caches
df.count()      # second action: should be much faster

Look for: the Storage tab now lists a cached dataset with its size and "100% cached". In the Jobs tab, the second count() job is dramatically shorter. Lesson: Storage tab proves whether your cache actually took effect - an empty Storage tab means it did not.

Sample 5 - Confirm your config in the Environment tab:

spark.conf.set("spark.sql.shuffle.partitions", 8)
spark.range(100000).groupBy("id").count().count()

Look for: the Environment tab shows spark.sql.shuffle.partitions = 8, and the shuffle stage now has 8 tasks instead of the default 200. Lesson: Environment tells you what config is really in effect - useful when a setting "isn't working".

Pro tipDo these five in order and you will have seen, with your own eyes, a job, a stage boundary, a shuffle, skew, a cache, and a config change - which is most of what the UI is used for in real work.

8. Live UI vs History Server

AspectLive Web UI (4040)History Server (18080)
When availableOnly while the app is runningAfter the app has finished
Default port4040 (then 4041, 4042 ...)18080
Needs setup?No - automaticYes - enable event logging, start the server
Best forWatching a job as it runsPost-mortem on a finished/failed job

Enable the History Server (so finished jobs are still viewable):

# in spark-defaults.conf (or as --conf flags)
spark.eventLog.enabled           true
spark.eventLog.dir               file:///tmp/spark-events
spark.history.fs.logDirectory    file:///tmp/spark-events

# create the folder, then start the server:
mkdir -p /tmp/spark-events
$SPARK_HOME/sbin/start-history-server.sh
# browse http://localhost:18080

What & why: spark.eventLog.enabled tells Spark to write a log of everything the UI showed. eventLog.dir is where those logs go, and history.fs.logDirectory is where the History Server reads them from - these two must match, which is the most common mistake. The folder must exist before the app runs.

When to pick which: use the live UI (4040) while developing and watching a job; use the History Server (18080) to investigate a job that already finished or failed overnight - which is most production debugging.

9. Best Practices

Do:

  • Explore in the pyspark shell. Why: the shell keeps the app alive, so the UI stays up while you click around.
  • Debug top-down: Jobs → Stages → Tasks → Executors. Why: it narrows the problem instead of guessing.
  • Always compare median vs max task time. Why: it is the fastest way to distinguish skew from a genuinely heavy stage.
  • Enable event logging in any job you might need to debug later. Why: without it, the evidence vanishes when the app exits.
  • Check the Environment tab when a config "isn't working". Why: it shows what is actually in effect, not what you think you set.

Common mistakes:

  • Running a script and then wondering why 4040 is dead. Why bad: the UI dies with the app - keep it alive or use the History Server.
  • Reading only total stage time. Why bad: it hides a single straggler task (skew).
  • Ignoring Shuffle Read/Write columns. Why bad: they are the clearest evidence of the most expensive operation in Spark.
  • Mismatching eventLog.dir and history.fs.logDirectory. Why bad: the History Server shows an empty list and you assume it is broken.

10. Edge Cases & Gotchas

  • Port drift: a second concurrent app takes 4041, not 4040. Read the driver's startup log - it prints the actual UI port.
  • Port already in use: a leftover Spark process can hold 4040; kill it or let Spark move to the next port.
  • Remote clusters: localhost:4040 will not work from your laptop - you must tunnel or use the cluster manager's proxy link (e.g. YARN's tracking URL).
  • Nothing in the Jobs tab: you only ran transformations. Until an action runs, there is no job to show - this is lazy evaluation, not a broken UI.
  • Empty Storage tab despite calling cache(): cache is lazy too - it only materialises after an action.
  • 200 tiny tasks: a default spark.sql.shuffle.partitions=200 on small data creates lots of near-empty tasks; the UI makes this obvious.
Watch out"The UI is empty" is almost never a UI bug. It usually means no action has run yet, or the application already exited.

12. Interview Q&A (15)

BeginnerWhat is the Spark Web UI?
A dashboard served by the driver that shows a running application's jobs, stages, tasks, executors, storage, environment, and SQL plans in real time.
BeginnerWhat is the default port?
4040. A second concurrent application uses 4041, a third 4042, and so on.
BeginnerWhy does the UI disappear when my script ends?
The UI is served by the driver process; when the application exits, the driver stops and the web server goes with it.
BeginnerWhat does the Jobs tab show?
One entry per action, with its status, duration, and the stages it contains.
BeginnerWhat does the Executors tab show?
Each executor's cores, memory usage, task counts, GC time, spill, and failures.
IntermediateHow do you tell from the UI that a shuffle happened?
The job splits into more than one stage, and the Stages tab shows non-zero Shuffle Write on one stage and Shuffle Read on the next.
IntermediateHow do you detect data skew in the UI?
Open the slow stage's task summary: if the max task duration is far greater than the median, one partition is oversized - that is skew.
IntermediateWhat is a spill and where do you see it?
When data does not fit in memory Spark writes it to disk. It appears as Spill (Memory)/Spill (Disk) in the stage and executor metrics, and signals memory pressure.
IntermediateWhat is the Storage tab for?
It lists cached/persisted datasets with their size and how much is actually cached - the way to verify a cache took effect.
IntermediateWhy might the Jobs tab be empty?
Because only transformations have run. Transformations are lazy; no job exists until an action triggers execution.
AdvancedWhat is the History Server and how do you enable it?
A server (default port 18080) that renders finished applications from event logs. Enable spark.eventLog.enabled, set eventLog.dir, point history.fs.logDirectory at the same path, and start start-history-server.sh.
AdvancedWalk me through debugging a slow job with the UI.
Top-down: Jobs tab to find the slow job, Stages tab to find the dominant stage and check shuffle bytes, the task summary to compare median vs max (skew), then Executors for spill, GC, and failures. Fix the cause and compare the same screens after rerunning.
AdvancedHow does the SQL tab help?
It shows the query plan for DataFrame/SQL operations with per-operator metrics, letting you see how Spark actually planned and executed the query, including where the exchanges (shuffles) are.
AdvancedWhy can't you reach localhost:4040 on a cluster job?
The UI runs on the driver node, not your laptop. You reach it through the cluster manager's proxy/tracking URL or an SSH tunnel.
AdvancedThe Environment tab shows a config different from what you set. Why?
Configs set after the SparkContext starts, or overridden by spark-defaults/submit flags, may not apply. The Environment tab shows what is actually in effect, which is why it is the place to check.

15. Glossary

  • DAG - the graph of how operations connect; shown per stage.
  • Data skew - one partition far larger than others, causing a straggler task.
  • Driver - the process that coordinates the app and serves the Web UI.
  • Event log - the recorded history of an app, used by the History Server.
  • History Server - dashboard for finished apps, default port 18080.
  • Job - all work triggered by one action.
  • Port 4040 - default live Web UI port (4041, 4042 for further apps).
  • Shuffle Read/Write - bytes moved across the network; the sign of a wide operation.
  • Spill - data written to disk because it did not fit in memory.
  • Stage - group of tasks with no shuffle between them.
  • Straggler - the one task holding up a stage.
  • Task - the smallest unit of work; one per partition.

16. Summary

The Spark Web UI is the driver's live dashboard at localhost:4040. It turns your invisible execution into something you can read: Jobs (one per action), Stages (split at shuffles), Tasks (one per partition), plus Storage, Environment, Executors, and SQL. Debug top-down, and use the History Server (18080) for jobs that have already finished.

Quick-revision cheat list:

  • Live UI = localhost:4040 · second app = 4041 · History Server = 18080.
  • UI dies with the app - use the pyspark shell, a pause, or event logging.
  • One action = one Job. A new Stage starts at every shuffle.
  • Shuffle Read/Write > 0 = a wide operation happened.
  • Skew = max task time far greater than median task time.
  • Spill = ran out of memory, wrote to disk = a slowness red flag.
  • Storage tab proves a cache worked; Environment tab proves a config took effect.
  • Debug order: Jobs → Stages → Tasks → Executors.

What to learn next: reading the SQL plan & Exchange operators → fixing skew (salting, broadcast joins) → tuning spark.sql.shuffle.partitions → memory tuning and spill reduction.
