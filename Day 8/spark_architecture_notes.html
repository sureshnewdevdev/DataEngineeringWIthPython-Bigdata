Apache Spark Architecture Study Notes

Core Concepts, Runtime Framework, and Component Interactions

These study notes break down the operational execution framework of Apache Spark. Understanding how user-submitted high-level API code transforms into physical execution across computing nodes is key to building performance-optimized data pipelines.

1. The Execution Component Hierarchy

Every Apache Spark application acts as a single centralized coordinator system distributing small operational tasks across an independent cluster of computational environments. The execution hierarchy consists of four main abstraction levels:

  • DAG (Directed Acyclic Graph): The high-level logical sequence of resilient distributed dataset (RDD) transitions. It serves as an optimal blueprint mapping data flow pipelines from origins (e.g., storage reading) to destinations (e.g., action outputs).
  • Job: The absolute highest execution layer. A brand new job is constructed instantaneously every single time a developer triggers an Action within the driver code loop.
  • Stage: The physical segmentation sub-layer within a job. Jobs are systematically broken into distinct stages based on whether the downstream data transformations introduce network shuffle boundaries.
  • Task: The single smallest discrete unit of work within Spark's engine. A task represents a specific code fragment running against a singular independent partition chunk of data inside a standalone CPU core slot.

3. Granular Step-by-Step System Events

Step 1: Program Entry and Lazy Lineage Logging

The system execution life cycle initiates inside the Driver Node. As code declarations containing basic transformations are encountered, the system operates completely under a lazy evaluation methodology. Data partitioning structures are verified, metadata references are logged, and operations are systematically recorded into the initial Logical Lineage Graph. No distributed compute workers are engaged.

Driver-Side Activity Context
Running standard main application parsing threads, establishing context configurations via the internal SparkSession cluster interface, and registering programmatic dependencies.

Step 2: Action Identification & Graph Assembly

When the driver thread hits an active Action (e.g., .count(), .saveAsTextFile(), .collect()), lazy parsing immediately concludes. The DAGScheduler interprets the full logical transformation roadmap and constructs the physical dependency DAG.

Step 3: Stage Boundary Isolation

The DAGScheduler optimizes execution by establishing precise boundary divisions across the graph:

  • Narrow Transformations: Operations such as map() and filter() apply computational logic directly to rows within a single isolated partition block. These operations are combined using an approach called Pipelining to prevent network or disk resource consumption.
  • Wide Transformations: Operations such as reduceByKey() or join() require grouping similar keys across different dataset partitions. This forces a Shuffle Boundary, causing the scheduler to split the graph into two distinct execution blocks (Stage 1 and Stage 2).

Step 4: Task Serialization & Distribution Scheduling

The Driver's TaskScheduler determines the exact number of tasks required based on the data partitions in the source files. The driver serializes the code logic, packages it into independent executable task bundles, and routes them via the network to the specific Executors where data partitions reside (maximizing **Data Locality**).

Step 5: Distributed Stage 1 Compute Pipelines

The Executors take over processing workloads on the cluster nodes:

  1. The task reads individual blocks into worker memory.
  2. Pipelined operations (e.g., filter, map) are executed within worker memory loops.
  3. Shuffle Write Phase: Data is organized locally in memory buffers sorted by key hash definitions, and then written out to temporary local disk storage. Executors notify the Driver that Stage 1 tasks are complete.
Executor Worker Task Responsibility
Opening disk data connections, hosting local task thread execution, maintaining partition caches, and performing high-velocity intermediate shuffle spills to local storage disks.

Step 6: Network Shuffle and Stage 2 Finalization

The Driver coordinates the launch of Stage 2. Executors fetch their required dataset keys from the local disks of Stage 1 nodes over the network (**Shuffle Read Phase**). They perform the final aggregations and stream the results out to the permanent target storage directory.
