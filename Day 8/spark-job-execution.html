Understanding Spark Job Execution

Follow one word-count pipeline from lazy transformations to Job 0, a DAG, two stages, six tasks, executor work, and the final collected result.

Lifecycle stated early: Application → you call an ACTION → JOB → DAG built → cut into STAGES at shuffle boundaries → each stage creates TASKS → executors run them → results return.

Section 2

Why Execution Internals Matter

A Spark program can contain five transformations yet do no distributed work. Spark records how data should be transformed, then waits for an action. This delay is called lazy evaluation. It allows Spark to examine the whole lineage before scheduling tasks, combine compatible operations, and recover lost data from dependencies.

When performance is poor, the source code alone is not enough. A one-line reduceByKey can move data across machines, write shuffle files, create a stage boundary, and produce uneven tasks when a few words dominate the input. Understanding the execution structure tells you where time, network, memory, and disk are being used.

The Spark Web UI turns that structure into evidence. The Jobs tab shows action-level work. The Stages tab shows task counts, durations, input, shuffle read, and shuffle write. The Executors tab shows where tasks ran. Reading these views without understanding Job, Stage, Task, and DAG leads to incorrect conclusions—for example, calling every slow item a “job” or assuming each transformation creates a stage.

Key PointTransformations create no running Job. They extend the recorded lineage. Calling collect() creates the Job and triggers the scheduler.
Common ConfusionA stage is not one transformation. Several narrow transformations can run as one pipelined stage because each output partition depends on only one input partition.

Remember it as: code describes first; action demands later.

Section 3

Beginner Explanation: Restaurant Order Analogy

Imagine a restaurant that has written preparation instructions but has not received a customer order. The instructions describe how to wash ingredients, cut vegetables, discard unsuitable pieces, group ingredients by dish, and plate results. These written instructions resemble Spark transformations: useful plans, but no cooking begins yet.

Analogy mapping

  • Restaurant → Spark application.
  • Customer places order → action is called.
  • One meal request → Job.
  • Kitchen route plan → DAG.
  • Preparation phases → Stages.
  • Tray assigned to one cook → one Task for one partition.
  • Passing grouped ingredients → Shuffle.
  • Cooks and kitchen stations → Executors.

Why the analogy helps

The restaurant does not create active cook assignments while writing recipes. It assigns cooks only when an order arrives. Spark behaves the same way: lineage exists before execution; tasks exist when a job is submitted.

The analogy has a limit: Spark tasks are deterministic program units, can be retried, and are coordinated by schedulers rather than human judgment.

Numbered breakdown

  1. The application starts and the recipe—our RDD lineage—is written.
  2. The customer calls collect(); this is the action that creates Job 0.
  3. The head chef, like the DAGScheduler, reads the recipe dependencies and builds the DAG.
  4. The “group ingredients by dish” step needs exchange, so the DAG is split into Stage 0 and Stage 1 at the shuffle.
  5. Four source trays create four Stage 0 tasks; two grouped output trays create two Stage 1 tasks.
  6. The TaskScheduler sends those tasks to available executors, which run them and return the final word counts.
RememberOne order can have several kitchen phases, and each phase can have several cooks. Likewise, one Job can have several Stages, and each Stage can have several Tasks.
The customer placing the order, which maps to calling an action such as collect().
Section 4

Technical Foundations

Application

The complete Spark program from driver startup until shutdown. It owns a driver and uses executors.

Remember it as: the whole restaurant session.

Job

Execution work created because an action asks Spark for output. A clean run of our collect() creates Job 0.

Remember it as: one customer order.

DAG

A Directed Acyclic Graph of dependency nodes and one-way edges. “Acyclic” means the dependency chain does not loop back to itself.

Remember it as: the route map.

Stage

A set of parallel tasks that can run without crossing another shuffle boundary. The DAG is cut at wide dependencies.

Remember it as: one kitchen phase.

Task

The smallest scheduled unit. One task processes one partition for one stage. Retries may produce multiple attempts for that logical task.

Remember it as: one cook handling one tray.

Shuffle

Redistribution of records so identical keys reach the same downstream partition. It can use network, memory, and disk.

Remember it as: regrouping ingredients between stations.

Schedulers and dependencies

The DAGScheduler receives the action-level job, studies RDD lineage, identifies shuffle dependencies, and creates stages. The TaskScheduler receives task sets for a stage and sends individual tasks to executors based on available resources and data locality.

A narrow dependency means each downstream partition depends on a small number of upstream partitions, often one. Examples in our scenario are flatMap, filter, and map. A wide dependency means one downstream partition needs data from many upstream partitions. reduceByKey creates this wide dependency because the same word can appear in all four input partitions.

Watch OutRDD reduceByKey uses its own partitioner or the numPartitions argument. spark.sql.shuffle.partitions controls DataFrame/SQL shuffles, not this RDD example.
Section 5

The Example Scenario and Line-by-Line Trace

Our only scenario is a file named words.txt. Spark reads it using four input partitions, splits lines into words, keeps words longer than three characters, converts each word to a pair, counts occurrences with two reduce partitions, and collects the final pairs to the driver.

lines = sc.textFile("words.txt", minPartitions=4)
words = lines.flatMap(lambda line: line.split())
long_words = words.filter(lambda word: len(word) > 3)
pairs = long_words.map(lambda word: (word.lower(), 1))
counts = pairs.reduceByKey(lambda a, b: a + b, numPartitions=2)
result = counts.collect()

Section 6

Stage-by-Stage Walkthrough

Worked NumbersThe file has 4 input partitions, so Stage 0 has 4 tasks. reduceByKey(..., numPartitions=2) creates 2 reduce partitions, so Stage 1 has 2 tasks. Total logical tasks without retry or speculation: 6.

Stage 0 — Read, split, filter, pair, and shuffle-write

4input partitions
4tasks
0shuffle read
Yesshuffle write

Code lines: textFile → flatMap → filter → map, plus the map-side part of reduceByKey that prepares shuffle output.

  1. Task 0 reads input partition P0; Tasks 1–3 independently read P1–P3.
  2. Inside each task, lines are split, short words are discarded, and surviving words become lowercase pairs.
  3. Local combining may reduce repeated keys inside each map partition before network transfer.
  4. Each task hashes keys into destination reduce partition R0 or R1 and writes shuffle blocks.
  5. Stage 0 finishes after all required shuffle-map outputs are available.

Reads: text-file input partitions. Writes: shuffle files for two reduce partitions. Why four tasks? The stage has four source partitions and each task processes one partition.

Stage 1 — Shuffle-read, final reduce, and collect result

2reduce partitions
2tasks
Yesshuffle read
Driverfinal output

Code lines: downstream part of reduceByKey and the collect() result action.

  1. Task 0 for reduce partition R0 fetches its shuffle block from every completed Stage 0 task.
  2. Task 1 does the same for reduce partition R1.
  3. Each task merges values for identical word keys and creates final (word,count) pairs.
  4. The two result partitions are serialized and sent to the driver because collect() requested all results.

Reads: shuffle blocks from Stage 0. Writes: final result sent to the driver. Why two tasks? numPartitions=2 makes two partitions in the reduced RDD.

Why the boundary exists exactly thereBefore reduceByKey, each partition can work independently. To finish a count for the word “spark,” every “spark” pair from all four input partitions must reach the same reduce partition. That many-to-many dependency requires a shuffle, so the DAGScheduler creates a stage boundary.
Four tasks exist. Only two can run at one time, so the other two wait for free slots.
Section 7

Code and What the Spark UI Shows

from pyspark.sql import SparkSession

spark = (
    SparkSession.builder
    .appName("WordExecutionTrace")
    .master("local[4]")
    .getOrCreate()
)
sc = spark.sparkContext

# Four source partitions. Still lazy.
lines = sc.textFile("words.txt", minPartitions=4)

# Narrow transformations. Still lazy.
words = lines.flatMap(lambda line: line.split())
long_words = words.filter(lambda word: len(word) > 3)
pairs = long_words.map(lambda word: (word.lower(), 1))

# Wide dependency recorded. Two reduce partitions. Still lazy.
counts = pairs.reduceByKey(lambda a, b: a + b, numPartitions=2)

# ACTION: creates Job 0 and triggers Stage 0, then Stage 1.
result = counts.collect()
print(result)

input("Open http://localhost:4040, inspect the UI, then press Enter...")
spark.stop()

Comparison: Job vs Stage vs Task

ConceptCreated whenScopeOur exampleUse the word when...
JobAn action is calledAll work needed for that actionJob 0 from collect()You are discussing action-level execution.
StageThe DAG is divided at shuffle dependenciesA pipeline of tasks without another internal shuffle boundaryStage 0 and Stage 1You are discussing a shuffle-separated phase.
TaskA stage is submitted; one per partitionSmallest scheduled unit4 + 2 = 6 tasksYou are discussing work on one partition.

Wide dependencyRecorded during transformations; acted on after actionChild partition needs many parent partitionsreduceByKey shuffleData must be redistributed.

When to pick which word: Say “Job” when referring to the request created by an action. Say “Stage” when referring to one shuffle-bounded execution phase. Say “Task” when referring to one partition’s unit of work. Do not call a transformation a Job, and do not assume one line of code equals one Stage.

Memory HookOne action request → one Job; each shuffle cuts Stages; each partition creates a Task.
Section 9

Best Practices

Do

  1. Count actions. Why: every action can create another job and repeat lineage.
  2. Count shuffle boundaries. Why: they reveal likely stage divisions and expensive data movement.
  3. Inspect task distributions, not only averages. Why: one straggler can delay a whole stage.
  4. Choose partition counts deliberately. Why: too few reduce partitions limit parallelism; too many create overhead.
  5. Use reduceByKey instead of groupByKey for summation. Why: local combining reduces shuffle volume.
  6. Keep the UI or History Server available. Why: runtime evidence is more reliable than guessing.

Avoid

  1. Do not assume transformations run immediately. Why: this causes incorrect timing and debugging conclusions.
  2. Do not equate executor slots with task count. Why: slots control concurrency, not how many tasks exist.
  3. Do not collect large output. Why: every result partition is transferred into driver memory.
  4. Do not tune only by adding memory. Why: skew and poor partitioning may remain unchanged.
  5. Do not apply spark.sql.shuffle.partitions to RDD APIs. Why: this RDD uses numPartitions or its partitioner.
  6. Do not count stage IDs as universal constants. Why: IDs depend on earlier actions in the same application.

Remember it as: Actions, Shuffles, Partitions, Evidence — ASPE.

Section 10

Edge Cases and Gotchas

Multiple actions

counts.count() followed by counts.collect() can create two jobs. Without persistence, Spark may recompute the lineage.

Caching and skipped stages

If counts.persist() has already materialized partitions, later jobs may reuse cached data. The UI may show skipped stages because their outputs are already available.

One action, many stages

Every additional wide dependency can add another stage boundary. A single action can therefore produce three, five, or more stages.

Retries and attempts

A stage with four logical tasks may show more attempts if a task fails or speculation launches a duplicate attempt.

Local mode is still structured

local[4] uses one machine but still creates jobs, stages, and tasks. The same concepts apply in a cluster.

Input partition request is not always exact

minPartitions=4 is a minimum suggestion. File format, block layout, and implementation can influence actual partitions. Verify with lines.getNumPartitions().

Common ConfusionA shuffle boundary is a dependency boundary, not necessarily a moment when every byte waits globally in all execution modes. Spark can pipeline and fetch available outputs, but Stage 1 still depends on shuffle-map outputs from Stage 0.

Section 12

Interview Questions and Spoken Answers

Beginner1. What creates a Spark Job?
A Spark Job is created when an action asks for a result. In our example, all transformations remain lazy until collect() is called. That call creates Job 0 and starts scheduling.
Beginner2. What is a Spark Stage?
A Stage is a group of parallel tasks that can execute without crossing another shuffle boundary. Our word-count Job has Stage 0 before the reduce shuffle and Stage 1 after it.
Beginner3. What is a Spark Task?
A Task is the smallest scheduled unit of work. It runs the stage’s operator pipeline for one partition. Four input partitions therefore create four Stage 0 tasks.
Beginner4. What is a DAG?
A DAG is a Directed Acyclic Graph representing transformations and dependencies. The DAGScheduler uses it to identify wide dependencies and divide the job into stages.
Beginner5. Do transformations run immediately?
No. Transformations such as flatMap, filter, and map record lineage. They run only after an action triggers a job.
Intermediate6. Why does reduceByKey create a stage boundary?
The same key can exist in several parent partitions. Spark must redistribute pairs so all values for one key reach the same reduce partition. That wide dependency is a shuffle, which splits stages.
Intermediate7. How many stages and tasks does this tutorial example create?
In a clean run it creates one Job with two Stages. Stage 0 has four tasks because the input has four partitions. Stage 1 has two tasks because reduceByKey uses two result partitions. That is six logical tasks.
Intermediate8. What is the difference between DAGScheduler and TaskScheduler?
The DAGScheduler works at dependency and stage level: it builds the stage graph and submits ready stages. The TaskScheduler works at task level: it assigns task instances to executor slots.
Intermediate9. Why can one Job contain several Stages?
Each wide dependency creates a shuffle boundary. A job with several joins, groupings, or repartitions may have several such boundaries, so one action can require many stages.
Intermediate10. Does having four tasks mean four tasks always run together?
No. Four tasks exist because there are four partitions. Concurrency depends on available executor slots. With two slots, two tasks run while two wait.
Advanced11. Why might the UI show more task attempts than partitions?
A logical task can have multiple attempts because of failure retry or speculative execution. Partition count determines logical tasks; attempts reflect execution history.
Advanced12. How can caching change later job graphs?
After cached partitions are materialized, later jobs can read them instead of recomputing their ancestors. The UI may show skipped stages or a shorter active path because outputs are already available.
Advanced13. Why is reduceByKey preferred over groupByKey for counting?
reduceByKey can combine repeated keys locally before shuffle, which reduces network and disk data. groupByKey transfers all values and usually has a larger shuffle footprint.
Advanced14. How would you diagnose a skewed reduce stage?
I would compare task duration and shuffle-read size within the same stage. A few tasks that are much larger than the median suggest hot keys or uneven record sizes. I would then inspect keys and choose salting, partition changes, or domain-specific aggregation.
Advanced15. Is spark.sql.shuffle.partitions relevant to this RDD reduceByKey?
No. That setting controls Spark SQL and DataFrame shuffle partition defaults. This RDD operation uses its partitioner or the explicit numPartitions=2 argument.
Advanced16. What does a shuffle-map task write?
It writes partitioned shuffle blocks, one logical bucket per downstream reduce partition, often with local combine and sorting behavior depending on the shuffle implementation. Reduce tasks later fetch their assigned blocks.
Advanced17. Why can stage IDs differ from the tutorial?
Stage IDs are counters within an application. If another action or internal operation ran earlier, the same logical stages may receive later numbers. The dependency pattern and task counts are more meaningful than fixed IDs.
Advanced18. What happens if collect returns too much data?
Every result partition is transferred to driver memory. If the result exceeds available driver memory, the driver can slow down or fail. A distributed write or bounded sample is safer.

Section 15

Glossary

Action

An operation that asks Spark to return or write a result and therefore triggers a Job.

Application

The complete Spark process containing the driver and associated executors.

DAG

A Directed Acyclic Graph describing operator dependencies without cycles.

DAGScheduler

The scheduler that converts a job’s lineage into stages separated by shuffle dependencies.

Dependency

The relationship describing which parent partitions are needed to compute a child partition.

Executor

A process that runs tasks and can store application data.

Job

Action-level execution work.

Lazy evaluation

Recording transformations until an action requires output.

Narrow dependency

A dependency where a child partition needs only a small number of parent partitions.

Partition

A logical slice of data processed by one task in a stage.

Shuffle

Redistribution of records across partitions, usually through intermediate blocks.

Stage

A shuffle-bounded set of parallel tasks.

Task

The smallest scheduled work unit for one partition in one stage.

Task attempt

One execution attempt of a logical task; retries can create more than one.

TaskScheduler

The scheduler that assigns task instances to executor resources.

Wide dependency

A dependency where a child partition needs records from many parent partitions.

Summary, Cheat List, and What to Learn Next

Core summary

  • Transformations record lineage; they do not execute the word pipeline.
  • collect() creates one Job in the clean example.
  • The DAGScheduler builds the DAG and cuts it at the reduceByKey shuffle.
  • Four input partitions create four Stage 0 tasks.
  • Two reduce partitions create two Stage 1 tasks.
  • The TaskScheduler sends six logical tasks to executor slots.
When Does It Run? Recap
When is a Job created? → when an ACTION is called.
When is the DAG built? → when the job starts, from the recorded lineage.
When is a Stage created? → when the DAG is cut at a shuffle boundary.
When is a Task created? → one per partition of a stage, at submit time.
When does anything ACTUALLY execute? → only after the action; never before.

Cheat list

Mnemonic: Act, Job, Draw, Split, Task.
Hierarchy: Application → Job → Stage → Task.
Boundary rule: one shuffle creates a split between stages.
Count rule: one task per partition per stage.
Example: Stage 0 = 4 tasks; Stage 1 = 2 tasks.
UI rule: Jobs for overview; Stages for task and shuffle evidence.
