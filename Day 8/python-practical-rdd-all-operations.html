Section 2

RDD Foundations

An RDD, or Resilient Distributed Dataset, is Spark’s low-level distributed collection abstraction. It is resilient because Spark can recompute missing partitions from lineage, distributed because partitions can live across executors, and a dataset because it contains records that can be processed in parallel.

RDDs are immutable. A transformation never changes the original RDD; it returns a new RDD description. This protects parallel computations from accidental shared-state mutation and gives Spark a lineage graph for fault recovery.

For structured tables, DataFrames are usually preferred because Catalyst and Tungsten can optimize the query plan. RDDs remain useful for unstructured records, custom partition-level algorithms, legacy Spark code, low-level control, and transformations that do not fit relational expressions.

Use an RDD when

  • Records are unstructured Python objects.
  • You need mapPartitions() or custom key partitioning.
  • You are teaching Spark execution fundamentals.
  • You must maintain an existing RDD codebase.

Prefer a DataFrame when

  • Data has columns and a known schema.
  • You need SQL, joins, aggregation, or BI workloads.
  • Query optimization and efficient serialization matter.
  • You want Spark Connect compatibility.
Common ConfusionAn RDD is not a Python list spread across machines. It is a distributed, lazy computation whose records are available only when tasks materialize partitions.
The lineage graph. If a partition is lost, Spark can recompute it from its parent transformations instead of requiring a full replicated copy.

Section 5

Execution Model and Operation Families

RDD APIs fall into two primary families. A transformation returns another RDD and is normally lazy. An action returns a driver value, writes external output, or performs a side effect and therefore triggers a Spark job.

Narrow transformation

Each child partition depends on a small number of parent partitions. Examples: map, filter, flatMap, mapPartitions.

Wide transformation

Records move across partitions. Examples: reduceByKey, groupByKey, distinct, join, sortByKey.

Action

Requests a result or output. Examples: count, collect, take, reduce, and save methods.

city_revenue = (transactions
    .filter(lambda t: t[STATUS] == "PAID")
    .map(lambda t: (t[CITY], t[QTY] * t[PRICE]))
    .reduceByKey(lambda a, b: a + b, numPartitions=2))

print(city_revenue.toDebugString().decode("utf-8"))
print(city_revenue.collect())
Memory HookNarrow stays near; wide travels worldwide. Narrow work usually remains within a partition. Wide work redistributes records.

Core Narrow Transformations

1. map — one input record to one output record

amounts = transactions.map(lambda t: t[QTY] * t[PRICE])
print(amounts.collect())

2. flatMap — one input record to zero, one, or many outputs

sentences = sc.parallelize(["spark is fast", "python with spark"], 2)
words = sentences.flatMap(lambda line: line.split())
print(words.collect())
# ['spark', 'is', 'fast', 'python', 'with', 'spark']

3. filter — retain records matching a predicate

paid = transactions.filter(lambda t: t[STATUS] == "PAID")
large_paid = paid.filter(lambda t: t[QTY] * t[PRICE] >= 10_000)
print(large_paid.collect())

4. mapPartitions — initialize once per partition

def summarize_partition(rows):
    rows = list(rows)
    yield {
        "row_count": len(rows),
        "revenue": sum(t[QTY] * t[PRICE] for t in rows)
    }

partition_summaries = transactions.mapPartitions(summarize_partition)
print(partition_summaries.collect())

5. mapPartitionsWithIndex — include partition number

def inspect_partition(index, rows):
    rows = list(rows)
    yield (index, len(rows), [t[TXN_ID] for t in rows])

print(transactions.mapPartitionsWithIndex(inspect_partition).collect())

6. glom — view each partition as a Python list

for partition_no, rows in enumerate(transactions.glom().collect()):
    print(partition_no, rows)

7. key/value helpers

customer_keys = by_customer.keys()
original_rows = by_customer.values()

Section 8

Pair RDD Operations

A Pair RDD contains (key, value) tuples. Key-based APIs enable aggregation, partitioning, lookup, joins, and grouped processing.

Aggregation choices

amount_by_category = transactions.map(
    lambda t: (t[CATEGORY], t[QTY] * t[PRICE])
)

# Sum values; map-side combine reduces shuffle traffic
sum_by_category = amount_by_category.reduceByKey(lambda a, b: a + b)

# Collect every value per key; can consume large memory
all_amounts = amount_by_category.groupByKey().mapValues(list)

# Zero value + within-partition function + between-partition function
stats_by_category = amount_by_category.aggregateByKey(
    (0.0, 0),
    lambda acc, value: (acc[0] + value, acc[1] + 1),
    lambda left, right: (left[0] + right[0], left[1] + right[1])
)
average_by_category = stats_by_category.mapValues(lambda x: x[0] / x[1])

# Generic aggregation foundation
average2 = amount_by_category.combineByKey(
    lambda value: (value, 1),
    lambda acc, value: (acc[0] + value, acc[1] + 1),
    lambda a, b: (a[0] + b[0], a[1] + b[1])
).mapValues(lambda x: x[0] / x[1])

# Neutral zero value added with an associative function
folded = amount_by_category.foldByKey(0.0, lambda a, b: a + b)
Selection RuleUse reduceByKey for same-type associative aggregation; aggregateByKey or combineByKey when the output type differs; avoid groupByKey when you only need a sum, count, min, max, or average.

Joins and grouped relationships

txn_by_product = transactions.map(lambda t: (t[PRODUCT], t))

inner = txn_by_product.join(products)
left = txn_by_product.leftOuterJoin(products)
right = txn_by_product.rightOuterJoin(products)
full = txn_by_product.fullOuterJoin(products)

# Values from each RDD are grouped separately per key
co_grouped = txn_by_product.cogroup(products)
grouped_many = txn_by_product.groupWith(products)

print(inner.take(3))

Other Pair RDD tools

# Remove keys found in another RDD
blocked_customers = sc.parallelize([("C102", True), ("C105", True)])

# Return matching values to the driver
print(by_customer.lookup("C101"))

# Number of records per key, returned as a driver dictionary
print(by_customer.countByKey())

# Reduce and return a driver dictionary
print(amount_by_category.reduceByKeyLocally(lambda a, b: a + b))

# Apply a stable partitioner
partitioned = amount_by_category.partitionBy(4)

# Partition and sort in one operation
sorted_partitioned = amount_by_category.repartitionAndSortWithinPartitions(
    numPartitions=4
)

Persistence, Checkpointing and Lineage

When the same expensive RDD feeds multiple actions, Spark recomputes it unless it is persisted. Persistence stores materialized partitions after the first action. Checkpointing truncates long lineage by saving a reliable copy to checkpoint storage.

from pyspark import StorageLevel

paid_enriched = (transactions
    .filter(lambda t: t[STATUS] == "PAID")
    .map(lambda t: (t, t[QTY] * t[PRICE])))

# Shortcut: default RDD cache level in PySpark
paid_enriched.cache()

# Or choose a level explicitly before materialization
paid_enriched.persist(StorageLevel.MEMORY_AND_DISK)

# First action computes and stores partitions
print(paid_enriched.count())
# Later action can reuse cached partitions
print(paid_enriched.take(3))

print(paid_enriched.getStorageLevel())
paid_enriched.unpersist(blocking=False)

Reliable checkpoint

sc.setCheckpointDir("checkpoint/rdd")
long_lineage = transactions.map(lambda x: x).filter(lambda x: True)
long_lineage.checkpoint()
long_lineage.count()  # materializes the checkpoint
print(long_lineage.isCheckpointed())
print(long_lineage.getCheckpointFile())

Local checkpoint

temporary_lineage_cut = transactions.map(lambda x: x)
temporary_lineage_cut.localCheckpoint()
temporary_lineage_cut.count()
print(temporary_lineage_cut.isLocallyCheckpointed())

Partitions and Performance Tuning

A partition is the unit a task processes. Too few partitions underuse the cluster; too many tiny partitions increase scheduling overhead. Wide operations also create new partitioning decisions that affect later joins and aggregations.

print(transactions.getNumPartitions())
print(transactions.glom().map(len).collect())
print(transactions.toDebugString().decode("utf-8"))

# Inspect records per partition
sizes = transactions.mapPartitionsWithIndex(
    lambda idx, rows: [(idx, sum(1 for _ in rows))]
)
print(sizes.collect())

Custom partition function

def customer_partitioner(customer_id):
    return int(customer_id[1:]) % 4

customer_amount = transactions.map(
    lambda t: (t[CUSTOMER], t[QTY] * t[PRICE])
)
partitioned = customer_amount.partitionBy(4, customer_partitioner)
print(partitioned.glom().collect())

Practical tuning checklist

Reduce data before shuffle

Filter rows and map to only necessary fields before reduceByKey, joins, distinct, or sorting.

Aggregate locally

Prefer reduceByKey, aggregateByKey, or combineByKey over groupByKey for aggregates.

Reuse partitioners

Key-preserving APIs such as mapValues can retain partitioning and reduce future shuffle.

Inspect skew

Compare task input, shuffle-read size, and duration in the Stages tab. A few oversized partitions create stragglers.

coalesce versus repartitionrepartition(n) always shuffles and balances. coalesce(n) usually avoids shuffle and is efficient for reducing partitions, but may create uneven partition sizes.

Accumulator — executor-to-driver counter

invalid_status_count = sc.accumulator(0)
valid_statuses = {"PAID", "PENDING", "CANCELLED"}

def validate(row):
    if row[STATUS] not in valid_statuses:
        invalid_status_count.add(1)
    return row

validated = transactions.map(validate)
validated.count()  # an action is required
print("Invalid statuses:", invalid_status_count.value)
Accumulator SemanticsDo not use an accumulator to produce business-critical exactly-once results. Tasks can retry, so updates inside transformations may be applied more than once. Use accumulators mainly for diagnostics and counters.
Closure MistakeUpdating a normal Python variable inside map() does not update the driver variable. Every task receives a serialized copy.
When the same large read-only lookup is reused by many tasks or stages. Broadcasting avoids repeatedly shipping a separate copy with task closures.

End-to-End Retail Practical Project

Business requirement

A retail operations team needs paid-order revenue by category, customer-level spending statistics, product-segment enrichment, data-quality counters, and the three highest-value transactions. The same cleaned RDD feeds several outputs, so it should be persisted.

Expected deliverables

  • Paid record count
  • Rejected record count
  • Category revenue sorted descending
  • Customer total, order count, and average
  • Product-segment left join
  • Top-three transaction list
  • Distributed text output
from operator import add
from pyspark import StorageLevel

# 1. Validate and count rejected rows
invalid_rows = sc.accumulator(0)
valid_statuses = {"PAID", "PENDING", "CANCELLED"}

def validate(t):
    if t[STATUS] not in valid_statuses or t[QTY] <= 0 or t[PRICE] < 0:
        invalid_rows.add(1)
        return False
    return True

valid = transactions.filter(validate)

# 2. Keep paid transactions and calculate gross amount
paid = (valid
    .filter(lambda t: t[STATUS] == "PAID")
    .map(lambda t: (t, t[QTY] * t[PRICE]))
    .persist(StorageLevel.MEMORY_AND_DISK))

# 3. First action materializes cache
paid_count = paid.count()

# 4. Revenue by category
category_revenue = (paid
    .map(lambda x: (x[0][CATEGORY], x[1]))
    .reduceByKey(add, numPartitions=2)
    .sortBy(lambda pair: pair[1], ascending=False))

# 5. Customer statistics: total and order count
customer_stats = (paid
    .map(lambda x: (x[0][CUSTOMER], x[1]))
    .aggregateByKey(
        (0.0, 0),
        lambda acc, amount: (acc[0] + amount, acc[1] + 1),
        lambda a, b: (a[0] + b[0], a[1] + b[1])
    )
    .mapValues(lambda x: {"total": x[0], "orders": x[1], "average": x[0] / x[1]}))

# 6. Join transactions to product segment
transaction_products = paid.map(lambda x: (x[0][PRODUCT], (x[0], x[1])))
enriched = transaction_products.leftOuterJoin(products)

# 7. Top three transactions without collecting everything
result_top3 = paid.takeOrdered(3, key=lambda x: -x[1])

# 8. Inspect and save small curated outputs
print("Paid count:", paid_count)
print("Invalid rows:", invalid_rows.value)
print("Category revenue:", category_revenue.collect())
print("Customer stats:", customer_stats.collect())
print("Top 3:", result_top3)

enriched.map(str).saveAsTextFile("output/retail_enriched")
paid.unpersist()
ConclusionThis project demonstrates the practical lifecycle of an RDD application: create, validate, transform, persist, aggregate, join, inspect safely, write distributed output, and release cached storage.

Best practices with reasons

Filter and project early

Why: fewer and smaller records enter expensive shuffle operations.

Use associative and commutative reducers

Why: Spark may combine values in different orders across partitions.

Persist only reused expensive RDDs

Why: caching consumes executor storage and may evict useful data.

Use mapPartitions for expensive setup

Why: one connection/model initialization per partition is cheaper than per row.

Name important RDDs

Why: the Web UI Storage tab becomes easier to interpret.

Prefer DataFrames for structured analytics

Why: Spark can optimize schemas, expressions, joins, and storage.

Never Do ThisDo not modify external systems non-idempotently inside map() or foreach() without handling task retries. A failed task can run again and duplicate side effects.
Section 17

Interview Questions and Flash Review

Read the answer aloud as though responding in an interview. State the definition, practical reason, and one risk or example.

Beginner

1. What is an RDD?

An RDD is an immutable, partitioned, fault-tolerant distributed collection. Spark records how to compute each partition and executes that recipe in parallel after an action.

Beginner

2. What is the difference between a transformation and an action?

A transformation returns another RDD and is lazy. An action requests a driver result, external output, or side effect and triggers a job.

Beginner

3. Why is an RDD immutable?

Immutability avoids unsafe shared updates and allows Spark to preserve lineage for deterministic recomputation after failures.

Beginner

4. What is a partition?

A partition is a logical slice of an RDD. In a stage, Spark normally creates one task for each partition.

Beginner

5. map versus flatMap?

map returns one output per input. flatMap returns an iterable per input and flattens those iterables, so one input can produce zero or many outputs.

Beginner

6. Why should collect be used carefully?

collect transfers every record to driver memory. A large RDD can crash the driver or cause heavy network transfer.

Intermediate

7. reduceByKey versus groupByKey?

reduceByKey can combine values locally before shuffle, so it is normally better for sums, counts, min, max, and similar aggregates. groupByKey sends all values and stores grouped collections.

Intermediate

8. repartition versus coalesce?

repartition performs a full shuffle and balances data. coalesce usually avoids shuffle and is efficient for reducing partitions, but can leave them uneven.

Intermediate

9. Why use mapPartitions?

It lets one function process a partition iterator and reuse expensive setup such as a connection, parser, or model for many records.

Intermediate

10. What does mapValues preserve?

It changes only values in a Pair RDD and can preserve the existing key partitioner, which may avoid later shuffle.

Intermediate

11. aggregateByKey versus combineByKey?

aggregateByKey provides a zero value plus separate within- and between-partition functions. combineByKey is the most generic key combiner and explicitly defines creation, value merge, and combiner merge.

Intermediate

12. When should an RDD be cached?

Cache it when it is expensive to compute and reused by multiple actions or branches. Materialize it with an action and unpersist it when no longer needed.

Intermediate

13. What is a broadcast variable?

A broadcast variable distributes a large read-only object efficiently so tasks can access the executor-local cached copy instead of receiving repeated closure copies.

Advanced

14. How does lineage provide fault tolerance?

Each RDD stores dependencies on parent RDDs. When a partition is lost, Spark reruns only the required parent transformations to reconstruct it.

Advanced

15. Why must reduce functions be associative and commutative?

Distributed partial results may be combined in different grouping orders. Associative and commutative functions produce stable results regardless of that order.

Advanced

16. Why can accumulators overcount?

A failed or speculative task can execute again. Updates performed inside transformations can therefore occur more than once.

Advanced

17. How do partitioners reduce shuffle?

If two key-value RDDs already share a compatible partitioner, key-preserving operations and joins can reuse partition placement instead of redistributing every record.

Advanced

18. What is the purpose of checkpointing?

Checkpointing materializes an RDD to checkpoint storage and truncates long lineage, reducing recovery cost and scheduler complexity for iterative workflows.

Advanced

19. What is data skew?

Skew occurs when a few partitions contain much more work or data than others. Those tasks become stragglers while other executor slots sit idle.

Advanced

20. When would you use treeAggregate or treeReduce?

Use tree-based aggregation when merging many partition results into the driver would create a bottleneck. The multi-level tree distributes part of the merge work.

Because reduceByKey can merge values within each source partition before the network shuffle, reducing transferred data and downstream memory usage.

Glossary, Summary and Certificate

Glossary

Action

An operation that triggers a Spark job and returns/writes a result.

Accumulator

A shared add-only variable used mainly for counters and diagnostics.

Broadcast variable

A read-only value efficiently cached on executors.

Checkpoint

A materialized copy used to truncate lineage.

Closure

The function and captured variables serialized to executors.

DAG

Directed acyclic graph of RDD dependencies.

Driver

The process coordinating the Spark application.

Executor

A worker process that runs tasks and stores cached data.

Lineage

The transformation recipe used to compute an RDD.

Narrow dependency

A child partition depends on a small number of parent partitions.

Pair RDD

An RDD whose records are key-value tuples.

Partition

A logical data slice processed by one task in a stage.

Partitioner

A rule that maps a key to a partition index.

Persistence

Keeping computed RDD partitions for reuse.

RDD

Resilient Distributed Dataset.

Shuffle

Redistribution of records across partitions.

Stage

A shuffle-bounded set of parallel tasks.

Task

The smallest scheduled work unit for one stage partition.

Transformation

A lazy operation returning a new RDD.

Wide dependency

A child partition depends on records from many parent partitions.

Pair RDD: reduceByKey, aggregateByKey, combineByKey, join.
Performance: filter early, avoid groupByKey, inspect partitions, persist reused RDDs.
