1. Spark Job Execution - Overview

When you run Spark code, five things exist behind the scenes: the Application, its Jobs, each job's DAG, the Stages the DAG is cut into, and the Tasks inside each stage. This tutorial takes ONE small example and follows it all the way through, so you can see exactly what each unit is and when it comes into existence.

Learning objectives:

  • Define Application, Job, DAG, Stage and Task, and say how they nest.
  • State precisely WHEN each one is created - and why nothing runs before the action.
  • Predict the number of jobs, stages and tasks for a given piece of code.
  • Explain why a shuffle forces a new stage.
Before you startPrerequisites: you can read a simple PySpark chain (map/filter/reduceByKey). Estimated time: 45-60 min. After this you can whiteboard Spark's execution model in an interview.

2. Introduction

Reading Spark code tells you what you asked for. It does not tell you how Spark will run it - and that gap is where every performance problem hides. Two lines of code that look almost identical can produce completely different execution: one runs as a single pass over the data, the other shuffles gigabytes across the network. The only way to tell them apart is to understand the execution model.

Why it matters. Every Spark interview asks some version of "walk me through what happens when you call an action". Every slow job is debugged by looking at jobs, stages and tasks in the Spark UI. If you cannot say why a job has 3 stages instead of 1, or why one stage has 200 tasks, you cannot tune it. This is the foundational mental model that everything else in Spark sits on.

The one sentence version. Transformations record a plan; an action triggers a Job; the job's DAG is cut into Stages at every shuffle; each stage becomes one Task per partition; executors run the tasks. Everything below is that sentence, unpacked.

Memory hookRemember it as: An action starts a Job. The DAG is the plan. Shuffles cut it into Stages. Partitions cut Stages into Tasks.

3. Everyday Analogy (No Jargon Yet)

Imagine a restaurant kitchen. A customer's order arrives at the pass.

  1. The head chef reads the order. He is the Driver. He does no cooking - he plans and coordinates.
  2. The order itself - "one full meal" - is a Job. It exists only because the customer ordered. Until someone orders, the kitchen writes nothing down and cooks nothing. That order is the action.
  3. The chef writes the recipe: chop, then fry, then plate. That written plan is the DAG.
  4. He notices some work cannot start until earlier work is completely finished - you cannot plate until all the frying is done. Those hard boundaries split the recipe into Stages.
  5. Inside a stage, the work is handed to individual cooks: "you chop this crate of onions, you chop that one". Each cook with one crate is a Task. The cooks are Executors.

The hard boundary in step 4 is the important one. In Spark, that boundary is a shuffle: a step where data must be regrouped across machines before the next step can begin.

Where the analogy breaks down: a real chef might start prepping before the order arrives, out of habit. Spark never does. It is strictly lazy - nothing at all happens until the action.

Remember it asOrder = Job · Recipe = DAG · Phases = Stages · One cook, one crate = Task.
Check yourself: the customer never orders. How much cooking happens?

None. No action means no job, which means nothing executes - no matter how many transformations you wrote.

Check yourself: what does the "A" in DAG stand for, and why does it matter?

Acyclic - meaning "no loops". The arrows never go back to an earlier step, so data always flows forward. That is what lets Spark plan the work as a clean sequence of stages.

5. The Five Units (Technical)

Precise definitions. Every term is defined the first time it appears.

  • Application - one complete Spark program: one driver plus its executors, alive from getOrCreate() to stop(). It can run many jobs.
  • Driver - the process running your main code. Builds the DAG, schedules work, serves the Web UI. Does not process the bulk data itself.
  • Executor - a worker process that actually runs tasks and holds cached data.
  • Job - all the work triggered by one action. One action = one job.
  • Action - an operation that returns a result or writes output (collect, count, save). It is the trigger.
  • Transformation - a lazy operation returning a new RDD/DataFrame (map, filter). Records a step; runs nothing.
  • DAG - Directed Acyclic Graph: the operator graph built from the lineage when the job starts. It is a plan.
  • Stage - a set of tasks that can run with no shuffle between them. The DAG is cut at every shuffle.
  • Task - the smallest unit of work: one task processes one partition.
  • Shuffle - redistributing data across the network so related records land together. It is what forces a stage boundary.
  • Partition - one slice of the data. The number of partitions decides the number of tasks.
Common confusionAn Application is NOT a Job. The application is the container and the resources (one driver, one set of executors). A job is a unit of work inside it, created every time you call an action.

7. The Example and its DAG

One example, used for the rest of this tutorial - counting long words in a text file:

rdd = sc.textFile("words.txt")            # 4 partitions
         .flatMap(lambda l: l.split(" "))  # split into words
         .filter(lambda w: len(w) > 3)     # keep long words
         .map(lambda w: (w, 1))            # pair with 1
         .reduceByKey(lambda a, b: a + b)  # count per word  <-- SHUFFLE

rdd.collect()                              # ACTION -> triggers everything

9. Stage-by-Stage Walkthrough

Stage 0 - the narrow stage (4 tasks)

Which code: textFile → flatMap → filter → map.

What physically happens:

  1. The file is already split into 4 partitions, so Spark creates 4 tasks - one per partition.
  2. Each task takes its own partition and runs all four operators in a single pass. Because every step is narrow, a record can stream through read → split → filter → pair without ever stopping. This is called pipelining.
  3. No data leaves the machine during this stage. That is what "narrow" means.
  4. At the end, each task writes its (word, 1) pairs to local disk, bucketed by word. This is the shuffle write.

Why the stage ends here: the next step (reduceByKey) needs all copies of a word together, and they are scattered across all 4 partitions. Data must cross the network - so Spark cuts the stage.

Stage 1 - the reduce stage (2 tasks)

Which code: reduceByKey → collect.

What physically happens:

  1. Stage 1 cannot start until Stage 0 has completely finished - you cannot total a word's count until every partition has reported.
  2. There are 2 shuffle partitions, so Spark creates 2 tasks.
  3. Each task shuffle reads the buckets belonging to its words from every Stage 0 task.
  4. It sums the counts per word, and collect() ships the small final result back to the driver.
Pro tipStages run sequentially; tasks within a stage run in parallel. That single sentence explains most Spark performance behaviour.

Common confusionFive lines of code and zero execution - until line six. This is why an error inside your map lambda appears at the collect() line: the stack trace points at the action, not at the mistake.
Memory hook - "when does it run?" Job created when? → when an ACTION is called.
DAG built when? → when the job starts, from the lineage.
Stage created when? → when the DAG is cut at a shuffle.
Task created when? → one per partition, when a stage is submitted.
Anything actually executes when? → only after the action. Never before.

How to keep the words straight: if you are talking about what the user asked for, say job. If you are talking about a phase of work bounded by data movement, say stage. If you are talking about one machine chewing one slice of data, say task. Interviewers listen for exactly this precision.

12. Best Practices

Do:

  • Count your actions before you run. Why: each one is a whole extra job that recomputes the chain.
  • cache() a DataFrame you reuse across actions. Why: without it, the second action re-runs every stage from scratch.
  • Filter early in the chain. Why: less data reaches the shuffle, which is the expensive part.
  • Count your shuffles to predict your stages. Why: stages = shuffles + 1; fewer shuffles is nearly always faster.

Common mistakes:

  • Expecting a transformation to run where you wrote it. Why bad: it is lazy - errors and cost appear at the action.
  • Calling collect() on large data. Why bad: the whole dataset is pulled into driver memory and can crash it.
  • Calling several actions on an uncached chain. Why bad: you silently recompute everything, once per action.
  • Leaving spark.sql.shuffle.partitions at 200 on small data. Why bad: 200 near-empty tasks add pure overhead.

14. Interview Q&A (15)

BeginnerWhat is a Spark application?
One complete program: a driver plus its executors, alive from getOrCreate() to stop(). It can run many jobs - one per action.
BeginnerWhat creates a job?
An action (collect, count, save). One action equals one job. Transformations create no job.
BeginnerWhat is a task?
The smallest unit of work: one task processes exactly one partition, running on one executor core.
BeginnerWhat is a stage?
A group of tasks that can run without any shuffle between them. Stages are separated by shuffle boundaries.
BeginnerWhat is the DAG?
The Directed Acyclic Graph of operations, built by the DAGScheduler from the recorded lineage when the job starts. It is the execution plan.
IntermediateWhy are transformations lazy?
So Spark sees the whole plan before running it: it can push filters earlier, pipeline narrow steps into one pass, skip unneeded work, and rebuild lost partitions from lineage.
IntermediateHow many stages will my job have?
Number of shuffles + 1. Each wide transformation (groupByKey, reduceByKey, join, distinct, repartition) adds a boundary.
IntermediateHow many tasks will a stage have?
One per partition in that stage. Input partitions for the first stage; spark.sql.shuffle.partitions for a shuffle stage.
IntermediateDo stages run in parallel?
No - dependent stages run sequentially, because a stage needs the previous stage's shuffle output. Tasks *within* a stage run in parallel.
IntermediateNarrow vs wide dependency?
Narrow: each output partition depends on one input partition, no data movement (map, filter). Wide: output depends on many input partitions, forcing a shuffle (reduceByKey, join).
AdvancedWalk me through what happens when I call collect().
A job is created. The DAGScheduler builds the DAG from the lineage and cuts it at each shuffle into stages. The TaskScheduler creates one task per partition and sends them to executors. Stage 0's tasks run in parallel, write shuffle files, and only when all finish does Stage 1 read them, reduce, and return results to the driver.
AdvancedWhy does calling two actions run everything twice?
Each action creates its own job, and jobs recompute from lineage. Caching the DataFrame after the first action lets the second reuse the materialised result instead of recomputing the stages.
AdvancedWhat are DAGScheduler and TaskScheduler?
The DAGScheduler is driver-side and stage-oriented: it builds the DAG and splits it into stages at shuffles. The TaskScheduler takes each stage's tasks and assigns them to executors.
AdvancedHow does the DAG give fault tolerance?
The DAG records the lineage - the recipe for each partition. If a partition is lost, Spark recomputes just that one from its parents rather than replicating all data.
AdvancedCan one action create several jobs?
Usually one, but some operations internally trigger extra jobs (for example, certain sorts or file-listing steps), so the UI may show more than you expect.

17. Summary + Glossary

Transformations record a plan. An action creates a Job. The DAG is built from the lineage, then cut into Stages at every shuffle. Each stage becomes one Task per partition, and executors finally do the work.

Quick-revision cheat list:

  • Application = driver + executors, getOrCreate() → stop(). Contains many jobs.
  • Job created by an action. Jobs = number of actions.
  • DAG built by the DAGScheduler when the job starts. It is the plan.
  • Stages = shuffles + 1. They run sequentially.
  • Tasks = one per partition. They run in parallel.
  • Nothing executes before the action. Ever.
  • Our example: 1 job · 2 stages · 6 tasks (4 + 2).
  • Narrow = pipelined in one pass. Wide = shuffle = new stage.

Glossary - every technical word from this tutorial:

  • Action - a step that produces a real result and triggers execution (collect, count, save).
  • Application - one whole Spark program: a driver plus its executors.
  • Cache - keep a computed dataset in memory so later actions do not recompute it.
  • DAG - Directed Acyclic Graph: a one-way flow chart of your steps, with no loops. Spark's plan.
  • DAGScheduler - the driver component that builds the DAG and cuts it into stages.
  • Driver - the boss process: runs your code, plans, schedules, serves the Web UI.
  • Executor - a worker process that runs tasks and holds data.
  • Job - all the work triggered by one action.
  • Key-value pair - a record like (word, 1); the key is grouped on, the value is combined.
  • Lazy - Spark records a step but does not run it until an action demands a result.
  • Lineage - the recorded list of steps used to build a dataset; the "recipe" Spark keeps for planning and for rebuilding lost data.
  • Narrow transformation - needs only its own partition; no data moves (map, filter, flatMap).
  • Partition - one slice of the data; one task processes one partition.
  • Pipelining - running several narrow steps in a single pass over the data.
  • RDD - Resilient Distributed Dataset: data spread across machines that can rebuild itself.
  • Shuffle - moving data across the network so related records land together; expensive.
  • Shuffle read / write - the data a task writes for, or collects after, a shuffle.
  • Skew - one partition far bigger than the rest, causing one slow straggler task.
  • Stage - a chunk of the DAG with no shuffle inside it. Stages run in order.
  • Task - the smallest unit of work: one partition, one executor core. Tasks run in parallel.
  • TaskScheduler - the driver component that assigns tasks to executors.
  • Transformation - a lazy step producing new data (map, filter, reduceByKey).
  • Wide transformation - needs many partitions, so it forces a shuffle (reduceByKey, join).
